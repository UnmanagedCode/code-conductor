## Stack
- Node 22+ (`node:test`, top-level await, `crypto.randomUUID`).
- `express` (REST + static) + `ws` (WebSocket).
- Vanilla HTML/CSS/JS in `public/`, no build step.
- `happy-dom` (dev-only) for DOM-backed rendering tests.
- No DB — projects in `<projectsRoot>/`, sessions in `~/.claude/projects/<encoded-cwd>/*.jsonl` (the jsonl root is overridable via `CLAUDE_PROJECTS_ROOT`).

## Component layout
- **server.js** — Express + ws boot, mounts `/api` → `/mcp` → `/plugins` (reverse proxy) → static, binds `127.0.0.1:8787`. The WebSocketServer is `noServer: true` with a manual `server.on('upgrade')` dispatcher: `/ws` → wsHub, `/plugins/<id>/…` → plugin proxy raw-socket pipe, anything else destroyed. (With `noServer` the ws lib never attaches to the http server, so the old EADDRINUSE error-forwarding guard around `listenWithRetry` is gone — `tests/eaddrinuse-retry.test.mjs` covers the retry path.) Creates the plugin host and, after listen, feeds it the bound port + fires its adopt-on-boot init. Sets `process.title = 'code-conductor'` on entry so `pkill -f code-conductor` targets it precisely and `pkill -f server.js` misses it. Launched via `conductor.sh` (`npm start`).
- **src/instances.js** — Instance class, InstanceManager, capped event ring (`EventLog`: drop-oldest at `ORCH_EVENT_RING_CAP`, default 2000; `_seq` stamped at push and never renumbered; `trimmedBefore` = first retained `_seq`; trim snaps the surviving head to an outer `user_echo` turn boundary, falling back to the nearest **quiescent point** (`firstQuiescentAtOrAfter`, parser.js) when no echo is in reach so the gap-case head still opens on whole blocks — plain cut only for a giant non-quiescent span; `snapshotTail()` serves the ≤`ORCH_SNAPSHOT_TAIL` (default 500) trailing slice for WS subscribe, quiescent-snapped via `snapStartToQuiescent`), absolute `userIndex` stamping on outer user_echoes (the rewind/fork anchor — immune to trimming), control_request round-trip, mode validation. **Resume idempotency:** `create()` is a thin synchronous wrapper over `_doCreate()` that guards duplicate resumes of one sessionId — an already-live session (`i.proc`) throws 409 `already attached`, and a still-in-flight resume (tracked in `this._resuming: sessionId→Promise`) is **coalesced** (the second concurrent caller gets the same restored instance, one subprocess). The checks run in the wrapper's synchronous prefix (before any `await`), closing the window `_doCreate` opens between the guard and `spawn()` attaching `.proc`; the map entry is released when the create settles (success OR failure). This makes the restart anchor auto-resume racing the manifest restore — and manual stop+resume — safe regardless of caller.
- **src/idleSubscriptions.js** — `IdleSubscriptionHub`, the idle-subscription graph extracted from `InstanceManager` as a composed collaborator. Owns the `targetInstanceId → Map<callerInstanceId,{timerId}>` map — **keyed by the stable instanceId** (the event stream carries it, and an in-place `/clear` rotates the sessionId while keeping the instanceId, so keying by instanceId needs no migration). sessionId lives only at the boundary: `subscribe`/`unsubscribe` translate the MCP-supplied sessionIds to instanceIds on entry (via `liveForSession`), `deliver` translates the target instanceId back to its current sessionId for the wake stub, and the test-only `snapshot()` projects instanceIds back to sessionIds. Subscriptions are purged on `remove()` and never persisted, so there is nothing to "survive a restart." One-shot consume-on-fire by whichever of **three trigger paths** lands first — `turn_end`, the **idle task-drain settle** (`ORCH_IDLE_DRAIN_SETTLE_MS`, default 5 s — see the wake state machine below), or the **always-armed** `timeoutMs` watchdog (explicit finite value, else `DEFAULT_SUBSCRIBE_TIMEOUT_MS` = 30 min via `ORCH_SUBSCRIBE_TIMEOUT_MS`; `.unref()`'d) — plus the wake-stub text. `onEvent` receives every instance event: `_onTurnEnd` **defers** delivery while the target's `activeAgentTaskCount > 0` (keeping the subscription + watchdog armed), so a wake fires only once the worker's turn ends with no live background subagents; `_onTaskEvent` covers the drain-at-idle-with-no-re-invocation case; a subagent that never finishes is caught by the watchdog. Cross-instance lookups + `subscription_changed` emission route back through the manager. `InstanceManager` delegates the `subscribeIdle`/`unsubscribeIdle`/`isIdleCaller`/`idleSubscriptionsOf`/… surface and exposes the live `_idleSubscribers` getter. `subscriptionsOf(callerInstanceId)` (the inverse of `snapshot()`, caller-indexed) backs the `renew_session` mechanical state block (see `src/sessionRenew.js`). `deliver()` picks one of three stubs (`_plainStub`/`_buildFoldedStub`): a real `turn_end` to an already-idle caller **folds** the worker's recent output inline (reusing `buildRecentMessages` from `mcp/handlers.js` + `flattenPayload` from `mcp/content.js`, tagged with the `public/wakeCallback.js` marker for the collapsible UI bubble); the timeout-watchdog and live mid-turn steering paths keep the plain pointer stub (a mid-turn wake is injected live into the caller's running turn). No import cycle — nothing in `handlers.js`'s import graph reaches back to `instances.js`.
- **src/overageResume.js** — `OverageResumeController`, the overage auto-resume timer machine extracted from `InstanceManager` as a composed collaborator. Owns the `instanceId → fireAtMs` (wall-clock epoch-ms **deadline**) map (**keyed by the stable instanceId**, so a due-tick resolves the Instance by a direct `byId.get(id)` rather than a scan) and `AUTO_RESUME_TEXT` (re-exported from `instances.js` for back-compat); the per-Instance overage flags (`autoResumeAt`/`autoStoppedForOverage`/`_overageHandled`/`_overageResetsAt`) stay on the Instance. Firing is driven by a **single shared wall-clock sweep interval** (`ORCH_OVERAGE_RESUME_SWEEP_MS`, default 30 s, `unref()`'d — started on first arm, stopped when the map empties), **not** one `setTimeout` per session: each tick re-reads `Date.now()` and fires every deadline that is now due, so a deadline that lapsed while the process was suspended (Android Doze / Termux backgrounding freezes the libuv monotonic clock a `setTimeout` rides) fires on the first tick after wake. `resetsAt`-based scheduling, deadlines in-memory but persisted in the resume manifest (survive a graceful restart), never kills/respawns. **Usage-verified fire:** a due deadline does NOT resume on our clock alone — the tick does ONE coalesced `fetchUsage()` (`getAccountUsage`, injectable seam; its 180 s cache + backoff absorb the pressure — one fetch shared across all sessions due that cycle, guarded by `_ticking`) and only resumes when `usageOverThreshold(usage) === false` (five-hour util `< 100` and under any enabled threshold — the SAME evaluator the poll trip uses). Still-over → `_reschedule` to `max(resetsAt, now + ORCH_OVERAGE_RECHECK_MS)` (also pushes the manager's global `_overageResetsAt` forward so the lockout gate's future-resetsAt rail stays satisfied while parked), naturally polling ~every minute until the account clears — which is what kills the boundary re-trip (no resume turn is ever sent while still throttled). Can't-confirm (null/backoff/malformed) reschedules too, up to `FAIL_OPEN_AFTER`=5 consecutive failures, then fails open and resumes so a persistently-down usage API can't park a session forever. `arm()` no longer dead-ends on a missing/past `resetsAt` — it schedules the first check at `now + recheck`; **`auto_resume_skipped`** now fires ONLY for the genuine "process gone" case in `run()`. `armRestored` (restart) re-inserts its persisted deadline as-is (keyed by the freshly-created Instance's new instanceId — the manifest persists the deadline by sessionId, and restore re-arms off the live restored `inst` after `create()`) and inherits the same usage-verified fire. `InstanceManager` delegates the `_armAutoResume`/`_cancelAutoResume`/… surface (also called from the temp-session exit branch so a disposed temp session never leaves an orphan deadline) and exposes the live `_autoResumeTimers` getter. **Message queueing:** while a session is auto-stopped-and-armed, a non-internal `Instance.prompt()` is intercepted **before** `emit('user_prompt')` and pushed onto `inst._overageQueue` (`{text, attachments, ts}`) with an `overage_message_queued` UI event — it does NOT resume or cancel. `run()` flushes on fire: it snapshot-empties the queue, calls `cancel(sid)` FIRST (clears `autoStoppedForOverage`/`autoResumeAt` so the follow-on non-internal resume send isn't re-queued — `cancel` is idempotent so the send's own `user_prompt→cancel` is a no-op), emits an `auto_resume` system line, then sends ONE combined `prompt()` (`buildCombinedResumeText` = `AUTO_RESUME_TEXT` + numbered list, with all queued attachments concatenated). `cancel()` also clears `_overageQueue` (kill/remove/temp-exit/shutdown cleanup). `summary()` exposes `queuedCount` (rides every status emit → header/sidebar badges append `· N queued`). `fireNow()` is the sweep-internal + test seam and is also the primitive `InstanceManager.reevaluateOverageResumes()` drives (one `fireNow` per parked instanceId, snapshotting `timers.keys()` first since `fireNow` synchronously deletes its own entry) — called after a Settings → Models Apply that raises/disables the threshold, so a session parked under the OLD bar resolves against the new one immediately instead of waiting out its deadline. There is still **no** early-resume UI/button — this is a settings-save side effect, not a user-facing control.
- **src/usageWindowDomains.js** — the backend-scoped **usage-window domain** seam that decides whether an instance is subject to the overage flow at all. A usage/rate-limit window is a provider concept: `BACKEND_USAGE_DOMAIN` maps `claude→'anthropic'`, `ollama→'ollama'`, and `MONITORED_DOMAINS` (today just `{'anthropic'}`) names the domains with an active monitor. `InstanceManager` composes three helpers on top: `agentTreeBackends(inst)` (cycle-safe walk of `inst` + its conducted-worker descendants by `callerInstanceId` → `Set` of backendKinds; in-process Agent-tool subagents share the parent CLI's launch-fixed backend so they add nothing), `usageWindowDomainsOf(inst)`, and `_inUsageWindowFlow(inst)` (true iff the tree touches a monitored domain). A **purely-Ollama** tree → `{ollama}` → exempt; a tree with any Claude agent (incl. an Ollama conductor with Claude workers) → in-flow. Shipping an Ollama usage monitor later = add `'ollama'` to `MONITORED_DOMAINS`; no exemption logic changes.
- **src/usageOverageMonitor.js** — `UsageOverageMonitor`, the server-side usage poller composed by `InstanceManager` as a second, equal-footing overage-trip source alongside the stream `rate_limit_event`. On an `ORCH_USAGE_POLL_MS` interval (default 180 s; `unref()`'d), `_tick()` is heavily gated (skip if `_overageActive`, threshold disabled, `getOnOverageAction()==='none'`, or nothing mid-turn — so it doesn't fetch needlessly), then calls `getAccountUsage()` (injectable as `fetchUsage` for tests; a throw or null is swallowed → no false-trip) and, when the **five-hour** window's `utilization` (a 0–100 percent) `>= overageThresholdPct`, calls `manager._handleOverageTrip(null, { resetsAt })` — the *same* seam the stream trip lands in. It writes no parallel stop logic and injects no prompts (routing's `internal:true` prompts are inherited), so dedup (`_overageActive`), conductor/conducted/plain routing, and resume all behave identically regardless of source. `start()`/`stop()` are idempotent; the server starts it after `listen()`, and both `shutdown()` and `shutdownForResumeSync()` stop it. The point: the stream event only reports near Anthropic's own ~90% mark, so a *low* threshold (e.g. 25%) is invisible to it — only the poll catches it. `parseResetEpochSecs` is exported from `instances.js` and reused here to turn the bucket's ISO `resets_at` into epoch seconds. `forceTick()` is a thin public passthrough to `_tick()` — the same gates apply — used by the Settings → Models Apply handler (`src/routes.js`) so a *lower* threshold that's already crossed trips a mid-turn session immediately instead of waiting up to `ORCH_USAGE_POLL_MS` for the next scheduled tick.
- **src/sessionRenew.js** — `SessionRenewController`, the managed session-renewal machine composed by `InstanceManager` (the `renew_session` MCP tool). Owns a `instanceId → {state, opts, oldSid, timerId}` map. **Keyed by instanceId, not sessionId**, because renewal's whole job is to rotate the caller's sessionId out from under it — only the stable `byId` key survives. `arm(instanceId, {summary})` (idempotent; re-arm refreshes opts) registers a pending renewal; `onEvent` acts on `turn_end` for a pending instance via a two-state machine: `armed` → the caller's next `turn_end` drives `Instance.clearContext()` (a server-managed `/clear`, see protocol.md) and remembers the pre-clear sessionId → `clearing`; `clearing` → the first `turn_end` where `inst.sessionId` has rotated off the pre-clear id (i.e. `/clear`'s own turn_end, after its `system/init`) builds the mechanical state block (`buildStateBlock(manager, id)` — **at this point, not at arm time**, so it reflects current state) and injects `buildRenewSeed({...opts, stateBlock})` as the cleared session's first user turn (`prompt(..., {internal:true})`). Unlike the instanceId-keyed side structures, the **sessionId-keyed** durable markers (`temp`/`conducted`/title) DO need carrying: `clearing` calls `inst.carryMarkersAcrossRenewal(oldSid)` (fire-and-forget, best-effort) which marks the rotated id **first** (`markTemp`/`markConducted`/`setTitle` per the flags the instance carries — closing the window before the reseed `turn_end`'s incidental `_writeSessionMetadata` re-write, and carrying the title, which no `turn_end` path writes) then retires the abandoned old id **last** (`unmarkTemp` + `markArchived`, mirroring `_archiveTempSession` — conducted/title stay meaningful on the archived row); new-id-first ordering means a crash mid-way can't strand the new id unmarked while the old id is archived. No side-structure migration is needed — the idle-subscription graph and overage timers are keyed by the stable instanceId, which `/clear` preserves. A `turn_end` that has NOT rotated (a mid-turn-queued user turn the CLI ran before `/clear` took effect) is ignored, so an intervening turn can't reseed into the wrong session. The `armed` state **defers** the clear while the caller has an overage-queued turn (`_overageQueue.length`), live subagents (`activeAgentTaskCount`), or an owed re-invocation (`taskNotificationPending`) — so queued/background work is never stranded by the rotation (mirrors the idle hub's defer gate). A `.unref()`'d `ORCH_RENEW_CLEAR_TIMEOUT_MS` (default 60 s) watchdog abandons a renewal that never rotates rather than wedging the instance. `buildRenewSeed({summary, rolePreamble?, replayPrompt?, stateBlock?})` composes the seed text — phase 1 used only `summary`; `rolePreamble`/`replayPrompt` are seams for later phases (conductor-role injection, last-prompt replay), `stateBlock` is the mechanical state block. `buildStateBlock(manager, callerInstanceId)` enumerates `manager.liveOwnedBy(callerInstanceId)` (live instances the caller spawned, via the same `callerInstanceId` tracking the sub-agent panel uses — `{sessionId, project, worktree, status}`) and `manager.idleSubscriptionsOf(callerInstanceId)` (the caller's pending idle-subscription targets, an inversion of `IdleSubscriptionHub.snapshot()`) — framed as a safety net: it wins for **existence**, the self-authored summary wins for **intent**. `InstanceManager` delegates `armSessionRenew`; `remove()` calls `_sessionRenew.purge(id)`.
- **Overage trip detection + central routing** (in `src/instances.js`) — the *decision* of what to do on an overage trip moved off the per-Instance handler (which can't reach the idle-subscription graph) up to `InstanceManager`. Two sources reach `_handleOverageTrip`: the per-Instance stream path below, and the `UsageOverageMonitor` poll (above) — deduped by the global `_overageActive` one-shot so they never double-trip (whichever observes the crossing first wins). Flow: `Instance._isOverageTrip(data)` (hard `isUsingOverage` flag OR, when `getOverageThreshold().enabled`, `utilization >= pct/100` for any window) → `Instance` emits an `overage` signal (throttled per run by `_overageHandled`) → `InstanceManager._handleOverageTrip` holds a global one-shot `_overageActive`/`_overageResetsAt`/`_overageClearTimer`, returns early when `getOnOverageAction()==='none'`, else `_routeOverageStop` (steer-first: `_steerConductor` via `windDown`/`prompt` with `overageConductorSteerText({hasWorkers})` for in-control conductors of conducted workers, `_directOverageStop` via `interrupt()` for everything else still mid-turn — **except** the Conduct orchestrator (`project === '.conduct'`), which is ALWAYS steered gracefully even with no in-control workers to halt, `hasWorkers:false` dropping the worker-halting sentence so it isn't sent chasing phantom workers; owner resolved by `_ownerConductor` = `byId.get(worker.callerInstanceId)`), and arms `_armOverageClear(resetsAt)`. **Global release is unified with the per-session rechecks:** the clear timer's callback is now `_maybeReleaseOverageLock` (a BACKSTOP for the never-armed case), which clears the global flag ONLY when nothing is parked (`_overageResume.timers.size === 0`) AND no live instance is still `autoStoppedForOverage` (a session soft-interrupted but not yet arrived at idle will arm imminently — closes the past-resetsAt backstop race). The resume controller calls `_maybeReleaseOverageLock` after every deadline-removing outcome (verified resume, fail-open resume, process-gone teardown), so the lockout lifts exactly when the last parked session resolves — the release and the resumes can't disagree because they're the same usage-verified sweep. `_clearOverage` (backstop fire, that release path, or manual takeover of an `autoStoppedForOverage` session via the `user_prompt` handler) releases the flag and resets every instance's `_overageHandled`. For `stop-resume`, **both** conductor-steer branches arm resume — mid-turn (`windDown`, flags set before) and idle+subscribed (`prompt`, flags set *after* the call since `prompt()` emits `user_prompt` synchronously) — alongside `_directOverageStop`; the timer arms on the steered/stopped session's next status→idle. Resume timing keys off the five-hour `resetsAt` (read camelCase epoch from the live event by `parseResetEpochSecs`), **not** the far-future `overageResetsAt`. Orchestrator-injected prompts — the idle-subscription wake stub and the conductor overage steer — pass `prompt(..., {internal:true})` so their `user_prompt` does **not** cancel a pending resume (an overage-stopped conductor still gets woken when its worker finishes); only a genuine user/MCP prompt, or the auto-resume's own non-internal fire, runs `_cancelAutoResume`. **Domain-scoped exemption:** every routing decision gates on `_inUsageWindowFlow(inst)` (see `usageWindowDomains.js`) so a session whose agent tree is purely in an unmonitored domain (e.g. Ollama-only) sits entirely outside the flow: `_handleOverageTrip` ignores a stream trip from such an instance (the `null`-inst poll trip is account-global and unaffected); `_routeOverageStop` filters those instances out of `live` (never stopped/steered/marked); the per-instance `_overageGate` returns `active:false` for them (so their sends never queue **and** `summary().overageActive` stays false → no overage badge); and `resumeRestart.js` skips re-arming a restored deadline onto one. Arming is thus unreachable for exempt instances without extra guards.

- **src/hookBroker.js** — Per-instance broker for the PreToolUse http hook, pending-response map keyed by `tool_use_id`, 540 s timeout.
- **src/settings.js** — Builds the inline `--settings` JSON. Pure values → JSON string; no Instance state.
- **src/transcript.js** — Replays `~/.claude/projects/<encoded-cwd>/<sid>.jsonl` into UI-event shape on resume; best-effort `last-prompt`/`permission-mode` appends; exports `isPureUserPromptLine`.
- **src/eventArchive.js** — `pageInstanceEvents(inst, {before|after, limit})` behind `GET /api/instances/:id/events`: retained events come from the ring; evicted ones are reconstructed by replaying the jsonl (`loadPersistedTranscript`) into a dense archive seq space (0..H-1, clamped strictly below `trimmedBefore`), cut at the ring head's stamped `userIndex` so ring + archive never overlap (the echo ordinal is the only correlator between the two seq spaces — the stitch stays turn-anchored even though page seams are quiescent; mid-turn head ⇒ gap, never duplication, marked with a seq-less `history_gap` event spliced before the first ring-side event). Backward pages return an opaque `nextBefore` cursor that jumps the numeric gap between the two seq spaces; window starts snap to **quiescent points** (`snapStartToQuiescent`, parser.js: no open outer block, every `tool_use` resolved; outer `user_echo`/`turn_end` force-reset the scan; sub-agent events are scan-opaque — nested-block wholeness comes from the group-integrity pull, which drags a child's Task head and thus the whole group into the page; the archive→ring seam is passed as `resetIdx` so state is never scanned across missing events). Also splices synthetic `task_completion` bubbles into paged history (see `taskReconstruct.js`).
- **src/taskReconstruct.js** — `reconstructTasks(events)` → `{completions:[{afterSeq,tasks}], activeAtEnd}`: single-pass, chronological reconstruction of task-batch state from raw `TaskCreate`/`TaskUpdate` `tool_use`/`tool_result` events. Server-side mirror of the client `public/tasks.js` `TaskTracker` (same create-id binding, batch rollover, and completion edge — keep the two in sync). Feeds the snapshot's `tasksAtTailStart` (via `Instance.reconstructActiveTasks`) and `eventArchive.js`'s lazy-page `task_completion` injection. `task_completion` is never a real wire event — the tail/live path still synthesizes it client-side in `wsRouter.js`; this recovers the below-tail batches those paths can't see.
- **src/sessionEdit.js** — Atomic destructive jsonl edits: `truncateSessionAtUserMessage` (tmp → fsync → rename, appends fresh metadata at the new leaf), `forkSessionAtUserMessage` (copies prefix into a new `<sid>.jsonl` with `sessionId` rewritten on every line). Both return `droppedText` for composer prefill.
- **src/parser.js** — stream-json line → UI event normalization. Delta merging by `(msgId, blockIdx)`, `thinking_redacted` emission, `parentToolUseId` routing, structured `user_question` / `plan_request` events.
- **src/projects.js** — FS ops on `<projectsRoot>/`, cwd encoding for `~/.claude/projects/`, CLAUDE.md seeding.
- **src/conductedSessions.js** — sidecar store (`<store>/conducted-sessions.json`) of sessionIds spawned via MCP `spawn_instance` (the worker agents an orchestrator conducts). `loadAll()`/`isConducted(sid)`/`markConducted(sid)`/`unmarkConducted(sid)`, atomic write + serialised write-chain, mirroring `sessionTitles.js`. The durable half of the conducted axis.
- **session sidecar stores** — sibling `<store>` JSON sidecars sharing the same atomic-write + advisory-lock shape as `conductedSessions.js`: `sessionTitles.js` (`{sessionId: customLabel}` — `getTitle`/`setTitle`/`deleteTitle`), `sessionBackends.js` (Map `{sid: taggedModel|null}` of Ollama-backed sessions — `isOllamaSession`/`getOllamaSession`/`markOllamaSession(sid,model)`/`unmarkOllamaSession`; the durable marker that survives every resume path — written at spawn, read in `_doCreate` on resume to recover `backendKind` **and** the model tag, carried to the new sid in `carryMarkersAcrossRenewal`. It carries the model TAG because the inner CLI records `message.model` bare in the jsonl, so the tagless jsonl value can't rebuild `ollama launch --model <tag>`; a `null` entry is tag-unknown (legacy, migrated from the old set form by `0018`) and falls back to the jsonl. Claude sessions store nothing), `archivedSessions.js` (archived-sid set — `isArchived`/`markArchived`/`unmarkArchived`), `tempSessions.js` (durable temp-session markers that survive SIGKILL so a temp stays resumable across the resume-after-restart path). `storeLock.js` provides the cross-process advisory lock (`withLock()`, `O_EXCL` create) they serialise on: a held lock is reclaimed **only when the owner PID is dead** — a live owner is never evicted on age, no matter how slow. This is load-bearing on Termux: post-restart CPU throttle (~10–75×) can starve a live holder for many seconds mid-write, and the old-server/new-server overlap during a restart is the only window with cross-process contention; the previous 10 s age-based eviction reclaimed the live holder there, letting two writers run the read-modify-write concurrently and drop each other's entries (a lost update) — which silently un-archived a subset of sessions on every restart. A slow holder is now waited out via bounded retries (a truly wedged holder throws on acquire rather than corrupting), and release is **ownership-token-checked** so a process can't delete a successor's lock. `archivedSessions.js` additionally guards against a **silent full-reset** under the host's frequent OOM/low-disk churn: it keeps a rolling `.bak` (refreshed under the lock only from a non-empty write that is a **superset** of the current backup — never a shrink — so a spurious empty/tiny base can't canonize over the last-good snapshot; consequently `.bak` may lag behind un-archives by design, the safe direction: a recovery from a stale `.bak` only re-archives a few since-unarchived ids, never loses data); a missing primary recovers from `.bak` (missing primary **and** `.bak` = legitimately empty); a corrupt primary is quarantined to `.corrupt-<pid>-<ts>` (under the lock, in the mutation path) and recovered from `.bak` rather than bricking archiving; and a drain-to-empty now writes `{"sessions":[]}` atomically **instead of unlinking**, so an absent primary unambiguously means external loss. Only a genuine unrecoverable I/O error (EACCES etc.) still aborts a mutation. `tempCleanup.js` writes/sweeps `<store>/pending-temp-cleanup.json` to GC temp jsonl orphaned after exit; it and `resumeManifest.js` write their sync-before-exit manifests via tmp-write + `renameSync` so an OOM mid-write can't leave a torn manifest.

Cache-miss detection is **cross-turn**, captured live in `src/instances.js`, and catches *partial* (minority) evictions a stateless single-request rule cannot. Each turn is one or more API requests; `message_start` carries per-request cumulative usage and `cache_read` accumulates across a turn's tool-call iterations (there is no per-iteration array — `parser.js` passes `result.usage` through verbatim). So the detector uses two points: `read_N` = this turn's **first**-request `cache_read` (`_turnFirstReqCacheRead`, pre-tool-call); `P` = the **previous** turn's **last**-request full prefix `cache_read + cache_creation` (`_prevTurnPrefix`), captured by overwriting `_turnLastReqPrefix` on every `message_start` and latching it at `turn_end`. **Miss ⇔ `read_N < P − max(1024, 1% of P)`** — this turn served less of the prefix than was cached at the end of last turn ⇒ eviction (full or partial); a warm continuation reads exactly `P` (drop 0, never flagged), the tolerance absorbs tokenization-boundary noise. **Turn 1** (`_prevTurnPrefix === null`) and any turn where `_prefixBaselineInvalid` is set fall back to the stateless `creation > read` rule, which still flags a genuinely cold prefix (cold start, expiry, resume, cold rewind). **Guards** against legitimate prefix shrink: compaction (`system`/`compacting` in `_handleStdoutLine`), model switch (`_trackModel`'s real-switch branch — cache is model-specific), and rewind/respawn (`_wipeForResume`) each set `_prefixBaselineInvalid`; the next turn's first request consumes it (forces the fallback, re-establishes `P` fresh) so the cross-turn rule never false-fires. **Known limitation:** a miss on a request *after* the first (e.g. a turn outliving the ~1h cache TTL) isn't caught — first-request-only. Per-turn capture (`_turnFirstReqCacheRead`/`Creation`/`_turnMissDetected`/`_turnLastReqPrefix`/`_turnEvicted`) resets in `_setStatus`'s into-`turn` branch and on (re)spawn; cross-turn state (`_prevTurnPrefix`, `_prefixBaselineInvalid`) intentionally persists across the turn boundary. Each `turn_end` is enriched with `cacheMiss`/`firstReqCacheRead`/`firstReqCacheCreation`/`firstReqEvicted` (`P − read_N` on the cross-turn path), persisted as `cache_miss`/`first_req_cache_read`/`first_req_cache_creation`/`first_req_evicted` (additive), and an informational `system`/`cache_miss` notice fires once per miss turn. The old read-time session-relative heuristic (`cache_creation_tokens >= max(50000, 4 × session median)`) was retired to a one-shot backfill, **migration 0014**, which stamps `cache_miss` on legacy rows; a row still carrying the old `cache_flush` key (a decisive verdict captured before the field was renamed) is renamed in place rather than recomputed, so historical rows are heuristic-or-renamed while new rows are decisive. Migration 0014 additionally stamps `cache_miss_heuristic: true` on any row whose verdict IT decided (neither key present, or a renamed `cache_flush` row lacking `first_req_cache_read` — an old-migration guess, not a live capture); presence is checked, not truthiness, since a decisive row can legitimately have `first_req_cache_read: 0`. `appendCostRow` never sets this field, so its absence always means "decisive."

## Conventions

Load-bearing rules this codebase follows — keep new work inside them so the project stays manageable as it grows.

1. **YAGNI — build for the requirement in front of you, not the one you imagine.** Implement only what a current, concrete caller or test needs; don't add speculative abstractions, config knobs, or extension points "for later" — they add surface area without a real user, and the eventual real requirement rarely matches the guess. If code stops being exercised by a real caller or test, delete it rather than keeping it "just in case" or "to trim later" — dead code rots silently and misleads the next reader into thinking it's load-bearing.
2. **Frontend: `installX({…})` modules; `app.js` is wiring-only.** Feature logic lives in a `public/` module exposing an `installX({…getters,…callbacks})` factory (or a class for a stateful widget). `app.js` only builds `state`/the DOM map, calls each `installX` once, holds the returned handles, and wires `wsRouter` last. Modules read live state through injected getters (`getActiveId: () => state.activeId`) — they never reach into globals. Don't grow `app.js` with feature logic; add a module and wire it.
3. **No god-modules — extract composed collaborators.** When a module takes on a second responsibility, lift it into a collaborator the owner *composes*, keeping the public surface stable via delegation. Cf. `InstanceManager` → `IdleSubscriptionHub` + `OverageResumeController`; `mcp/handlers.js` → `diffPaging` + `messageReconstruction` + `content`; whisper/tts install → the `makeInstallRunner` factory.
4. **One service layer for REST + MCP — don't reimplement per surface.** Git / worktree / diff / session logic lives once in `src/worktrees.js` (+ siblings) and is imported by **both** `src/routes.js` and `src/mcp/handlers.js` (shared: `isGitRepo`, `mergeWorktreeIntoParent`, `syncWorktree`, `getWorktree`, `getWorktreeMergeStatus`, `buildRebasePrompt`, `removeWorktree`, …). A new git/worktree op goes in the service layer and is called from both surfaces — never duplicated.
5. **Single-source-of-truth catalogs, shipped to the client — never client literals.** Curated catalogs (`modelVersions.js`, `whisperModels.js`, `ttsModels.js`) own the authoritative list + allow-list + policy server-side; the client *fetches* them (`/api/settings/models` → `public/models.js`) instead of hardcoding the canonical set. (`public/models.js` keeps a `DEFAULT_VERSIONS` first-paint fallback — a fallback, not a second source of truth.)
6. **Cross-surface magic strings get one owner module.** A canonical string shared across surfaces lives in a single module. Cf. `planApproval.js` — the approve/reject prompts shared by the UI buttons, the server-side auto-approve, and the MCP `approve_plan`/`reject_plan` tools; `public/userQuestionAnswers.js` — the AskUserQuestion answer formatter shared by the UI question card and the MCP `answer_question` tool (a DOM-free module under `public/` that the server imports directly, avoiding a mirror); `public/wakeCallback.js` — the idle wake-stub marker (`WAKE_CALLBACK_MARKER`/`WAKE_BODY_SEP`) shared by the server (build) and the client (parse/render); `SOFT_INTERRUPT_MARKER` in `parser.js`; `isTaskNotificationContent` in `parser.js` — recognizes the CLI's re-injected `<task-notification>` tag by shape (no marker to inject, since the CLI produces the string) and is checked at the same three sites as `SOFT_INTERRUPT_MARKER`.

## Instance lifecycle
```
create → spawning → idle ←─ turn ─→ turn_end ─┐
            ↓        ↓                          ↓
        load-hist  prompt                  exited / crashed
        fails                                   │
                                                ▼ respawn --resume <sid>
                                            (back to spawning)
```

On **resume**: `loadHistory(sessionId)` runs before flipping to `idle` — replays jsonl into UI events and emits a `system/history_replayed` divider. A history longer than the ring cap trims during the replay itself; the evicted prefix stays reachable via `GET /api/instances/:id/events`. On **turn end** or **mode change**: append `{"type":"last-prompt", …}` + `{"type":"permission-mode", …}` lines so `claude --resume`'s interactive picker can discover the session.

**`displayStatus` overlay (additive, not a 6th state).** The diagram above is the real `status` enum — untouched. A backgrounded `Agent` tool call's tool_result resolves immediately (`isAsync:true`), so `turn_end` (→ `idle`) can fire while the CLI's own `system/task_started`/`task_updated` task-tracking is still open for that call. `Instance._activeAgentTasks` (a `Map<task_id, tool_use_id>`) tracks this: incremented on `task_started` **unless** `task_type === 'local_bash'` (backgrounded/promoted Bash — a promoted long-lived command never emits a terminal event and would pin the count forever; unknown `task_type`s still count), decremented on a terminal `task_updated` (`patch.status` one of `completed|failed|cancelled|error` — an unrecognized status is treated as non-terminal on purpose, to avoid flipping back to idle early) or on `task_notification` (belt-and-suspenders). `summary().displayStatus` reads `'running'` instead of `'idle'` while the map is non-empty; every other `status` value passes through unchanged. Reset on (re)spawn. `Instance.activeAgentTaskCount` exposes the live map size. `IdleSubscriptionHub._onTurnEnd` **gates on TWO conditions**: a `turn_end` **defers** the idle wake (the subscription + its watchdog stay armed) while **either** `activeAgentTaskCount > 0` (a subagent is still running) **or** `taskNotificationPending` is set — a `task_notification` fired *during* the turn now ending **and no top-level tool_result followed it**, which per the CLI's queue semantics means the notification is still queued and an unprompted **re-invocation turn** will deliver it. Both cleared, the wake fires — so a wake means "the worker AND all its subagents finished, and no re-invocation turn is still owed." `Instance._taskNotificationPending` (getter `taskNotificationPending`) mirrors the CLI's stdout-invisible message queue by event order: SET in the `task_notification` handler only when `status === 'turn'` (a completion while *idle* re-invokes immediately — that turn is its processing turn); CLEARED by any top-level (`parentToolUseId == null`) `tool_result` event — a sync-delivered task's own held-open result (fast Agent calls, long Bash promoted to a task, e.g. a full test run) or the outer tool round-trip the CLI attaches queued notifications to (batched flush) — by every idle→turn transition in `_setStatus` (the dequeued notification IS the new turn's input), and on (re)spawn. Nested (subagent-forwarded) tool_results never clear it. Every status-gated frontend control still keys off the real `status`; the cosmetic surfaces (sidebar dot, sub-agents strip, header chip) read `displayStatus`.

**Idle task-drain settle (the third wake path).** A background task can finish while its worker is ALREADY idle with **no re-invocation turn ever following** (verified trace: a subagent's nested Monitor completed after the subagent — the stream's last lines were `task_updated` + `task_notification`, no `turn_end` ever re-sampled the gate, and the subscription orphaned until the watchdog). `IdleSubscriptionHub._onTaskEvent` runs on every `task_updated`/`task_notification`: when a watched target is fully drained (`status === 'idle'`, `activeAgentTaskCount === 0`, `!taskNotificationPending`, `!idleWindowDirty`) it arms a **settle timer** (`ORCH_IDLE_DRAIN_SETTLE_MS`, default 5 s) recording the target's proc identity and `ring.nextSeq`; `_fireSettle` delivers the wake only if the subscription still exists, the SAME proc is live and still fully drained, and `ring.nextSeq` is **unchanged** — literally zero events since the arming task event. The trap this must not reintroduce is waking ONE TURN EARLY: an idle completion that DOES get a re-invocation turn announces it with CLI-local `system/init` + `system/status` writes within ms of the notification, while the `message_start` that flips `status` to `'turn'` waits on the API (seconds — no settle length can safely outwait it). Re-invocation evidence is therefore caught structurally on both sides of the drain event: **after the arm** by the frozen-seq check (init/status advance the ring before the timer fires → drop, the re-invocation's `turn_end` owns the wake), and **before the arm** by `Instance._idleWindowDirty` (getter `idleWindowDirty`) — SET when any non-task-lifecycle event (`TASK_LIFECYCLE_SUBTYPES` = task_started/updated/notification) is processed while `status === 'idle'`, CLEARED at `turn_end` and on (re)spawn — covering a task that drains *inside* another task's already-opening re-invocation window. Every failed re-check drops the settle silently (subscription + watchdog stay armed): the fail direction is always "wake later via turn_end/watchdog", never "wake early" or "wake twice". The proc pin exists because respawn/rewind reuse the Instance AND its id but mint a new proc and reset the ring (`_wipeForResume`), making cross-run seq comparisons meaningless; history replay can't arm (it runs while `status === 'spawning'` and always trails a seq-advancing `history_replayed` event) and can't corrupt the dirty flag (replay bypasses `_handleStdoutLine`). `_justConsumed` stays turn_end-only — no worker `turn_notification` exists at settle-fire time. Accepted residuals: a CLI stalling longer than the settle between the notification and init writes (consecutive process-local writes, normally sub-ms — the window never scales with API latency) would wake one turn early; non-task noise in an idle window with no turn following falls back to the watchdog, same as before this path existed.

## Migrations
`migrations/` holds idempotent migration scripts run automatically on boot (entrypoint `migrations/index.mjs`). Each self-checks "already applied?" and is a fast no-op in steady state; a script that throws **aborts the boot**. See [`migrations/migrations.md`](../migrations/migrations.md) for the listing and conventions for adding new ones.

## Testing
All tests via `node tests/run.mjs` (programmatic node:test runner — Termux's glibc-runner wrapper hoists leading `--flags` into `NODE_OPTIONS` and refuses `--test`).

Test **files run in parallel** (node:test `isolation:'process'` — one child process per file). The suite is isolation-safe: `bootServer` binds an ephemeral port (`listen(0)`) and `mkdtemp`s a unique home per server, so files never contend over ports or paths. Concurrency defaults to `min(4, cores/2)`. Override with `TEST_CONCURRENCY=<n>` (`1` = fully serial). Measured ~79s serial → ~29s at the default on an 8-core device.

**Store isolation (`tests/safeStoreRoot.mjs`).** The sidecar stores resolve their on-disk root from `PROJECTS_ROOT`, falling back to a source-relative default (`src/../..`) that IS the real production `.code-conductor` when it's unset — so a test touching a store with `PROJECTS_ROOT` unset corrupts the live store. `run.mjs` pins the whole run to a throwaway temp root (`createSafeRoot()`) before any file forks — every child inherits it — and asserts via `assertStoreIsolated(orchStoreRoot())` that the store never resolves into the real workspace (aborting the run if it would). `bootServer` teardown restores `PROJECTS_ROOT`/`CLAUDE_PROJECTS_ROOT` to that safe root, never to unset, so out-of-window writes stay in the temp dir. `tests/store-isolation.test.mjs` is the regression guard.

**ClaudeLauncher seam (in-process fake by default).** `Instance` never calls `child_process.spawn` directly — it spawns through an injected `ClaudeLauncher.launch({command,args,cwd,env})` (`src/claudeLauncher.js`), threaded `createServer({claudeLauncher})` → `new InstanceManager({claudeLauncher})` → `Instance`. Production default is `RealClaudeLauncher` (a raw `child_process.spawn` — behavior byte-for-byte unchanged). `src/claudeLauncher.js` also owns the two pure launch-resolution primitives — `resolveClaudeBin()` (splits `CLAUDE_BIN`, default `claude`, into `{command,prefixArgs}`; test injection via `CLAUDE_BIN="node fake.mjs"`) and `resolveBackendLaunch(backendKind, model, claudeBin)` — so one-shot spawners that need no `Instance` (`health.js`'s boot probe, `summarize.js`'s summary generation, `claudeShellEnv.js`'s bundle-gen) can import just these without pulling in the 2000+-line `instances.js`. **Backend-agnostic spawn:** `resolveBackendLaunch` computes only command + prefix from `backendKind`; every caller then appends the same claude args (including `--model <model>`) uniformly. `claude` → the passed-in `claudeBin.{command,prefixArgs}` unchanged; `ollama` → command from `resolveOllamaBin()` (mirrors `resolveClaudeBin`'s `CLAUDE_BIN` convention via `OLLAMA_BIN`, default `ollama`), prefix `[...ollamaPrefixArgs,'launch','claude','--model',<model>,'--yes','--']` (the tag also rides in the forwarded `--model` — a harmless no-op duplicate). The helper itself owns the **null-model invariant** — it throws if `backendKind==='ollama'` and no `model`, rather than emit `--model undefined` — so `Instance.spawn()` wraps the call in a try/catch only to add its own `_setStatus('crashed')` side effect before rethrowing. No `OLLAMA_HOST` / host plumbing (localhost only). Only `model` + `backendKind` thread `create()→_doCreate→Instance ctor→spawn()`; both appear in `summary()` (so the resume manifest carries them) and survive an in-place `/clear` renewal. On resume, `backendKind` is recovered from the `session-backends.js` marker; `model` is recovered from that same marker's tagged value for ollama sessions (the jsonl `message.model` is bare — tag dropped by the inner CLI — so a jsonl-only recovery would relaunch the unpullable tagless name), falling back to the jsonl (`readLastSessionModel`) for Claude sessions and legacy tag-unknown ollama entries. Reachability is preflighted by `src/ollamaBackend.js` before a custom model is added **and** before every ollama-backed spawn/respawn/bundle-gen/summary-gen (`InstanceManager._preflightBackend` for spawn/respawn — run in `_doCreate` before the `Instance` is built and in `respawn()` before history is wiped; `claudeShellEnv.js`/`summarize.js` call `preflightOllamaBackend` directly when the Fast capability tier is Ollama-bound); every site throws via the shared `ollamaPreflightError(pre, prefix?)` (`src/ollamaBackend.js`) — `{statusCode:503, code:'OLLAMA_PREFLIGHT_FAILED'}` → REST 503 / MCP isError, with an optional module-name prefix on the message. A crashed ollama subprocess additionally emits a `system/launch_failed` event with captured stderr (`_handleExit`, gated by a `_killing` flag so a commanded kill isn't mislabeled). A live "Change model" (`Instance.setModel(model, backendKind)`) is **blocked** for any Ollama-involved switch (the running endpoint is fixed at launch) with a "kill and respawn" message; only Claude↔Claude live switches proceed. **`bootServer()` defaults to `InProcessClaudeLauncher`** (`tests/inProcessLauncher.mjs`): the fake-claude engine runs on the event loop against in-memory `PassThrough` streams — **zero OS subprocesses per instance**. This is the forget-proof default (an Android phantom-process-killer defense: the app UID is killed above ~32 processes), so a normally-written test forks nothing. The in-process handle is a `FakeChildProcess` (EventEmitter + 3 PassThrough + `pid=null`); it defers engine start to `setImmediate`, gates `'exit'` on the readable-side `'end'` of stdout/stderr (so a trailing `turn_end` is never delivered after `'exit'`), and maps `kill('SIGKILL')` → `exit(null,'SIGKILL')`, `kill('SIGTERM')`/`stdin.end()` → graceful `exit(0,null)`. `pid=null` makes the manager's OS-`process.kill` sync-shutdown paths (`shutdownTempSync`/`shutdownForResumeSync`) a safe skip.

The fake-claude **scenario engine** lives once in `tests/fake-claude-engine.mjs` (`runFakeClaude({argv,env,cwd,stdin,stdout,stderr,onReaderReady})`): silent until first stdin message, auto-acks `control_request`s, emits canned events from `env.FAKE_CLAUDE_SCENARIO`, matches `control_response`s so scenarios can branch on allow/deny. `env.FAKE_CLAUDE_ARGV_DUMP` / `FAKE_CLAUDE_ENV_DUMP` capture the launch argv / computed child env; `FAKE_CLAUDE_TRANSCRIPT` records stdin. It reads everything from its params (not `process.*`), so the same engine drives both the in-process launcher and the thin subprocess entrypoint `tests/fake-claude.mjs`. Config flows through `env` (bootServer sets `FAKE_CLAUDE_SCENARIO`), and for the in-process path `CLAUDE_BIN` is a **bare sentinel** (`claude`, no script path) so `resolveClaudeBin`'s `prefixArgs` stay empty and the argv the engine sees equals the real CLI flag set.

**`bootServer({claudeLauncher})`** injects a custom launcher (e.g. `tests/ollama-spawn.test.mjs`'s `ControllableLauncher`, whose child stays alive until the test triggers a spontaneous `crash()` or a `kill()`) to exercise post-launch crash paths. **`fakeOllamaReachable()` / `fakeOllamaUnreachable()`** (helpers) swap `globalThis.fetch` to simulate a live / absent Ollama daemon at `localhost:11434` (other requests pass through), so tests that spawn ollama-backed workers pass — or deliberately fail — the spawn preflight without a real daemon.

**`bootServer({realProcess:true})`** opts a test back into a real `node fake-claude.mjs` subprocess (`RealClaudeLauncher`) — needed only where a test uses a real `.pid` / OS `process.kill(pid)` / the SIGKILL+reap sync-shutdown path: the `instances.test.mjs` crash+respawn test, and the two `shutdownTempSync` tests (`archive-sessions.test.mjs`, `restart-temp-cleanup.test.mjs`). Each opts in per-test (booting its own dedicated realProcess server) so the rest of its file stays in-process. `shutdownForResumeSync` is graceful (`stdin.end()`), so `resume-after-restart.test.mjs` stays fully in-process.

**Process-count guardrail** (`tests/run.mjs`): samples the peak concurrent `fake-claude.mjs` subprocess count via `/proc/*/cmdline` (~100 ms) and fails the run if it exceeds `FAKE_CLAUDE_BUDGET` (12) — catching a regression that flips the default back to subprocess or a new test that spawns real processes without `realProcess:true`. A clean default run peaks at ~2–3 (only the realProcess opt-in tests). Enforcement is skipped under `RUN_REAL_CLAUDE=1` and when `/proc` is unavailable. (Use `/proc` reads only here — `pkill`/`lsof` misbehave on this host.)

The subagent-aware idle-wake deferral (`IdleSubscriptionHub._onTurnEnd` gating on `activeAgentTaskCount` **and** `taskNotificationPending`) is covered by `tests/idle-subagent-defer.test.mjs` (two layers: hub-gate tests with injected fake instances driving `manager.emit('event', turn_end)`, and Instance-lifecycle tests feeding captured-shape raw stdout lines to a real unspawned `Instance._handleStdoutLine` — sync-delivered result clears, attach-batch clears, nested tool_result does NOT clear, a queued notification survives turn_end until the re-invocation turn start, an idle notification never sets) and end-to-end in `tests/mcp-subscribe-to-idle.test.mjs` via four fixtures — `scenario-bg-task-hang.json` (turn_end while a subagent is live, never completes → wake deferred), `scenario-bg-task-complete.json` (subagent completes *between* turns + follow-up turn_end → single delivery), `scenario-bg-task-midturn-complete.json` (mid-turn `task_notification` with no tool_result after it → wake deferred to the re-invocation turn's turn_end, single delivery), and `scenario-bg-task-midturn-consumed.json` (mid-turn `task_notification` followed by a top-level tool_result → consumed in-turn, wake fires at that same turn_end — regression for the indefinite-defer hang).

The idle task-drain settle path is covered by `tests/idle-drain-settle.test.mjs` (sets `ORCH_IDLE_DRAIN_SETTLE_MS=400` before a dynamic import — the settle length is a module-load-time constant). Three layers: hub-gate tests (fakes extended with `ring.nextSeq` + `idleWindowDirty` — orphan drain wakes once and cancels the watchdog; post-arm seq advance and pre-arm dirty window both refuse the early wake; fire-time drops on status flip / removal / proc replacement; re-arm resets the freeze baseline; turn_end/purge/unsubscribe/watchdog all cancel a pending settle; `wasConsumed` stays false on a settle-fire), Instance-lifecycle tests for the `_idleWindowDirty` edges, and two end-to-end fixtures — `scenario-bg-task-idle-drain.json` (turn_end deferred on a live bg task, then `task_updated`+`task_notification` at idle and silence → the settle delivers the folded wake, exactly once, with no second turn_end ever) and `scenario-bg-task-idle-reinvoke.json` (same drain followed by the re-invocation's `init`/`status`/`message_start`/`turn_end` at 150 ms spacing, so the 400 ms settle deadline falls inside the pre-`message_start` silence → no early wake, single wake at the re-invocation `turn_end`).

Opt-in real-claude smoke (`smoke.real.test.mjs`, `RUN_REAL_CLAUDE=1`): spawns the actual CLI, asserts `system/init` + ≥1 `text_delta` + non-error `turn_end`. Cleans the session jsonl on exit.

The plugin system tests against **`tests/fixtures/fake-plugin/`** — a dependency-free `node:http` child (health, relative-asset HTML, streaming `/echo`, hand-rolled WS echo, MCP endpoint with `echo`/`sleep`/`fail`/`transport-bug` tools pinning both halves of the child contract) plus `crash.mjs`/`slow-ready.mjs`/`forker.mjs` for the supervisor edge cases; `tests/plugin-helpers.mjs` clones it into temp projects roots. `bootServer` exposes `pluginHost` (and `pluginLibrary`) and stops plugin children on close. `tests/plugins-library.test.mjs` unit-tests `createPluginLibrary()` directly (mirrors `plugins-registry.test.mjs`'s direct-construction style) with `_cloneImpl`/`_runHookImpl` test-only overrides standing in for the real `git clone` child process and postClone/postPull commands — no network needed; its `update()` pull-success test uses a small real local git fixture (a bare "remote" + a seed clone that pushes a second commit, mirroring `plugins-registry.test.mjs`'s own local-git-fixture `git()` helper) so the real `runGit`-based `--ff-only` pull is exercised end-to-end. `tests/plugins-api.test.mjs` covers the `/api/plugins/library*` HTTP surface for listing + validation-error paths (invalid scheme, already-installed, not-installed) without ever invoking a real clone or pull.
