## Subprocess protocol
```bash
claude -p \
  --input-format=stream-json --output-format=stream-json \
  --verbose --include-partial-messages --include-hook-events \
  --allow-dangerously-skip-permissions \
  --permission-prompt-tool stdio \
  --permission-mode <plan|bypassPermissions> --effort <effort> --thinking <thinking> \
  --settings '{"hooks":{"PreToolUse":[…]}}' \
  --mcp-config '{"mcpServers":{"code-conductor":{"type":"http","url":"http://127.0.0.1:<port>/mcp"}}}' \
  [--model <name>] \
  --session-id <fresh-uuid> | --resume <existing-uuid>
```

The `--mcp-config` line is omitted when `ORCH_DISABLE_MCP_AUTOREGISTER=1` (no MCP toolbelt auto-registered into the spawned process).

**Interactive tools under CLI ≥ 2.1.x — `--permission-prompt-tool stdio`.** Headless stream-json (`-p`) sessions are non-interactive (`isNonInteractiveSession=true`); the CLI serves a coordinator/agent tool profile and **strips ExitPlanMode / EnterPlanMode / AskUserQuestion** unless the client presents as a permission consumer. Passing **`--permission-prompt-tool stdio`** does exactly that: the CLI routes those tools' permission prompts to us as `can_use_tool` control_requests over the stream-json channel (verified: it re-adds the three tools to `system/init` `tools[]` and they become callable). It is **additive** to `--allow-dangerously-skip-permissions` (normal tools stay auto-allowed and never reach `can_use_tool`; only the interactive tools do). Kill-switch: `ORCH_DISABLE_STDIO_PERMISSIONS=1` reverts to the fail-closed behavior (tools stripped).

The orchestrator answers `can_use_tool` in `Instance._handleStdoutLine`: for `ExitPlanMode`/`EnterPlanMode`/`AskUserQuestion` it replies `control_response {behavior:"deny", message:<AWAITING_INPUT_MESSAGE>}`; any other tool (rare) is allowed. The **deny ends the turn** (`stop_reason: end_turn`, verified) — this is deliberate and load-bearing: the `plan_request`/`user_question` card is emitted from the tool-use, and ending the turn is what lets a conductor's `subscribe_to_idle` wake on `turn_end` and then call `approve_plan`/`reject_plan`/`answer_question`. Holding the request open for an *in-turn* answer would suppress `turn_end` and deadlock that loop, so we do not. `approve_plan`/`reject_plan` are unchanged (setMode + approval prompt); `answer_question` delivers the worker a structured AskUserQuestion answer that is **byte-identical** to a UI question-card submit — both call the one DOM-free formatter in `public/userQuestionAnswers.js` (re-exported by `public/blocks.js`, imported server-side by `src/mcp/handlers.js`). The flag alone exposes and makes the tools callable — **no `initialize`/`supportedDialogKinds` handshake is sent** (probe-verified unnecessary), and the CLI never issues `request_user_dialog` in this configuration (permission flows via `can_use_tool`).

`--allow-dangerously-skip-permissions` is **always** passed at spawn — even for `plan`-mode instances — because without it the CLI rejects any later runtime `set_permission_mode bypassPermissions` (i.e. the plan-approve flow). The flag only *permits* the switch; it doesn't activate bypass on its own.

One `PreToolUse` hook via inline `--settings` JSON (the old static `command` deny on **AskUserQuestion|ExitPlanMode** is **removed** — those tools are now gated at the `can_use_tool` layer, see above):
1. **Edit|Write|NotebookEdit|Bash** — `http` hook → `POST /api/instances/<id>/hook-callback`, **660 s** CLI-side timeout. Orchestrator auto-allows when the mode isn't `ask`; in `ask` mode holds the response open up to **540 s** (deliberately under 660 s — an HTTP timeout would make the CLI treat the hook as a non-blocking error and proceed, the opposite of intent). `ask` is orchestrator-tracked only and maps to `bypassPermissions` at the CLI level; the hook callback inspects orchestrator-side mode to gate.

Inbound: `user` (text or `[{type:"text", text:"..."}, …]` blocks; attachments use `` Attached file: `<rel-path>` ``), `control_request` (`set_permission_mode` / `interrupt`), `control_response` (our reply to the CLI's `can_use_tool`), `keep_alive`. A `user` message written while a turn is running is **delivered mid-turn and interleaved into the live turn** (steering) — not queued until turn_end; the CLI persists such a mid-turn prompt as a `type:"attachment"` `queued_command` line (array `prompt`) rather than a `type:"user"` line.

**Two-tier interrupt.** SOFT (default) injects a hidden steering `user` message suffixed with `[[cc:soft-interrupt]]` (`SOFT_INTERRUPT_MARKER`) telling the model to stop all work and end its turn silently — graceful wind-down, partial work preserved. At send time, `Instance.interrupt()` also emits a live `system`/`soft_interrupted` event with `data.text` set to the steer text — not a `user_echo`, so it never shifts `_userEchoCount`/the rewind-fork index. The marker itself is filtered everywhere it could surface (`parser._handleUser`, `transcript.replayPersistedLine`, `transcript.isPureUserPromptLine`) so on JSONL replay it never renders a user bubble, produces the same bare `soft_interrupted` annotation (no `data.text`) instead, and never shifts the rewind/fork index. FORCED (`force:true`) is the unchanged hard `control_request` `subtype:interrupt` abort (discards partial work). Both gated by `if (status !== 'turn') return`.

**Post-hard-abort drain window.** The CLI's stdin queue is not cleared by a hard abort. Any `user` message written before the abort (the soft-interrupt steer, or prompts sent mid-turn) remains queued; after the abort the CLI dequeues them and starts a spurious new turn for each. After every hard abort the orchestrator opens a drain window (`POST_ABORT_DRAIN_WINDOW_MS`, default 3 s) and watches for `system/init` events on the event channel (`Instance._openDrainWindow`). The `system/init` event arrives ~39 ms after the abort and ~3.7 s before any API round-trip, so the drain fires an immediate `control_request subtype:interrupt` to sever the spurious turn at zero token cost. Each drain resets the window deadline (sliding window). Calling `prompt()` closes the window so intentional follow-up prompts proceed normally. The drain emits a `system/drain_abort` event (kind `'system'`, subtype `'drain_abort'`, data `{count}`) visible in the conversation as "⏹ Drained queued turn after interrupt".

Outbound: `system` + `subtype:"init"` (bundled with first turn's response, not at startup; carries `session_id`, `model`, `tools`, `permissionMode`), `stream_event` (live SSE deltas — primary feed), `assistant` (reconciled message content — one multi-block envelope per message on older CLIs; the async-worker CLI instead finalizes **each content block as its own single-block envelope**, all sharing `message.id`, and persists the same one-block-per-line shape to the session jsonl; `buildMessageFromRing` merges all envelopes of a `msgId`, the UI ignores outer-turn envelopes either way, and disk replay threads a per-file `blockCursor` so a fragmented message's blocks get distinct `blockIdx` values matching the live stream indices. Tools dispatch per-block, so a `tool_result` `user` line can land *between* two envelopes of the same `msgId`. Sub-agent (Task/Agent) turns arrive as finals-only `assistant`/`user` envelopes tagged `parent_tool_use_id` — their partial `stream_event` frames are never emitted (every CLI `stream_event` hardcodes `parent_tool_use_id:null`; the forwarding knob `forwardSubagentText` is SDK-only, no CLI flag)), `user` (`tool_result` blocks), `result` (turn-end with `duration_ms`, `usage`, `total_cost_usd`, `stop_reason`, `is_error`), `hook_event`, `control_response`, `control_request` (`can_use_tool` — the CLI asking permission for an interactive tool; we `deny` the three gated tools, see above).

**UI event kinds (`ev.kind`):** `text_delta`, `text_end`, `thinking_start/delta/end/redacted`, `tool_use_start/input_delta/use`, `tool_result`, `user_echo` (outer echoes carry `userIndex` — the absolute 0-based ordinal among pure user-prompt jsonl lines; the rewind/fork anchor, immune to ring trimming), `system` (subtypes incl. `init`, `history_replayed`), `hook`, `turn_end`, `assistant_message`, `control_response`, `user_question` (`toolUseId`, `questions[]`), `plan_request` (`toolUseId`, `plan`, `planPath`, optional `autoApproved`), `permission_request` (`toolUseId`, `toolName`, `toolInput`), `permission_resolved` (`toolUseId`, `allow`), `overage_message_queued` (a message queued while the session is overage-paused: `data:{text, attachmentCount, ts, queuedCount}` — rendered as a muted ghost bubble; **no** `user_echo`/turn is sent), `raw`. Each carries `parentToolUseId` (or `null`) — non-null routes into a nested sub-Conversation. New `system` subtype `auto_resume` (`data:{count}`) marks the overage resume firing ("Window reset — delivered N queued"). `auto_resume_skipped` (`data:{reason}`) now fires ONLY for the genuine "session no longer running" case — a missing/past `resetsAt` no longer dead-ends; it schedules a usage recheck instead (see the usage-verified fire in architecture.md).

**Overage queue intercept (`Instance.prompt`).** A **non-internal** `prompt()` is **queued** (`{text, attachments, ts}` onto `_overageQueue`), emits `overage_message_queued`, and returns **before** `user_prompt` — so it neither resumes the session nor cancels the pending resume — when **either**: (a) the session was stopped mid-turn and is armed (`autoStoppedForOverage && autoResumeAt`), **or** (b) the **GLOBAL gate** is active. The gate is a small callback injected per-instance at `create()` (`inst._overageGate() → {active, resetsAt}`); `active` is true iff `_overageActive && _overageResumeMode` (i.e. `stop-resume`, not plain `stop`) **and** `_overageResetsAt` is a valid **FUTURE** epoch-secs instant. That future-resetsAt check is the **safety rail**: a queued send bypasses the manual-resume clear, so a missing/past/NaN reset must mean `active:false` (sends flow normally) or every session would lock out permanently. Path (b) makes it a **global lockout**: idle/existing and brand-new sessions queue too. A queued-only session (no `autoResumeAt` yet) emits an internal **`overage_queued`** (`{resetsAt}`) that the manager (`_armQueuedOnly`) turns into an immediate resume deadline via the controller — it can't wait for a mid-turn→idle transition. The one choke point covers the WS `prompt`, MCP `send_prompt`, and any REST path. `internal:true` prompts (idle-wake stub, conductor steer, and the auto-resume's own send — which first clears the flags via `cancel`) fall through and bypass both intercept paths. On fire, `OverageResumeController.run` delivers all queued messages as **one combined** `prompt({internal:true})` (preamble + numbered list + concatenated attachments); an empty queue is the unchanged single resume. The preamble is softened for a **queued-only** session (`_overageWasStopped === false`) — "Delivering the messages you queued while paused" vs. "…continue where you left off" for one stopped mid-work. `summary().queuedCount` carries the depth; `summary().overageActive` + `overageResetsAt` (read from the gate) let the client paint the paused state before the first message is typed. `_clearOverage` (window reset) clears `_overageResumeMode` and emits a fresh `status` for every instance so paused banners drop everywhere.

**`postWorktreeCreate` result field** — present on `POST /api/instances` responses and MCP `create_worktree` / `spawn_instance({createWorktree:true})` results when a worktree was created. Shape: `{ ran: boolean, exitCode: number|null, durationMs: number, output: string, truncated?: true, timedOut?: true, skipped?: 'disabled', error?: true }`. `ran:false` means no hook script existed (or the kill-switch `ORCH_DISABLE_POST_WORKTREE_HOOK=1` was set — in that case `skipped:'disabled'`). When `ran:true`, `exitCode` is the script's exit code (0 = success; non-zero = failure, but worktree creation still succeeded). `output` is the tail of combined stdout+stderr (≤16 KB; `truncated:true` when the raw output exceeded the cap). `timedOut:true` when the script was killed at the timeout (`ORCH_POST_WORKTREE_TIMEOUT_MS`, default 120000 ms) — `exitCode` is null in that case. The hook is non-fatal: a failure or timeout never rolls back the worktree.

**`<store>/pending-resume.json` entry schema** (one per carried-over session, written by `drainToManifest`, consumed by `restoreFromResumeManifest`): `project`, `sessionId` (the cross-restart identity + `--resume` key), `cwd`, `mode`, `effort`, `thinking`, `model`, `worktreeName`, `temp`, `conducted`, `debug`, `title`, `firstPrompt`, `autoApprovePlan`, `group` (`conductor`/`worker`/`other`), `wasBusy` (re-prompt gate), `workers` (conductor entries only). Pending **overage auto-resume** state — persisted so a resume that was armed in memory survives the restart: `overageResumeAt` (fire deadline, epoch secs = window reset + buffer; `null`/absent ⇒ no pending resume), `overageStopped` (stop-resume intent boolean), `overageWasStopped` (stopped-mid-work vs. queued-only — selects the resume preamble after restore), `overageResetsAt` (window reset, epoch secs), `overageQueue` (messages queued during the wait window — `[{text, attachments, ts}]`, restored onto `_overageQueue` so they still flush with the resume). On boot, an entry with `overageStopped` + finite `overageResumeAt` re-arms the wall-clock sweep after the session is live + idle; a deadline already past fires `AUTO_RESUME_TEXT` on the first sweep tick (and the `RESUME_TEXT` re-prompt is suppressed for it).

## MCP tool protocol

Mounted at `POST /mcp` (Streamable HTTP, JSON-RPC 2.0). Tools are listed in [features.md → MCP interface](features.md#mcp-interface). This section specifies the wire contract — a **clean break** with no backwards-compatibility aliases.

**Input params.** The worker handle is always `sessionId` (the stable session id; the per-process `instanceId` is internal and never accepted — a legacy `{id}` arg hard-fails as unknown). Every worker-addressing tool takes `{sessionId}`: `get_transcript`, `send_prompt`, `wait_for_idle`, `set_mode`, `approve_plan`, `reject_plan`, `answer_question`, `set_auto_approve_plan`, `interrupt_turn`, `kill_instance`, `respawn_instance`, `promote_session`, `sync_worktree`, `merge_worktree` ({sessionId} form; the `{project, worktree}` form is unchanged), `get_recent_messages`, `subscribe_to_idle`, `unsubscribe_from_idle`. `spawn_instance`'s `resume` already takes a sessionId. Worktree is always `worktree` (never `worktreeName`). `spawn_instance` takes `createWorktree:true` (fresh worktree off HEAD) and/or `worktree:"<name>"` (attach existing) — the old `worktree: boolean|string` overload is removed.

**Prefix resolution (MCP boundary).** Before strict-live resolution, the MCP dispatch layer (`src/mcp/server.js`) resolves the incoming `sessionId` arg of any tool that declares one through `InstanceManager.resolveSessionRef` — so a caller may pass an **unambiguous prefix** (e.g. first 8 chars) in place of the full UUID. It is the single, uniform chokepoint (covers `respawn_instance`/`locate_session` too) and is **non-destructive**: an exact full-id match always wins (even if it is also a prefix of another id) and passes through unchanged; a prefix that matches exactly one id is rewritten to that full id; no match is left untouched (so the strict-live paths below still produce `SESSION_NOT_LIVE`/`SESSION_UNKNOWN`/on-disk lookups). A prefix that matches **more than one** id (or a prefix shorter than `SESSION_PREFIX_MIN` = 4 chars that matches any) soft-refuses `{ok:false, code:'SESSION_AMBIGUOUS', sessionId, reason, matches:[<8-char ids>]}` (`isError:false`). The resolution **universe is the distinct sessionIds across all in-memory instances** (live **and** exited in `byId`) — broader than live-only, so a prefix unique among live workers but shared with an exited in-memory session is ambiguous rather than mis-resolved; historical disk-only sessions are out of scope (address them by full id). The canonical stored/persisted sessionId is unchanged (full UUID everywhere): `--session-id`/`--resume`, the `${sessionId}.jsonl` transcript, the subagent dir, and the `?caller=` URL all stay full ids. `resume` (spawn_instance) is **not** prefix-resolved.

**Worker resolution (strict-live, soft-erroring).** `sessionId` (post prefix-resolution) resolves to its single *running* (proc-attached) instance. If the session is known but has no running process → `{ok:false, code:'SESSION_NOT_LIVE', sessionId, reason}` (resume via `spawn_instance({resume:sessionId})`); if no such session anywhere → `{ok:false, code:'SESSION_UNKNOWN', sessionId, reason}`. Both are normal results (`isError:false`), matching the soft-refusal convention. The resolver **never auto-respawns** and never special-cases reads. `respawn_instance` is the exception — it targets a non-live, still-in-`byId` instance by sessionId (and 409s if actually running). `kill_instance` is now live-only (it cannot reap an already-exited non-temp instance by sessionId — accepted under strict-live).

**Emitted handles.** Returns never carry an `instanceId`: `toConductorView()` strips `id` + `callerInstanceId` from every summary (`spawn_instance`, `respawn_instance`, `list_instances`, `wait_for_idle`, `promote_session`), and inline returns emit `sessionId` instead of `id`. `?caller=<sessionId>` on the per-worker MCP URL is the caller's own stable sessionId (baked in at spawn after the sessionId is minted); for `spawn_instance` it is resolved back to the conductor's live `instanceId` and stored as `callerInstanceId` (internal/REST only). The idle-subscription graph (`_idleSubscribers`) is keyed by `sessionId`.

**Idle wake-stub text (`IdleSubscriptionHub.deliver`).** Three variants, injected as an `internal:true` `user_echo` into the caller:
- **Folded** (real `turn_end` delivered to an already-idle caller): `[[cc:wake-callback]]<summary>[[cc:wake-body]]<payload>`. The summary names the worker + says it finished; `<payload>` is `flattenPayload(meta, bodies)` — the compact-JSON metadata block + each raw body block joined by blank lines, i.e. **byte-identical to the content a default `get_recent_messages({sessionId})` call returns** (via the shared `buildRecentMessages` builder in `handlers.js`, so the fold never drifts from the tool). The `[[cc:*]]` markers (defined in `public/wakeCallback.js`, imported by both `src/idleSubscriptions.js` and `public/conversation.js`) survive persist/replay in the CLI jsonl and drive the collapsible UI bubble; they are stripped from display, not from the prompt the model reads.
- **Plain** (`turn_end` but caller was **mid-turn** at delivery, so the send is deferred to when it goes idle): the unchanged `Worker \`<sid>\` finished its turn. Call get_recent_messages(...) to inspect the result.` pointer — decided synchronously at delivery time, so the deferral keeps the plain stub even though the caller is idle by the time it actually fires.
- **Timeout** (watchdog): the unchanged `... did NOT finish — timed out after <ms>ms ...` pointer.
All three keep `internal:true` semantics (bypass the overage queue intercept, never cancel a pending auto-resume).

**Input validation (`validateArgs`).** Beyond type/enum/required, the server enforces the declared `pattern`, `minLength`/`maxLength`, `minimum`/`maximum`, and array `items.type`. **Unknown properties are rejected** (not silently dropped): `unexpected argument '<k>' — not a recognized parameter for <tool>. Allowed: <keys>`. Validation failures return `isError:true` with the message as a single text block. Schemas also declare `default`s mirroring handler defaults (documentation; handlers still apply them).

<a name="mcp-result-format"></a>
**Result format.** `content[0]` is always a **compact** JSON block (no pretty-print). Pure-metadata tools emit just that one block. **Text-payload tools** append separate raw, **un-escaped** text blocks in `content[1..]`:

| Tool | content[0] metadata | content[1..] bodies |
|---|---|---|
| `read_file` | `{path, size, truncated, encoding, lineCount, lineCountExact, startLine?, endLine?}` | one block: utf8 file text, or base64 (binary) |
| `get_worktree_diff` (diff mode) | `{project, worktree, baseRef, head:<sha>, contextLines, offset, truncated, nextOffset, totalLines, totalBytes, hasUncommittedChanges, untracked, includedFiles?, omittedFiles?}` | one block: raw unified diff, always including a `@@@ uncommitted working tree changes (git diff HEAD) @@@`-separated section when uncommitted changes exist (empty string when no diff and no uncommitted changes) |
| `get_worktree_diff` (summary mode) | single JSON block `{project, worktree, baseRef, head:<sha>, summary:true, totals, files[], uncommitted:{totals, files, untracked}}` | — |
| `get_recent_messages` | `{sessionId, messages:[{index, msgId, hasToolUse, textChars, textTruncated, plan?, questions?, blocks?}], source:"ring"\|"disk", omittedToolOnly, retained:{firstSeq,lastSeq,trimmed}, hint?}` | one raw prose block **per message**, block k+1 ↔ messages[k] (empty for plan/question-only) |

**`get_recent_messages` default-call bonding.** `count` defaults to 1 and clamps to `[1,50]`. On the **default** call only (`count` omitted — not `count:1`), when the last message is plain prose and the message immediately before it carries `plan`/`questions`, that predecessor is bonded in too (2 messages returned instead of 1) — a split turn's plan/question and its trailing prose are surfaced together without the caller needing `count:2`. Bonding never walks back more than one message, and a message that already carries its own `plan`/`questions` is always returned alone. Any explicit `count` is literal (`slice(-count)`), bonding never applies.

**Annotations.** `tools/list` entries carry `annotations`: `readOnlyHint` (all `list_*`, `read_file`, `project_status`, `get_*`, `locate_session`), `destructiveHint` (`kill_instance`, `delete_worktree`, `merge_worktree`), `idempotentHint` (`set_mode`, `set_auto_approve_plan`, `set_project_workspace`, `unsubscribe_from_idle`, `create_workspace`, `delete_workspace`, `rename_workspace`, `kill_instance`).

**`ok` / refusals.** `ok` is reserved for **soft refusals** only. Acknowledgement tools (`send_prompt`, `kill_instance`, `subscribe_to_idle`/`unsubscribe_from_idle`, `approve_plan`/`reject_plan`, `answer_question` success, `set_auto_approve_plan`, the workspace tools, `set_project_workspace`, `delete_worktree` success) return bare data with **no `ok`**. Expected business refusals return `{ok:false, reason, code}` and are **not thrown**, with stable codes: `WORKTREE_ATTACHED`, `WORKTREE_DIRTY` (`delete_worktree`); `SESSION_NOT_LIVE` / `SESSION_UNKNOWN` (`sync_worktree` and every worker-addressing tool, from strict-live resolution); `SESSION_AMBIGUOUS` (any worker-addressing tool, when a sessionId **prefix** matches >1 in-memory session or is shorter than `SESSION_PREFIX_MIN`; carries `matches:[<8-char ids>]`); `NO_PENDING_QUESTION`, `ANSWER_COUNT_MISMATCH`, `INVALID_OPTION`, `NOT_MULTISELECT`, `EMPTY_ANSWER` (`answer_question`); `WORKTREE_BEHIND`, `BASE_BRANCH_MISMATCH`, `PARENT_DIRTY`, `MERGE_FAILED` (`merge_worktree`). The only constant `ok:true` is `merge_worktree`/`sync_worktree` success (shape shared with the REST land endpoints).

**Errors.** True faults are caught and returned `isError:true` as a prose block — `<message> (HTTP <statusCode>)` when the handler set a `statusCode` — followed by a structured `{error, code, statusCode}` block. `code` derives from `statusCode` (`400→BAD_REQUEST`, `404→NOT_FOUND`, `409→CONFLICT`, `500→INTERNAL`). Example sources: `locate_session`/`read_file` 404, invalid `baseRef`/`promote_session` non-temp 400.

**Disk-backed reads (ring-first, fallback-on-demand).** The in-memory event ring is a fixed-cap (`DEFAULT_RING_CAP=2000`, drop-oldest) cache, but ring eviction is **invisible** to these two tools:
- `get_transcript({sessionId, sinceSeq, limit})` pages via `pageInstanceEvents` (`src/eventArchive.js`). Served from the ring; when `sinceSeq < trimmedBefore` (an evicted range) the on-disk session transcript is loaded and reconciled with the ring by `_seq` at a turn-aligned seam (no gap/dup). `sinceSeq` is forward paging (events with `_seq > sinceSeq`, oldest-first); the response adds `{hasMore, nextAfter}` — poll again with `sinceSeq = nextAfter`. `limit` clamped `[1,500]` (default 200). Replaces the old `truncated` field. Caveat: a single turn larger than the ring cap leaves a mid-turn `_seq` gap (covered by `get_recent_messages`).
- `get_recent_messages` is ring-first; only when the retained tail can't satisfy the requested recent **text** messages (and the ring was trimmed) does it read the on-disk transcript and merge by `msgId` (ring entry wins; bounded to the last `DISK_REPLAY_TAIL_CAP=5000` disk events). `source` reports `"ring"` vs `"disk"`. A missing transcript (exited temp session) degrades to ring-only + a `hint`; never crashes. On a **defaulted** call (no `count` passed) the ring must satisfy `n+1` text messages, not `n`, since the default path may bond in one preceding plan/question message (see below) — this raises the bar for staying ring-only before falling back to disk.

Note: the REST endpoints above keep their own `worktreeName` field — they are a separate contract from the MCP tools and were not renamed.
