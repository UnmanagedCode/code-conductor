## Subprocess protocol
```bash
claude -p \
  --input-format=stream-json --output-format=stream-json \
  --verbose --include-partial-messages --include-hook-events \
  --allow-dangerously-skip-permissions \
  --permission-prompt-tool stdio \
  --permission-mode <plan|bypassPermissions> --effort <effort> --thinking <thinking> \
  --settings '{"hooks":{"PreToolUse":[…]}}' \
  --mcp-config '{"mcpServers":{"code-conductor":{"type":"http","url":"http://127.0.0.1:<port>/mcp"}}}' \
  [--model <name>] \
  --session-id <fresh-uuid> | --resume <existing-uuid>
```

The `--mcp-config` line is omitted when `ORCH_DISABLE_MCP_AUTOREGISTER=1` (no MCP toolbelt auto-registered into the spawned process).

**Interactive tools under CLI ≥ 2.1.x — `--permission-prompt-tool stdio`.** Headless stream-json (`-p`) sessions are non-interactive (`isNonInteractiveSession=true`); the CLI serves a coordinator/agent tool profile and **strips ExitPlanMode / EnterPlanMode / AskUserQuestion** unless the client presents as a permission consumer. Passing **`--permission-prompt-tool stdio`** does exactly that: the CLI routes those tools' permission prompts to us as `can_use_tool` control_requests over the stream-json channel (verified: it re-adds the three tools to `system/init` `tools[]` and they become callable). It is **additive** to `--allow-dangerously-skip-permissions` (normal tools stay auto-allowed and never reach `can_use_tool`; only the interactive tools do). Kill-switch: `ORCH_DISABLE_STDIO_PERMISSIONS=1` reverts to the fail-closed behavior (tools stripped).

The orchestrator answers `can_use_tool` in `Instance._handleStdoutLine`: for `ExitPlanMode`/`EnterPlanMode`/`AskUserQuestion` it replies `control_response {behavior:"deny", message:<AWAITING_INPUT_MESSAGE>}`; any other tool (rare) is allowed. The **deny ends the turn** (`stop_reason: end_turn`, verified) — this is deliberate and load-bearing: the `plan_request`/`user_question` card is emitted from the tool-use, and ending the turn is what lets a conductor's `subscribe_to_idle` wake on `turn_end` and then call `approve_plan`/`reject_plan`/`answer_question`. Holding the request open for an *in-turn* answer would suppress `turn_end` and deadlock that loop, so we do not. `approve_plan`/`reject_plan` are unchanged (setMode + approval prompt); `answer_question` delivers the worker a structured AskUserQuestion answer that is **byte-identical** to a UI question-card submit — both call the one DOM-free formatter in `public/userQuestionAnswers.js` (re-exported by `public/blocks.js`, imported server-side by `src/mcp/handlers.js`). The flag alone exposes and makes the tools callable — **no `initialize`/`supportedDialogKinds` handshake is sent** (probe-verified unnecessary), and the CLI never issues `request_user_dialog` in this configuration (permission flows via `can_use_tool`).

`--allow-dangerously-skip-permissions` is **always** passed at spawn — even for `plan`-mode instances — because without it the CLI rejects any later runtime `set_permission_mode bypassPermissions` (i.e. the plan-approve flow). The flag only *permits* the switch; it doesn't activate bypass on its own.

One `PreToolUse` hook via inline `--settings` JSON (the old static `command` deny on **AskUserQuestion|ExitPlanMode** is **removed** — those tools are now gated at the `can_use_tool` layer, see above):
1. **Edit|Write|NotebookEdit|Bash** — `http` hook → `POST /api/instances/<id>/hook-callback`, **660 s** CLI-side timeout. Orchestrator auto-allows when the mode isn't `ask`; in `ask` mode holds the response open up to **540 s** (deliberately under 660 s — an HTTP timeout would make the CLI treat the hook as a non-blocking error and proceed, the opposite of intent). `ask` is orchestrator-tracked only and maps to `bypassPermissions` at the CLI level; the hook callback inspects orchestrator-side mode to gate.

Inbound: `user` (text or `[{type:"text", text:"..."}, …]` blocks; attachments use `` Attached file: `<rel-path>` ``), `control_request` (`set_permission_mode` / `set_model` / `interrupt`), `control_response` (our reply to the CLI's `can_use_tool`), `keep_alive`. A `user` message written while a turn is running is **delivered mid-turn and interleaved into the live turn** (steering) — not queued until turn_end; the CLI persists such a mid-turn prompt as a `type:"attachment"` `queued_command` line (array `prompt`) rather than a `type:"user"` line.

**Two-tier interrupt.** SOFT (default) injects a hidden steering `user` message suffixed with `[[cc:soft-interrupt]]` (`SOFT_INTERRUPT_MARKER`) telling the model to stop all work and end its turn after one brief acknowledgement line — graceful wind-down, partial work preserved (a fully silent reply would make the CLI inject a "no visible output" follow-up that re-engages the model). At send time, `Instance.interrupt()` also emits a live `system`/`soft_interrupted` event with `data.text` set to the steer text — not a `user_echo`, so it never shifts `_userEchoCount`/the rewind-fork index. The marker itself is filtered everywhere it could surface (`parser._handleUser`, `transcript.replayPersistedLine`, `transcript.isPureUserPromptLine`) so on JSONL replay it never renders a user bubble, produces the same bare `soft_interrupted` annotation (no `data.text`) instead, and never shifts the rewind/fork index. FORCED (`force:true`) is the unchanged hard `control_request` `subtype:interrupt` abort (discards partial work). Both gated by `if (status !== 'turn') return`.

**Managed `/clear` (self-compaction).** `Instance.clearContext()` writes a `type:'user'` message whose sole content is the text `/clear` on the same stdin path a normal prompt uses (via `_sendRaw`, but bypassing `prompt()`'s user_echo + overage-queue intercept — it is a server-internal control send). The CLI rotates its sessionId to a fresh UUID **in place** — same OS process/pid, old `<sid>.jsonl` preserved, a new `<newId>.jsonl` created — and emits a `system/init` carrying the new `session_id` (which `Instance`'s system/init handler folds into `this.sessionId`) plus a `result` (turn_end). Driven by `SessionCompactController` off the `compact_session` MCP tool — see [MCP tool protocol](#mcp-tool-protocol) and [architecture.md](architecture.md).

**Post-hard-abort drain window.** The CLI's stdin queue is not cleared by a hard abort. Any `user` message written before the abort (the soft-interrupt steer, or prompts sent mid-turn) remains queued; after the abort the CLI dequeues them and starts a spurious new turn for each. After every hard abort the orchestrator opens a drain window (`POST_ABORT_DRAIN_WINDOW_MS`, default 3 s) and watches for `system/init` events on the event channel (`Instance._openDrainWindow`). The `system/init` event arrives ~39 ms after the abort and ~3.7 s before any API round-trip, so the drain fires an immediate `control_request subtype:interrupt` to sever the spurious turn at zero token cost. Each drain resets the window deadline (sliding window). Calling `prompt()` closes the window so intentional follow-up prompts proceed normally. The drain emits a `system/drain_abort` event (kind `'system'`, subtype `'drain_abort'`, data `{count}`) visible in the conversation as "⏹ Drained queued turn after interrupt".

Outbound: `system` + `subtype:"init"` (bundled with first turn's response, not at startup; carries `session_id`, `model`, `tools`, `permissionMode`), `stream_event` (live SSE deltas — primary feed), `assistant` (reconciled message content — one multi-block envelope per message on older CLIs; the async-worker CLI instead finalizes **each content block as its own single-block envelope**, all sharing `message.id`, and persists the same one-block-per-line shape to the session jsonl; `buildMessageFromRing` merges all envelopes of a `msgId`, the UI ignores outer-turn envelopes either way, and disk replay threads a per-file `blockCursor` so a fragmented message's blocks get distinct `blockIdx` values matching the live stream indices. Tools dispatch per-block, so a `tool_result` `user` line can land *between* two envelopes of the same `msgId`. Sub-agent (Task/Agent) turns arrive as finals-only `assistant`/`user` envelopes tagged `parent_tool_use_id` — their partial `stream_event` frames are never emitted (every CLI `stream_event` hardcodes `parent_tool_use_id:null`; the forwarding knob `forwardSubagentText` is SDK-only, no CLI flag)), `user` (`tool_result` blocks), `result` (turn-end with `duration_ms`, `usage`, `total_cost_usd`, `stop_reason`, `is_error`), `hook_event`, `control_response`, `control_request` (`can_use_tool` — the CLI asking permission for an interactive tool; we `deny` the three gated tools, see above).

**UI event kinds (`ev.kind`):** `text_delta`, `text_end`, `thinking_start/delta/end/redacted`, `tool_use_start/input_delta/use`, `tool_result`, `user_echo` (outer echoes carry `userIndex` — the absolute 0-based ordinal among pure user-prompt jsonl lines; the rewind/fork anchor, immune to ring trimming; optionally carries `skillLoad: {skill}` — see below), `system` (subtypes incl. `init`, `history_replayed`), `hook`, `turn_end`, `assistant_message`, `control_response`, `user_question` (`toolUseId`, `questions[]`), `plan_request` (`toolUseId`, `plan`, `planPath`, optional `autoApproved`), `permission_request` (`toolUseId`, `toolName`, `toolInput`), `permission_resolved` (`toolUseId`, `allow`), `overage_message_queued` (a message queued while the session is overage-paused: `data:{text, attachmentCount, ts, queuedCount}` — rendered as a muted ghost bubble; **no** `user_echo`/turn is sent), `raw`. Each carries `parentToolUseId` (or `null`) — non-null routes into a nested sub-Conversation. New `system` subtype `auto_resume` (`data:{count}`) marks the overage resume firing ("Window reset — delivered N queued"). `auto_resume_skipped` (`data:{reason}`) now fires ONLY for the genuine "session no longer running" case — a missing/past `resetsAt` no longer dead-ends; it schedules a usage recheck instead (see the usage-verified fire in architecture.md). `system` subtype `cache_miss` (`data:{cacheRead, cacheCreation}`, plus `prevPrefix` + `evicted` on the cross-turn path) is an informational once-per-turn notice fired on a detected cache miss — rendered as a `SystemBlock`, no action. Detection is **cross-turn**: a miss when this turn's first-request `cache_read` (`read_N`) is below the previous turn's last-request prefix `P` (`cache_read + cache_creation`), i.e. `read_N < P − max(1024, 1% of P)` — catches full AND partial evictions. Turn 1 (no prior `P`) and the turn after a compaction (`system`/`compacting`), model switch (`_trackModel`), or rewind/respawn (`_wipeForResume`) — each sets `_prefixBaselineInvalid`, consumed next turn — fall back to the stateless `creation > read` rule (still flags cold start / expiry / resume / cold rewind). Not flagged: warm continuations (read == `P`) or content-addressed hits. A mid-turn eviction (past the first request) is not caught (first-request-only). `turn_end` additionally carries `cacheMiss` (bool) + `firstReqCacheRead`/`firstReqCacheCreation` (the deciding first-request usage) + `firstReqEvicted` (`P − read_N` on the cross-turn path, else 0), persisted to `costs.jsonl` as `cache_miss`/`first_req_cache_read`/`first_req_cache_creation`/`first_req_evicted`.

**Skill-load correlation (`attachSkillLoad`, `src/parser.js`).** A `Skill` tool call's SKILL.md content is injected back by the CLI as a separate `user` envelope with a top-level `isSynthetic:true` flag and no `tool_use_id` link to the tool call. That flag alone isn't a reliable "this is skill content" signal — the CLI reuses it for unrelated synthetic text (compaction-continuation summaries, Stop-hook feedback) — so the parser only tags the resulting `user_echo` with `skillLoad: {skill}` when it correlates the message with a FIFO queue of pending `{toolUseId, skill}` entries pushed when a `Skill` tool_use is seen; an `isSynthetic` message with an empty queue renders as an ordinary `user_echo`. `src/transcript.js`'s replay path threads the same queue (`pendingSkillLoads`, alongside the existing `blockCursor`) through `replayPersistedLine` so live and replayed sessions tag identically — replay always parses a session's jsonl in one complete pass from line 0 (never a partial/windowed range), and the queue is a fresh local array per call, so a `Skill` tool_use and its content injection are always correlated within the same pass regardless of how the resulting events are later paged. The FIFO has no identity link to the injection message, so a pending entry is bounded (`expireSkillLoads`) rather than left to live indefinitely: an erroring `tool_result` for its `toolUseId` drops it immediately, and any genuine (non-synthetic) `user_echo` — a real prompt, including the `queued_command` replay branch and `Instance.prompt()`/`windDown()`'s direct live emissions — clears the whole queue, since the synchronous ordering guarantee is broken once a new real turn begins. This prevents an orphaned entry (skill errored, or its turn was interrupted before the injection arrived) from surviving to mislabel a later, unrelated `isSynthetic` message. The client (`public/conversation.js` `_renderUserEcho`) renders a `skillLoad`-tagged echo as a collapsed `<details class="block skill">` — summary names the invoked skill, body holds the raw SKILL.md text — instead of the plain-text bubble a normal echo gets.

**Agent-tool (subagent) task events.** The CLI emits `system` events with `subtype` `task_started` (`{task_id, tool_use_id, description, subagent_type, task_type, prompt}`, fired the instant an `Agent` tool_use dispatches — foreground or backgrounded), `task_progress` (periodic `{task_id, tool_use_id, usage:{total_tokens,tool_uses,duration_ms}, last_tool_name}` ticks while it runs), `task_updated` (`{task_id, patch:{status, end_time}}` on a state transition — `status` one of `completed|failed|cancelled|error` for a terminal one), and `task_notification` (`{task_id, tool_use_id, status, output_file, summary, usage}`, the human/model-facing "it's done" ping — the same event a `<task-notification>` block wraps when injected back into a worker's own conversation). None carry `parent_tool_use_id`, so they land as top-level `system` events like `turn_end`. `shouldRenderSystem` (`public/blocks.js`) hides `task_progress`/`task_started`/`task_updated` from the chat feed (`task_notification` falls through the same allow-list to hidden-by-default). That re-injected `<task-notification>` block is itself persisted as a `type:"user"` jsonl line — `isTaskNotificationContent` (`src/parser.js`) recognizes it by tag shape and it's filtered at the same three sites as `SOFT_INTERRUPT_MARKER` (`parser._handleUser`, `transcript.isPureUserPromptLine`, `transcript.replayPersistedLine`), so it never becomes a `user_echo` or shifts a rewind/fork index. Server-side, `Instance` tracks in-flight ones in `_activeAgentTasks` (see `displayStatus` below). Task events fire for more than explicit `run_in_background` calls: background is the Agent tool's default, and long-running foreground Bash gets promoted to a task (every promoted Bash observed ran ≥3.5 s — a full test run always does). Delivery has two shapes: a **slow** task's tool_result resolves immediately with an "Async agent launched" ack (`isAsync:true`), so `turn_end` can fire while `task_updated` is still pending; a **fast/blocking** task holds the tool_result open and delivers the full output right after its mid-turn `task_notification`, always before `turn_end`.

**Overage queue intercept (`Instance.prompt`).** A **non-internal** `prompt()` is **queued** (`{text, attachments, ts}` onto `_overageQueue`), emits `overage_message_queued`, and returns **before** `user_prompt` — so it neither resumes the session nor cancels the pending resume — when **either**: (a) the session was stopped mid-turn and is armed (`autoStoppedForOverage && autoResumeAt`), **or** (b) the **GLOBAL gate** is active. The gate is a small callback injected per-instance at `create()` (`inst._overageGate() → {active, resetsAt}`); `active` is true iff `_overageActive && _overageResumeMode` (i.e. `stop-resume`, not plain `stop`) **and** `_overageResetsAt` is a valid **FUTURE** epoch-secs instant. That future-resetsAt check is the **safety rail**: a queued send bypasses the manual-resume clear, so a missing/past/NaN reset must mean `active:false` (sends flow normally) or every session would lock out permanently. Path (b) makes it a **global lockout**: idle/existing and brand-new sessions queue too. A queued-only session (no `autoResumeAt` yet) emits an internal **`overage_queued`** (`{resetsAt}`) that the manager (`_armQueuedOnly`) turns into an immediate resume deadline via the controller — it can't wait for a mid-turn→idle transition. The one choke point covers the WS `prompt`, MCP `send_prompt`, and any REST path. `internal:true` prompts (idle-wake stub, conductor steer, and the auto-resume's own send — which first clears the flags via `cancel`) fall through and bypass both intercept paths. On fire, `OverageResumeController.run` delivers all queued messages as **one combined** `prompt({internal:true})` (preamble + numbered list + concatenated attachments); an empty queue is the unchanged single resume. The preamble is softened for a **queued-only** session (`_overageWasStopped === false`) — "Delivering the messages you queued while paused" vs. "…continue where you left off" for one stopped mid-work. `summary().queuedCount` carries the depth; `summary().overageActive` + `overageResetsAt` (read from the gate) let the client paint the paused state before the first message is typed. `_clearOverage` (window reset) clears `_overageResumeMode` and emits a fresh `status` for every instance so paused banners drop everywhere.

**`summary().displayStatus` / `activeAgentTasks`.** Additive, display-only fields alongside `status` — `activeAgentTasks` is the count of in-flight Agent-tool tasks (`_activeAgentTasks`, keyed by `task_id`); `displayStatus` is `'running'` when `status === 'idle'` and `activeAgentTasks > 0`, else identical to `status`. Never repurposes the `status` enum itself, so nothing that gates on the literal 5 `status` values (composer, kill/resume buttons) is affected. `IdleSubscriptionHub.onTurnEnd` **does** read the count (via `Instance.activeAgentTaskCount`): a `turn_end` while `activeAgentTasks > 0` **defers** the idle wake until the follow-up `turn_end` after the last subagent finishes. It **also** defers on `Instance.taskNotificationPending` — set when a `task_notification` fires *mid-turn*, cleared by any top-level `tool_result` that follows it within the turn (the task's own sync-delivered result, or the outer tool round-trip the CLI attaches queued notifications to). Still set at `turn_end` therefore means the notification is genuinely queued and an unprompted **re-invocation turn** will deliver it; the wake waits for that turn's `turn_end`. A notification consumed in-turn wakes at the same `turn_end` — parked (AskUserQuestion/ExitPlanMode) or not. So the dispatch-and-wake callback fires only when the worker AND its subagents are done and no re-invocation turn is owed (a never-finishing subagent — or a CLI that never flushes its queue — falls back to the subscription watchdog). Every summary-based surface carries the fields: `GET /api/instances`, `list_instances`, `wait_for_idle`, `respawn_instance`, `promote_session`. `displayStatus:'running'` (not just raw `idle`) means "still busy"; `status:'idle'` alone doesn't prove a backgrounded subagent has finished.

Backward pages are **quiescent-aligned**: `limit` is a target, and the page start snaps to the nearest quiescent point (no open block, every `tool_use` resolved; outer `user_echo`/`turn_end` reset the scan, so an interrupted turn's dangling tool never poisons later boundaries), so each page holds only whole blocks and complete tool round-trips — the client's isolated per-page renderer never sees a half block, and its bubble merge reassembles a turn split across pages. Quiescent points are dense, so pages stay ~`limit`-sized (no whole-turn extension, no backstop); via the cursor, page ends align for free; `after=<seq>` pages forward (`sinceSeq` semantics, exact `limit`); neither ⇒ trailing page. `limit` clamped to `[1,500]`, default 200. Synthetic `{kind:'task_completion', tasks}` events (no `_seq`) are spliced in after the completing `TaskUpdate` of any task batch below the tail, so lazily-loaded history shows the finished-task bubble (reconstructed via `src/taskReconstruct.js`; batches completing **in** the tail are still synthesized client-side, partitioned by `tailStartSeq` so there is no overlap). 400 non-integer params, 404 unknown id. |

## MCP tool protocol

Mounted at `POST /mcp` (Streamable HTTP, JSON-RPC 2.0). Tools are listed in [features.md → MCP interface](features.md#mcp-interface). This section specifies the wire contract — a **clean break** with no backwards-compatibility aliases.

**Input params.** The worker handle is always `sessionId` (the stable session id; the per-process `instanceId` is internal and never accepted — a legacy `{id}` arg hard-fails as unknown). Every worker-addressing tool takes `{sessionId}`: `get_transcript`, `send_prompt`, `wait_for_idle`, `set_mode`, `approve_plan`, `reject_plan`, `answer_question`, `set_auto_approve_plan`, `interrupt_turn`, `kill_instance`, `respawn_instance`, `promote_session`, `sync_worktree`, `merge_worktree` ({sessionId} form; the `{project, worktree}` form is unchanged), `get_recent_messages`, `subscribe_to_idle`, `unsubscribe_from_idle`. `spawn_instance`'s `resume` already takes a sessionId. Worktree is always `worktree` (never `worktreeName`). `spawn_instance` takes `createWorktree:true` (fresh worktree off HEAD) and/or `worktree:"<name>"` (attach existing) — the old `worktree: boolean|string` overload is removed.

**Prefix resolution (MCP boundary).** Before strict-live resolution, the MCP dispatch layer (`src/mcp/server.js`) resolves the incoming `sessionId` arg of any tool that declares one through `InstanceManager.resolveSessionRef` — so a caller may pass an **unambiguous prefix** (e.g. first 8 chars) in place of the full UUID. It is the single, uniform chokepoint (covers `respawn_instance`/`locate_session` too) and is **non-destructive**: an exact full-id match always wins (even if it is also a prefix of another id) and passes through unchanged; a prefix that matches exactly one id is rewritten to that full id; no match is left untouched (so the strict-live paths below still produce `SESSION_NOT_LIVE`/`SESSION_UNKNOWN`/on-disk lookups). A prefix that matches **more than one** id (or a prefix shorter than `SESSION_PREFIX_MIN` = 4 chars that matches any) soft-refuses `{ok:false, code:'SESSION_AMBIGUOUS', sessionId, reason, matches:[<8-char ids>]}` (`isError:false`). The resolution **universe is the distinct sessionIds across all in-memory instances** (live **and** exited in `byId`) — broader than live-only, so a prefix unique among live workers but shared with an exited in-memory session is ambiguous rather than mis-resolved; historical disk-only sessions are out of scope (address them by full id). The canonical stored/persisted sessionId is unchanged (full UUID everywhere): `--session-id`/`--resume`, the `${sessionId}.jsonl` transcript, the subagent dir, and the `?caller=` URL all stay full ids. `resume` (spawn_instance) is **not** prefix-resolved.

**Worker resolution (strict-live, soft-erroring).** `sessionId` (post prefix-resolution) resolves to its single *running* (proc-attached) instance. If the session is known but has no running process → `{ok:false, code:'SESSION_NOT_LIVE', sessionId, reason}` (resume via `spawn_instance({resume:sessionId})`); if no such session anywhere → `{ok:false, code:'SESSION_UNKNOWN', sessionId, reason}`. Both are normal results (`isError:false`), matching the soft-refusal convention. The resolver **never auto-respawns** and never special-cases reads. `respawn_instance` is the exception — it targets a non-live, still-in-`byId` instance by sessionId (and 409s if actually running). `kill_instance` is now live-only (it cannot reap an already-exited non-temp instance by sessionId — accepted under strict-live).

**Resume pre-flight (`spawn_instance({resume})`).** Before constructing an Instance or spawning `claude --resume`, `InstanceManager._doCreate` verifies the resume id has a **resumable conversation** at the resolved cwd (`hasResumableConversation` in `src/transcript.js`: the `${sessionId}.jsonl` exists **and** holds ≥1 `user`/`assistant` record — mirroring the CLI's own "No conversation found" criterion, so a marker-only crash stub of `last-prompt`/`permission-mode` lines does **not** qualify). If it doesn't, `spawn_instance` soft-refuses `{ok:false, code:'SESSION_UNKNOWN', sessionId, reason}` (`isError:false`) instead of spawning a subprocess that would exit 1 and crash-loop. This guards the case a caller pins `project`+`worktree` (which bypasses the `findSessionLocation` "project required" net) with a mistyped/bogus resume id. Because the refusal fires before any Instance is registered in `byId`, a follow-up `respawn_instance` on the same bogus id then returns `SESSION_NOT_LIVE` (no phantom crashed instance to respawn).

**Emitted handles.** Returns never carry an `instanceId`: `toConductorView()` strips `id` + `callerInstanceId` from every summary (`spawn_instance`, `respawn_instance`, `list_instances`, `wait_for_idle`, `promote_session`), and inline returns emit `sessionId` instead of `id`. `?caller=<instanceId>` on the per-worker MCP URL is the caller's own **stable instanceId** (baked into `--mcp-config` at spawn). It is the instanceId — NOT the sessionId — deliberately: the URL is frozen in the subprocess config for the process lifetime, but a `/clear` rotates the sessionId in place, so a baked sessionId would go stale after the first self-compaction. The `/mcp` boundary (`src/mcp/server.js`) resolves the handle to the caller's **current** sessionId per request via `InstanceManager.callerSessionId(handle)` (`byId.get(handle)?.sessionId`), so every downstream handler still receives a live sessionId as `callerId` — valid across any number of rotations, with no event or re-registration. A handle naming no live instance yields `callerId=null` (the same "no caller" path as an absent param). The instanceId never leaves the server as an identity: the two sites that surface the caller (the `compact_session` return, the plugin `caller.sessionId` envelope) emit the resolved sessionId. For `spawn_instance` the resolved caller instance is stored as `callerInstanceId` (internal/REST only). Internally the idle-subscription graph (`_idleSubscribers`) and overage timers (`_autoResumeTimers`) are keyed by the **stable instanceId**; sessionId is translated to instanceId at the register boundary (`subscribe`/`unsubscribe`) and back for delivery — so an in-place `/clear` rotation needs no re-keying.

**Caller-addressed tools.** `compact_session` takes **no** target sessionId — it acts solely on the caller, resolved from `?caller=<instanceId>` (a missing/unresolvable caller throws). Because the handle is the stable instanceId (resolved to the current sessionId at the boundary, above), a session can call `compact_session` **repeatedly** — the caller stays resolvable across each `/clear` rotation. It arms a managed `/clear` on the caller (see [Subprocess protocol → Managed `/clear`](#subprocess-protocol)); because the `/clear` rotates the caller's sessionId, `SessionCompactController` is keyed by the caller's stable `instanceId`. The other internal side structures (idle-subscription graph, overage timers) are likewise instanceId-keyed, so the rotation strands nothing and needs no migration.

**Idle wake-stub text (`IdleSubscriptionHub.deliver`).** Three variants, injected as an `internal:true` `user_echo` into the caller. **All three carry the `[[cc:wake-callback]]` marker** (defined in `public/wakeCallback.js`, imported by both `src/idleSubscriptions.js` and `public/conversation.js`), so all three render as the wake-callback UI bubble with an orchestrator badge (🔔); only **Folded** additionally carries the `[[cc:wake-body]]` separator + payload. The markers survive persist/replay in the CLI jsonl and are stripped from display, not from the prompt the model reads. The presence of `[[cc:wake-body]]` is the "is it folded" signal — the client renders a collapsible `<details>` when a body is present, else a body-less summary line (no caret).
- **Folded** (real `turn_end` delivered to an already-idle caller): `[[cc:wake-callback]]<summary>[[cc:wake-body]]<payload>` (via `buildWakeStub`). The summary names the worker + says it finished; `<payload>` is `flattenPayload(meta, bodies)` — the compact-JSON metadata block + each raw body block joined by blank lines, i.e. **byte-identical to the content a default `get_recent_messages({sessionId})` call returns** (via the shared `buildRecentMessages` builder in `handlers.js`, so the fold never drifts from the tool).
- **Plain** (`turn_end` but caller was **mid-turn** at delivery, so the send is deferred to when it goes idle): `[[cc:wake-callback]]` + the unchanged `Worker \`<sid>\` finished its turn. Call get_recent_messages(...) to inspect the result.` pointer (via `markPlainStub`, body-less) — decided synchronously at delivery time, so the deferral keeps the plain stub even though the caller is idle by the time it actually fires.
- **Timeout** (watchdog): `[[cc:wake-callback]]` + the unchanged `... did NOT finish — timed out after <ms>ms ...` pointer (via `markPlainStub`, body-less).
All three keep `internal:true` semantics (bypass the overage queue intercept, never cancel a pending auto-resume).

**Input validation (`validateArgs`).** Beyond type/enum/required, the server enforces the declared `pattern`, `minLength`/`maxLength`, `minimum`/`maximum`, and array `items.type`. **Unknown properties are rejected** (not silently dropped): `unexpected argument '<k>' — not a recognized parameter for <tool>. Allowed: <keys>`. Validation failures return `isError:true` with the message as a single text block. Schemas also declare `default`s mirroring handler defaults (documentation; handlers still apply them).

<a name="mcp-result-format"></a>
**Result format.** `content[0]` is always a **compact** JSON block (no pretty-print). Pure-metadata tools emit just that one block. **Text-payload tools** append separate raw, **un-escaped** text blocks in `content[1..]`:

| Tool | content[0] metadata | content[1..] bodies |
|---|---|---|
| `project_read` | `{path, size, truncated, encoding, lineCount, lineCountExact, startLine?, endLine?}` | one block: utf8 file text, or base64 (binary) |
| `project_diff` (diff mode) | `{project, worktree, baseRef, head:<sha>, contextLines, offset, truncated, nextOffset, totalLines, totalBytes, hasUncommittedChanges, untracked, ahead, includedFiles?, omittedFiles?}` | one block: raw unified diff, always including a `@@@ uncommitted working tree changes (git diff HEAD) @@@`-separated section when uncommitted changes exist (empty string when no diff and no uncommitted changes) |
| `project_diff` (summary mode) | single JSON block `{project, worktree, baseRef, head:<sha>, summary:true, ahead, totals, files[], uncommitted:{totals, files, untracked}}` | — |
| `get_recent_messages` | `{sessionId, messages:[{index, msgId, hasToolUse, textChars, textTruncated, plan?, questions?, blocks?}], source:"ring"\|"disk", omittedToolOnly, retained:{firstSeq,lastSeq,trimmed}, hint?}` | one raw prose block **per message**, block k+1 ↔ messages[k] (empty for plan/question-only) |

**`get_recent_messages` default-call bonding.** `count` defaults to 1 and clamps to `[1,50]`. On the **default** call only (`count` omitted — not `count:1`), when the last message is plain prose the selection is bonded back to the turn's `plan`/`questions` message and spans from it through the end of that turn — so a split turn whose trailing prose spans two or more messages surfaces the plan/question together with all of it, without the caller needing an explicit `count`. The walk-back is **scoped to the current turn** (via the ring's `turn_end` seqs — a plan from an earlier turn is never pulled in), and a message that already carries its own `plan`/`questions` is returned alone. Any explicit `count` is literal (`slice(-count)`), bonding never applies.

**Annotations.** `tools/list` entries carry `annotations`: `readOnlyHint` (all `list_*`, `project_read`, `project_status`, `get_*`, `locate_session`), `destructiveHint` (`kill_instance`, `delete_worktree`, `merge_worktree`), `idempotentHint` (`set_mode`, `set_auto_approve_plan`, `set_project_workspace`, `unsubscribe_from_idle`, `create_workspace`, `delete_workspace`, `rename_workspace`, `kill_instance`).

**`ok` / refusals.** `ok` is reserved for **soft refusals** only. Acknowledgement tools (`send_prompt`, `kill_instance`, `subscribe_to_idle`/`unsubscribe_from_idle`, `approve_plan`/`reject_plan`, `answer_question` success, `set_auto_approve_plan`, the workspace tools, `set_project_workspace`, `delete_worktree` success) return bare data with **no `ok`**. Expected business refusals return `{ok:false, reason, code}` and are **not thrown**, with stable codes: `WORKTREE_ATTACHED`, `WORKTREE_DIRTY` (`delete_worktree`); `SESSION_NOT_LIVE` / `SESSION_UNKNOWN` (`sync_worktree` and every worker-addressing tool, from strict-live resolution); `SESSION_UNKNOWN` also from **`spawn_instance({resume})`** when the resume id has no resumable conversation on disk (resume pre-flight above); `SESSION_AMBIGUOUS` (any worker-addressing tool, when a sessionId **prefix** matches >1 in-memory session or is shorter than `SESSION_PREFIX_MIN`; carries `matches:[<8-char ids>]`); `NO_PENDING_QUESTION`, `ANSWER_COUNT_MISMATCH`, `INVALID_OPTION`, `NOT_MULTISELECT`, `EMPTY_ANSWER` (`answer_question`); `WORKTREE_BEHIND`, `BASE_BRANCH_MISMATCH`, `PARENT_DIRTY`, `WORKTREE_DIRTY`, `NOTHING_TO_MERGE`, `MERGE_FAILED` (`merge_worktree` — same `WORKTREE_DIRTY` meaning as `delete_worktree`'s (the worktree's own tree, not `PARENT_DIRTY`'s parent repo), overridable here via `allowDirty:true`); `INVALID_SUMMARY` (`compact_session`, empty summary). The only constant `ok:true` is `merge_worktree`/`sync_worktree` success (shape shared with the REST land endpoints).

**Errors.** True faults are caught and returned `isError:true` as a prose block — `<message> (HTTP <statusCode>)` when the handler set a `statusCode` — followed by a structured `{error, code, statusCode}` block. `code` derives from `statusCode` (`400→BAD_REQUEST`, `404→NOT_FOUND`, `409→CONFLICT`, `500→INTERNAL`). Example sources: `locate_session`/`project_read` 404, invalid `baseRef`/`promote_session` non-temp 400.

**Disk-backed reads (ring-first, fallback-on-demand).** The in-memory event ring is a fixed-cap (`DEFAULT_RING_CAP=2000`, drop-oldest) cache, but ring eviction is **invisible** to these two tools:
- `get_transcript({sessionId, sinceSeq, limit})` pages via `pageInstanceEvents` (`src/eventArchive.js`). Served from the ring; when `sinceSeq < trimmedBefore` (an evicted range) the on-disk session transcript is loaded and reconciled with the ring by `_seq` at a turn-aligned seam (no gap/dup). `sinceSeq` is forward paging (events with `_seq > sinceSeq`, oldest-first); the response adds `{hasMore, nextAfter}` — poll again with `sinceSeq = nextAfter`. `limit` clamped `[1,500]` (default 200). Replaces the old `truncated` field. Caveat: a single turn larger than the ring cap leaves a mid-turn `_seq` gap (covered by `get_recent_messages`).
- `get_recent_messages` is ring-first; only when the retained tail can't satisfy the requested recent **text** messages (and the ring was trimmed) does it read the on-disk transcript and merge by `msgId` (ring entry wins; bounded to the last `DISK_REPLAY_TAIL_CAP=5000` disk events). `source` reports `"ring"` vs `"disk"`. A missing transcript (exited temp session) degrades to ring-only + a `hint`; never crashes. On a **defaulted** call (no `count` passed) the ring must satisfy `n+1` text messages, not `n`, since the default path may bond in one preceding plan/question message (see below) — this raises the bar for staying ring-only before falling back to disk.

Note: the REST endpoints above keep their own `worktreeName` field — they are a separate contract from the MCP tools and were not renamed.

## Plugin system

### Manifest — `conductor.plugin.json` at the plugin project root

```jsonc
{
  "id": "code-hub",             // REQUIRED ^[a-z][a-z0-9-]*$ ≤40 chars, unique across the projects root
  "name": "Code Hub",           // REQUIRED display name
  "version": "0.3.0",           // REQUIRED informational
  "pluginApi": 1,               // REQUIRED int; conductor supports [1], anything else ⇒ state "incompatible"
  "backend": {                  // OPTIONAL (manifest-only/library plugins omit)
    "start": "npm start",       // bash -lc, cwd = active checkout, $PORT injected
    "healthPath": "/api/health",// optional readiness probe + on-demand liveness probe
    "readyWhen": "listening"    // optional stdout/stderr regex (precedence: readyWhen → healthPath → TCP)
  },
  "frontend": {                 // OPTIONAL, requires backend
    "path": "/",                // default "/"
    "navLabel": "Hub"           // default = name
  },
  "mcp": {                      // OPTIONAL, requires backend
    "endpoint": "/api/mcp",     // single POST endpoint on the child
    "scope": "project",         // accepted ("project" | "global") but INERT — tools are always globally visible
    "timeoutMs": 30000,         // per-call cap, clamped to 120000
    "tools": [{ "name": "...", "description": "...", "inputSchema": { "type": "object", ... } }]
  },
  "conventions": [              // OPTIONAL, NO backend required — project conventions
    { "slug": "visual-verification",           // ^[a-z][a-z0-9-]*$ ≤40, unique in this array
      "name": "Visual UX verification",        // REQUIRED
      "description": "verify UX via the harness", // REQUIRED
      "scope": "project",                      // REQUIRED enum — "project" only today (see below)
      "file": "conventions/visual-verification.md", // OPTIONAL CLAUDE.md fragment: relative .md path, no leading '/' or '..'
      "scaffold": {             // OPTIONAL facet (plugin-only) — a one-time project-setup directive
        "file": "scaffold/harness.md" }        // EXACTLY ONE of "text" | "file" (same path rules as conventions.file)
      // "scaffold": { "text": "Build a project-local harness wrapper ..." }  // inline form
      // At least one of "file" | "scaffold" is REQUIRED.
    }
  ]
  // "settings": reserved, validated-but-inert in v1
}
```

`conventions` is an **active pluginApi:1 capability** (an additive extension of v1, not a version bump — an existing manifest without it stays valid). It works with **no backend**, so a contributions-only plugin is `{id,name,version,pluginApi,conventions}`: it validates, enables, contributes, and is **never started** (its Settings row shows only Disable + contribution badges; `POST .../start` refuses 400). Entries join the project **Conventions** catalog namespaced `<plugin-id>/<slug>` (visible in the new-project dialog + `list_project_conventions`). Each entry carries a CLAUDE.md **`file`** fragment (applied inline to a project's CLAUDE.md at creation — the applied copy survives plugin disable/uninstall) and/or a **`scaffold`** facet (a one-time setup directive via exactly one of `text` | `file`; **plugin-only** — builtin/custom conventions are fragment-only); **at least one of `file`/`scaffold` is required**. A scaffold-bearing entry surfaces `hasScaffold:true` in `list_project_conventions`. Bodies for `file` refs (fragment and scaffold) are resolved against the active checkout; a **missing/stale `file` at manifest load makes the plugin `invalid`** (fail loud). Only **enabled + `ok`** plugins contribute — a disabled/crashed plugin never surfaces its conventions.

**Convention `scope` (required, explicit).** Every `conventions` entry MUST carry a `scope` — there is no silent default. The accepted enum is currently **`"project"` only** (routes into the project-Conventions catalog described above). `"workspace"` and `"conductor"` are recognised-but-rejected at manifest load with a specific error (`scope "workspace" not yet supported (only "project" is currently accepted)`); any other value gets the standard invalid-enum error and marks the plugin `invalid`. Expansion is **additive with no migration**: move a scope from planned→supported in `SUPPORTED_CONVENTION_SCOPES` (`src/plugins/manifest.js`) and wire that scope's group (from the scope-keyed object `pluginHost.conventions()` returns) into its catalog's provider (`workspaceModules`/`conductModules` already exist in `fragmentCatalog.js`). No stored data changes shape.

> ⚠️ **Conductor-scope conventions modify the orchestrator itself.** A future `scope:"conductor"` convention would inject a fragment into the conductor's **own** operating rules (the composed `.conduct/CONDUCT.md` modules) — i.e. a plugin could rewrite how the orchestrator delegates, reviews, and merges across every project. That is a deliberately powerful, trust-loaded capability, consistent with "plugins are trusted own code" but far beyond a per-project convention. It is intentionally **not** enabled yet; enabling it should come with a matching UX that makes the orchestrator-wide blast radius unmistakable.

**Scaffold delivery (conductor-directive, no persistence).** At project creation the picked conventions that carry a `scaffold` facet have their directive texts composed **in selection order** into one framed orchestrator-guidance block (`Project "<name>" was created with these setup steps…`) that `create_project` (MCP) / `POST /api/projects` (REST) **RETURN** under a `scaffold` field (empty/omitted when none). Nothing is persisted and nothing touches the spawn path. The conductor folds the returned `scaffold` into its **first** `send_prompt` to the project's first worker (see `conduct/core.md`); the UI shows it read-only in the create confirmation. The scaffold facet has **no `scope` field** by design: it fires at *project creation*, so it is inherently project-scoped — workspaces and the conductor aren't "created", so there is no scaffold trigger for them. *(Future work: persist a lightweight pending-scaffold the conductor can pick up for UI-created projects.)*

Unknown top-level keys are rejected. Every `inputSchema` must stay inside the conductor's `validateArgs` subset — a **flat** `type:"object"` schema (per-property `type`/`enum`/`minLength`/`maxLength`/`pattern`/`minimum`/`maximum`/`items.type`; boolean `additionalProperties` accepted and ignored). `$ref`/`oneOf`/`anyOf`/`allOf`/`not` and nested object `properties` are rejected at manifest load, so an unvalidatable schema can never register. Invalid ⇒ state `invalid`, listed with errors, never startable.

Child env: `$PORT` (conductor-allocated; default your own port when absent so the plugin stays standalone-runnable), `CONDUCTOR_PLUGIN_ID`, `CONDUCTOR_URL` (`http://127.0.0.1:<conductor-port>`). No fixed-port option in v1.

**Discovery rules.** The manifest is read from each project's main checkout. When the main checkout has **no manifest file at all**, the project's worktrees (sorted by name) are checked and the first **valid** manifest wins — so a plugin whose manifest exists only in an unmerged worktree (first-time plugin-ification) can bootstrap. Rows carry `manifestSource: {type:"main"} | {type:"worktree", name}`; enabling a worktree-sourced plugin defaults `activeVersion` to that worktree. A present-but-invalid main manifest keeps its `invalid` state (never masked by a worktree), and `POST /api/plugins/:id/version {type:"main"}` is refused with 400 while the main checkout lacks a valid matching manifest.

### Reverse proxy — `/plugins/<id>/*`

- `/plugins/<id>/foo?q=1` → child `/foo?q=1` (prefix strip). Injected headers: `X-Forwarded-Prefix: /plugins/<id>`, `X-Forwarded-Host`, `X-Forwarded-Proto`, `X-Forwarded-For`.
- `GET /plugins/<id>` → `301 /plugins/<id>/` (query preserved). Child `Location:` headers starting with `/` get the prefix re-added — the only header rewrite.
- Bodies are never parsed — pure req→upstream→res streaming (SSE works). WebSocket upgrades are raw-socket piped with the original header order/casing replayed.
- Requests to an enabled-but-stopped plugin wait through the lazy start (≤30 s). Unknown/disabled id → 404 JSON; `failed` → `503 {error, status:"failed", tail}`; crash-backoff window → `503 {error, status:"crashed", retryAfter, tail?}`; child unreachable mid-request → 502.

### Bridge protocol — `/pluginBridge.js`

Plugin frontends include `<script src="/pluginBridge.js" defer></script>` (served by the conductor; harmless 404 standalone, no-op when not iframed). Envelope `{cc:1, type, ...}` over `postMessage`, same-origin both ways. Exactly three messages:

| Direction | Type | Payload | Meaning |
|---|---|---|---|
| child → parent | `ready` | — | bridge alive (initial `route` follows) |
| child → parent | `route` | `{path}` | child-relative path (incl. search+hash); parent mirrors it into `#plugin/<id><path>` via `replaceState` |
| parent → child | `navigate` | `{path}` | external navigation; bridge `replaceState`s `<prefix><path>` and dispatches a synthetic `popstate` |

Inside the iframe the bridge patches `history.pushState` → `replaceState`, so a plugin visit adds exactly one joint-history entry (hardware Back exits to the conductor). Multi-page plugins bypass this and pollute history.

### REST — `/api/plugins`

| Method + path | Meaning |
|---|---|
| `GET /api/plugins` | merged discovery+registry+runtime rows: `{id, name, project, version, state, enabled, activeVersion, manifestSource, hasBackend, hasFrontend, navLabel, frontendPath, hasMcp, conventions:[{slug,name,description,hasScaffold}], port, pid, startedAt, gitHead, stale, errors, crashTail}` (convention slugs namespaced `<plugin-id>/<slug>`; `hasScaffold` flags a convention carrying a one-time scaffold directive). A backendless (contributions-only) enabled plugin has `state:"enabled"` (never `"stopped"`). `stale` is true only while `state:"ready"` and the active checkout's current HEAD differs from `gitHead` (the sha the child was started at); a non-git checkout or an unreadable HEAD is never stale. |
| `POST /api/plugins/rescan` | re-scan the projects root (auto-assigns any unassigned discovered plugin project to workspace `CC-Dev`); returns the list |
| `POST /api/plugins/:id/enable` | record + enable; recovery path out of `failed` (workspace auto-assign to `CC-Dev` happens on discovery, not enable specifically — see `rescan` below) |
| `POST /api/plugins/:id/disable` | stop the child + disable |
| `POST /api/plugins/:id/start` | explicit start (clears crash history); 502 + `tail` on start failure |
| `POST /api/plugins/:id/stop` | SIGTERM the process group (SIGKILL after 3 s) |
| `POST /api/plugins/:id/restart` | stop + start the running child in place (picks up new code from the active checkout); 409 if not running |
| `GET /api/plugins/:id/status` | row + live probe (flips a silently-dead child to `crashed`) |
| `POST /api/plugins/:id/version` | `{type:"main"}` \| `{type:"worktree", name}`; validates the target checkout (400 keeps previous state), restarts if running |
| `GET /api/plugins/library` | Plugin Library catalog: `{id, name, description, repo, installed, installedAs}[]` — `installed` is true when the repo's derived target directory already exists under `projectsRoot()`. |
| `POST /api/plugins/library/:id/install` | clone the entry's `repo` into `<projectsRoot>/<name>` (name derived from the URL), rescan, then run `postClone` if set; **never enables/starts** the result. `{id, name, project, path, postClone}` on success (`postClone` is `null` when unset, else `{ran:true, ok, code, tail}` — **a failed `postClone` does NOT fail the request or remove the clone**, see below). 404 unknown library id, 400 invalid/disallowed repo URL scheme or a name that can't be derived, 409 already installed, 502 + `tail` on clone failure. |
| `POST /api/plugins/library/:id/update` | `git pull --ff-only` in `<projectsRoot>/<name>`, rescan, then run `postPull` if set. `{id, name, project, path, postPull}` on success (same shape as `postClone`). 404 unknown library id or not installed, 502 + `tail` on pull failure (never attempts a merge). |

Errors: `{error}` JSON with 400/404/409/502/503 per the registry rules above.

#### Plugin Library — drop-in manifest, `<orchStoreRoot()>/plugins/library/*.json`

One JSON object per file (any filename ending `.json`), registering an installable plugin entry:

```jsonc
{
  "id": "my-plugin",             // REQUIRED, unique catalog key — overrides a built-in entry with the same id
  "name": "My Plugin",           // REQUIRED display name
  "description": "optional",     // OPTIONAL
  "repo": "https://github.com/org/my-plugin", // REQUIRED clone URL — scheme must be http:, https:, or git:
  "postClone": "bash install.sh", // OPTIONAL shell command, run via `bash -lc`, cwd = the cloned project dir
  "postPull": "bash install.sh"   // OPTIONAL shell command, run via `bash -lc` after a successful Update pull
}
```

A built-in `code-share` entry (`https://github.com/UnmanagedCode/code-share`, no post-hooks) and a `code-playwright` entry (`https://github.com/UnmanagedCode/code-playwright`, `postClone`/`postPull` both `bash install.sh` — Playwright + Chromium glue for visual UI debugging) are always present, even with no library dir. Malformed JSON or a file missing `id`/`name`/`repo` is skipped with a `console.warn` — never fatal to the list. The install target project name is the URL's last path segment with a trailing `.git` stripped (e.g. `.../org/my-plugin(.git)` → `my-plugin`), validated the same way as any other project name.

**`postClone`/`postPull` execution.** Run bounded (5 min timeout) via a detached process group so a command that spawns children of its own (`npm install`, a browser-binary downloader) can be fully killed on timeout, not just its direct child; output is captured (16 KB running cap, 4000-char tail surfaced in the response). This is a code-execution surface — acceptable because built-in entries are trusted and drop-in files come from trusted local tooling (the same trust stance that already applies to a plugin's own manifest `backend.start`). **Asymmetric failure handling is intentional:** a failed `git clone`/`git pull` is a hard failure (the request rejects; a failed clone is also rolled back) because the underlying operation itself didn't succeed, whereas a failed `postClone`/`postPull` is reported as a **soft warning on an otherwise-successful response** — the clone/pull already succeeded and is already discoverable, only the convenience command failed. The documented retry path for a failed `postClone` is hitting **Update** (which reruns `postPull`) rather than reinstalling — `code-playwright` sets both fields to the identical command specifically so Update is a true retry.

### Plugin MCP forwarding — child wire contract (pinned)

The conductor POSTs `{tool, arguments, caller:{sessionId, project}}` (JSON) to the manifest `mcp.endpoint`. The child returns **HTTP 200 for EVERY well-formed tool invocation** with body `{result: <any JSON>}` or `{error: "<message>"}` — unknown tool, bad arguments and tool-level failures are all `200 + {error}`. A **non-200 means a transport-level failure only** (malformed envelope, plugin bug) and surfaces to the MCP client as an HTTP-coded error; `200 + {error}` surfaces as a plain tool error. Calls are aborted at `mcp.timeoutMs`. Tool names are namespaced `<plugin-id>__<tool>`; argument validation against the declared `inputSchema` happens in the conductor **before** any forward. Visibility: every enabled plugin's tools are offered to **every** MCP caller — the conductor UI and workers in any project (`scope` is inert); a disabled plugin's tools are absent, so calling one refuses as an unknown tool.

### Plugin-compliance checklist

1. `conductor.plugin.json` at the repo root (schema above); keep `id` stable.
2. Respect `$PORT` when set; default your own port so the app stays standalone-runnable.
3. Base-path compliance: reachable under `X-Forwarded-Prefix` — relative asset URLs (or honor the prefix), root-relative redirects only (they get rewritten).
4. `<script src="/pluginBridge.js" defer></script>` in the frontend + SPA routing via pushState/replaceState (the bridge reports routes for you).
5. A `healthPath` endpoint (any HTTP response counts as alive).
6. Optional MCP endpoint following the 200-always contract above, tools declared in the manifest with flat schemas.
7. Expect to be killed at any time (Doze) and restarted lazily — persist state, start fast.
8. If a convention's fragment depends on something its `scaffold` facet sets up, **word the fragment to degrade gracefully** when the scaffold step wasn't run — the picked convention may land in a project where the setup directive was never carried out (e.g. "if a project-local harness wrapper exists, use it to visually verify UX changes; otherwise see the shared harness to create one").
