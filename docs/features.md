## Features

### Projects & workspaces
- **Project list** ‚Äî sidebar shows every dir under `<projectsRoot>/`. `+ New project` creates a dir with `CLAUDE.md` containing `@../CLAUDE.md`, optionally followed by inline snapshots of selected rule modules (see **Rule modules** below). Worktree-owned dirs are hidden from the list (filtered against a Set built from the central store).
- **Delete project** ‚Äî `√ó` per row, typed-name confirm; cascades through instances + worktrees then `rm -rf`s the project. `~/.claude/projects/<encoded>/` jsonls are left intact (still resumable by the standalone CLI).
- **Workspaces** ‚Äî `‚â°` hamburger menu hosts `+ New workspace`; nest projects under collapsible `<details>` headers, one workspace per project, empty workspaces persist. Edit (rename / re-pick members / delete) via the ‚úé button on each header. Workspaces default-collapsed; expand state persists in `localStorage` (`code-conductor:workspaces-expanded`). Stored server-side at `<store>/projects/<name>/project.json`; the workspace set is the union of `<store>/workspaces.json` and any referenced names. Workspace names use the same `^[a-zA-Z0-9._-]+$` regex as project names.
- **Resizable sidebar** ‚Äî desktop grid layout only (>720px). A drag handle on `#sidebar`'s right edge (`#sidebar-resize-handle`) resizes the column live via the `--sidebar-width` CSS variable (default `280px`), clamped to 220‚Äì560px. Persisted in `localStorage` (`code-conductor:sidebar-width`). Hidden below the 720px breakpoint, where the sidebar is a fixed-width slide-in drawer instead.
- **Git status pill** ‚Äî
  - **Project rows**: amber `‚ÜëN ‚ÜìM` against the currently-checked-out branch's *configured upstream* (e.g. `origin/<branch>`), using cached refs only (no `git fetch`). No pill when in sync / detached / upstream-less.
  - **Worktree rows**: same shape but compares the worktree branch to its captured *parent base branch* (may be a local branch, not a remote). `‚ÜëN` = ahead and FF-able, `‚ÜìM` = behind and FF-able, `‚ÜëN ‚ÜìM` = diverged ‚Üí rebase needed before merge.

### Worktrees
- **Isolated environments** ‚Äî "Run in isolated git worktree" checkbox in the new-instance dialog runs `git worktree add ../<project>_worktree_<short-id> -b code-conductor/<short-id> <currentSha>` and spawns the agent with `cwd` set to the worktree. The orchestrator captures `{baseBranch, baseSha, branch}` at creation so you can spawn off any branch and land back later. Worktrees survive instance death ‚Äî re-spawn into the same worktree from the sidebar.
- **Two-step rebase-back**:
  - **Sync** ‚Äî already in sync ‚Üí no-op; purely behind & clean ‚Üí server-side `git merge --ff-only <baseBranch>`; diverged & clean ‚Üí orchestrator attempts `git rebase <baseBranch>` directly (returns `rebased` on success); if rebase hits conflicts, it aborts cleanly and falls back to sending the agent a templated rebase prompt (`rebase-prompt-sent`; agent runs `git rebase` itself, asks the user via `AskUserQuestion` on non-trivial conflicts, replies `REBASE_DONE`); behind-but-dirty ‚Üí rebase prompt sent directly (agent commits/discards first).
  - **Merge** ‚Äî `git merge --no-ff --no-edit <worktreeBranch>` on the parent. Always produces a merge commit (default msg `Merge branch 'code-conductor/<id>'`) so each contribution stays visible in `git log --graph` and is revertible as one commit. Refuses inline (not as a 500) if the worktree is behind the parent, the parent is on a different branch than `baseBranch`, or the parent tree is dirty. On success, also fast-forwards the worktree's own branch up to the new merge commit (best-effort ‚Äî reported as `worktreeFastForwarded: true/false`) so a kept worktree lands at `ahead:0/behind:0` instead of looking one commit behind.
- **Delete** ‚Äî refused if there's a live instance or uncommitted work, with a `force=1` override.
- **Post-worktree-create hook** ‚Äî a `post-worktree-create.sh` script (need not be committed ‚Äî read from disk, not the new worktree) is resolved from two locations in priority order: **in-tree** `<parent>/.code-conductor/post-worktree-create.sh` first, else the **central store** `<root>/.code-conductor/projects/<name>/post-worktree-create.sh`. When found, the orchestrator runs it automatically with `cwd` in the new worktree, after the worktree directory, branch, and metadata are written. Use case: symlink `node_modules`, copy `.env`, run `npm ci`, etc. The script receives env vars `CC_WORKTREE_PATH`, `CC_PROJECT_NAME`, `CC_BRANCH`, `CC_BASE_BRANCH`, `CC_PARENT_PATH`. Non-fatal: a failure warns but does not roll back the worktree. Output is capped to the tail ~16 KB (`truncated:true` when exceeded). Timeout: 120 s (`ORCH_POST_WORKTREE_TIMEOUT_MS`). Kill-switch: `ORCH_DISABLE_POST_WORKTREE_HOOK=1`. Result is reported in the spawn/worktree-create response as `postWorktreeCreate: { ran, source, exitCode, durationMs, output, truncated?, timedOut? }` (`source` is `'in-tree'` or `'store'`). The spawn dialog re-opens briefly with a one-line summary (expandable output) when the hook ran.

### Diff / review & commit history
- **Worktree diff/review** ‚Äî `¬±` button on every worktree row opens a full-page, mobile-friendly diff browser (`#review-view`, `public/review.js`). Fetches structured per-file diff JSON from `GET /api/projects/:name/worktrees/:wt/diff` (`<baseBranch>...HEAD`) and renders collapsible `<details>` per file (status badge, `¬±` counts, hunks reusing `.diff-line.add/.del/.ctx`, `overflow-x:auto` for horizontal scroll). Diff capped at ~200 KB ‚Üí `(truncated)` in the header.
- **Commit history** ‚Äî `‚â°` button on every git-repo project row opens a full-page scrollable commit list (`#commits-view`, `public/commits.js`) of the project's **current branch (HEAD)**: short sha, subject, author, relative date, newest first. Backed by `GET /api/projects/:name/commits` (default 100, max 500; header shows `(showing latest N)` when truncated). Tapping a commit opens the **same diff renderer** showing just that commit's change (`GET /api/projects/:name/commits/:sha/diff`, via `git show`); the diff layers over the list and `‚Üê Back` returns to it. Merge commits render as "No changes" (combined diff not parsed). A **branch/merge graph rail** (git-log --graph style) is drawn to the **left** of each row: colored lanes, a dot per commit, diagonals where branches fork and merges converge. Topology is computed client-side in `computeGraph()` (`public/commits.js`) from each commit's `parents` (SHAs from `%P`); lanes use stable column indices so pass-through lines stay aligned. Each rail is a fixed-height SVG "node band" (dot + diagonals) plus full-height CSS filler lines, so it aligns with variable-height (wrapping) rows without measurement. Parents older than the cap / on unfetched branches simply trail off the bottom. Soft cap of 12 lanes (extras clamp to the last column, logged to console).

### Sessions & instances
- **Unified view** ‚Äî Sessions list per project (and per worktree) shows live + historical. Header: `Sessions (N) ¬∑ K live ¬∑ last <ago>`. Click to focus live instance, or resume stopped one (`POST /api/instances` with `--resume <sid>`). Synthetic `(new session)` row for spawned-but-no-jsonl-yet instances. Rows bucket into up to three sections: the normal list, then a dim `‚Äî temp ‚Äî` separator, then a dim `‚Äî conducted ‚Äî` separator (each section only renders when non-empty).
- **Conducted sessions** ‚Äî a session spawned via the MCP `spawn_instance` tool (orchestrator-driven) rather than the browser/HTTP path is marked **conducted** (a worker an orchestrator conducts; the orchestrator itself is the `.conduct` session). Orthogonal to temp: unlike temp (in-memory, wiped on exit) the conducted marker is **durable** ‚Äî persisted to `<store>/conducted-sessions.json` (a `{sessions:[sid,‚Ä¶]}` set, written on turn-end via the same path as last-prompt/permission-mode metadata) so it survives exit, restart, and `--resume` (a UI-resumed conducted session re-acquires the marker from the sidecar). Set **only** in the MCP `spawn_instance` handler ‚Äî plain UI spawns stay non-conducted. Grouping precedence: a conducted session renders under `‚Äî conducted ‚Äî`; a conducted **and** temp session renders under `‚Äî conducted ‚Äî` but keeps the warm temp preview colour (`.session-row.temp`); a temp-but-not-conducted session stays under `‚Äî temp ‚Äî`. Pure conducted rows get a cool preview tint (`.session-row.conducted:not(.temp)`). It's a grouping, never a filter ‚Äî conducted sessions stay fully visible in `list_sessions`, the HTTP session APIs, and the sidebar. The marker is cleaned up alongside the session-title sidecar on session delete / temp-exit / boot sweep.
- **Unread indicator** ‚Äî small accent pill next to a row when its instance finishes a turn while you're viewing a different session (per-instance, persisted in `localStorage` key `code-conductor:unread`).
- **Archive session (the `√ó` action)** ‚Äî the hover-revealed `√ó` on a session row **archives** it (never deletes): single confirm, `POST ‚Ä¶/sessions/:sid/archive`; on 409 the client auto-retries with `?force=1` (stops the live instance) without a second prompt. Archived sessions keep their jsonl (still resumable) + title + conducted markers and **disappear from the sidebar entirely** ‚Äî they're managed only from **‚öô Settings ‚Üí Archived**. There is no longer an inline `‚Äî archived ‚Äî` sidebar section, and sessions are never deleted from the sidebar.
- **Archived page (Settings ‚Üí Archived)** ‚Äî lists every archived session grouped by owning project (`GET /api/archived`). Each project group is a `<details>` **collapsed by default**; expanding reveals its archived sessions (title/first-prompt/sid + worktree tag + last-active). Per session: **Restore** (`POST ‚Ä¶/restore` un-archives it, then resumes/spawns it and navigates straight to it, closing Settings ‚Äî restoring alone doesn't create a live instance, so the client drives `resumeSession` itself via the `onSessionRestored` callback) and **Delete** (confirm ‚Üí `DELETE ‚Ä¶/sessions/:sid`, the single permanent jsonl-removal path; cannot be undone). The `.conduct` (orchestrator) project's group is always pinned first and rendered as `üéº Conductor` instead of its raw folder name ‚Äî sorting/labeling happens client-side in `renderArchived()`.
- **Session summary** ‚Äî `‚ãÆ` menu's `üìù Summarize` opens a modal dialog that generates and displays an AI-written summary of the session via a one-shot `claude -p` call (Haiku). Three tiers: **Short** (~40 content words, plain prose gist), **Medium** (~150 words, scannable recap with bullets/tables), **Long** (~400 words, thorough with `##` headings and structured tables). Summaries are rendered as full markdown (headings, tables, code) via the same `renderMarkdownInto` pipeline as assistant messages. A **stale indicator** (`‚öÝ Session has grown since this summary`) appears when the session has received more turns since the last generation; regeneration is always user-initiated. Stored in `<store>/session-summaries.json` keyed by `sessionId` ‚Äî `{ summary, length, generatedAt, messageCount }`; auto-removed when the session jsonl is deleted. `sessionId` is stable across `--resume`, so summaries simply persist across a resume.
- **Rename session** ‚Äî `‚ãÆ` menu's `‚úèÔ∏è Rename` opens a prompt to set a custom human-readable label (‚â§100 chars, empty clears) that overrides the first-prompt preview in the sidebar and prepends an italic chip to the active header. Stored in `<store>/session-titles.json` keyed by `sessionId`; auto-cleaned when the session jsonl is deleted or a temp session exits. Survives rewind (same `sessionId`) and `--resume` (`sessionId` is stable across resume); fork starts un-titled.
- **Change model** ‚Äî `‚ãÆ` menu's `üßÝ Change model` (directly below Rename) opens a capability-tier picker (Fast / Balanced / Powerful / Frontier) using the same catalog/source of truth as the spawn dialog (`Settings ‚Üí Models`). Selecting a tier resolves it to its bound `{backend, model}` and switches the **live** session's model via a `control_request` (`subtype:set_model`) ‚Äî no restart, no history loss. The picker highlights the tier the running session is on with `.qs-selected`. **Ollama-backed sessions can't switch model live** (the endpoint is fixed at launch): every tier is disabled with a "kill and respawn on that tier" note, and switching a Claude session *to* an Ollama-bound tier is likewise blocked (server-enforced too ‚Äî `409 BACKEND_KIND_LOCKED`).
- **Rewind & fork** ‚Äî every user message bubble has `‚Ü∂` (rewind in place, same `sessionId`) and `‚ëÇ` (fork to a new `sessionId`) on hover. Both atomically rewrite the jsonl, preserve the dropped prompt in the composer, and 409 during a running turn. Bubbles anchor on the **server-stamped `userIndex`** riding on each outer `user_echo` (the absolute Nth-pure-user-prompt ordinal) ‚Äî never a client-side bubble count, which would drift once the capped ring / tail-only snapshot starts the view mid-history. An echo without a stamp (rare orphan) renders without the buttons.
  - Rewinding **to the first user message** is special-cased: prefix is empty, so the orchestrator deletes the jsonl entirely and respawns with `--session-id <sid>` (not `--resume`, which the CLI rejects on zero-line files).
  - **Temp sessions cannot be forked** ‚Äî the on-exit cleanup races the fork's source copy.
  - UI-only ‚Äî no MCP equivalent yet.
- **Crash recovery** ‚Äî subprocess death ‚Üí `crashed`; Resume respawns with `--resume <sid>`, preserving the in-memory event ring + conversation.
- **Session anchor** ‚Äî `#session=<sid>` URL hash; on reload, auto-resumes by locating the jsonl on disk (worktree-aware) via `GET /api/sessions/:sid/locate`. `--resume` spawn costs zero API tokens until the next prompt. Stale anchors are silently cleared.

### Spawn options
- **Mode** ‚Äî `plan` (read-only, propose then `ExitPlanMode`), `ask` (every destructive tool gates through Allow/Deny card), `code` (full power, CLI's `bypassPermissions`). CLI's `default` / `acceptEdits` are not exposed: in `stream-json --print` (no SDK `canUseTool` callback) they auto-deny tool calls and the only recovery is forcing the model to regenerate the entire tool input.
- **Effort** ‚Äî `low` / `medium` / `high` (default) / `xhigh` / `max`.
- **Thinking** ‚Äî `adaptive` (default) / `enabled` / `disabled`.
- **Model** ‚Äî empty (account default) or pick a capability tier (Fast / Balanced / Powerful / Frontier; a tier hidden if disabled in Settings ‚Üí Models). Each tier binds to a **backend + model** (Settings ‚Üí Models): backend is **Claude** with a specific Claude version (defaults Fast‚ÜíHaiku, Balanced‚ÜíSonnet, Powerful‚ÜíOpus, Frontier‚ÜíFable 5) **or Ollama** with a custom model tag. An Ollama-bound tier resolves straight to its tag (no version choice) and spawns via `ollama launch claude` (see [protocol.md](protocol.md#subprocess-protocol)); its native context window comes from the catalog (`src/ollamaCloudModels.js`, e.g. 1M for DeepSeek/GLM/MiniMax, 256k for Qwen/Kimi/Mistral) or the custom model's declared window, and drives both the context-usage bar and `CLAUDE_CODE_AUTO_COMPACT_WINDOW` at spawn. Claude context windows: **Haiku 200k**, **Opus / Fable 5 1M**, **Sonnet 4.x** the stored preference (defaults 1M), **Sonnet 5 1M** (no 200k build). The window is re-derived on resume (`canonicalizeModel` in `src/modelVersions.js`), never persisted in the jsonl. MCP `spawn_instance` also accepts a legacy family alias (`opus`/`sonnet`/`haiku`/`fable`, ‚Üí that family's default version) or a full Claude model id.
- **Temp session** ‚Äî deletes the jsonl + sibling `subagents/` dir on exit, drops itself from the sidebar's Sessions list on exit/crash (no ghost row to clean up by hand), and skips `last-prompt`/`permission-mode` metadata appends during the run; mode defaults to `code`; header shows a `TEMP` pill.
- **Debug mode** ‚Äî mirrors raw CLI traffic (`claude-stdin.jsonl`, `claude-stdout.jsonl`, `claude-stderr.log`, `meta.json`) to `<store>/projects/<p>/[worktrees/<wt>/]debug/<instance-id>/`. Append-mode, survives session end. Header shows a `DEBUG` pill.

### Live conversation
- **Tail-first load + scroll-up history** ‚Äî subscribing renders only the ring tail (‚â§500 events, quiescent-aligned: whole blocks, resolved tools). When older history exists, a tappable `‚ãØ earlier messages` sentinel sits above the oldest content and scrolling near the top auto-fetches the previous page (`GET /api/instances/:id/events?before=<cursor>&limit=200`, flipping the sentinel to `loading earlier‚Ä¶`). Pages render through the standard block pipeline on a detached node and splice in above with the viewport preserved; a turn split across page seams **merges back into one assistant bubble** (the batch's trailing open bubble concatenates into the chunk below), and sub-agent (Task) groups always travel whole with their nested panel ‚Äî live children of a not-yet-loaded Task head are parked and adopted into the panel once its page loads. Events evicted from the in-memory ring are reconstructed from the session jsonl. Caveats: the task panel and usage chip rebuild from the tail only, and a single turn longer than the ring cap pages back with a gap (never duplicated content) ‚Äî rendered as a `‚îÄ‚îÄ ‚ãØ earlier messages unavailable ‚îÄ‚îÄ` divider at the exact seam; only whole blocks are ever missing, and a genuinely interrupted tool renders `¬∑ incomplete` rather than a stuck spinner.
- **Text** ‚Äî streams as text-node deltas; on `text_end` re-renders through `public/markdown.js` (headings, lists, code, **bold**, *italic*, links, autolinked bare `http(s)://‚Ä¶` URLs, `![alt](src)` images). Link schemes restricted to `http(s)/relative/fragment/mailto`; image `src` to `http(s)/file://`/absolute path; no `innerHTML`; raw HTML in source rendered as literal text.
- **Text-to-speech (read aloud)** ‚Äî the inverse of voice dictation: finalized assistant text blocks get a trailing `üîä` button (tap to hear it), and a global **auto-speak** toggle reads each reply as it lands. Local server-side Piper ‚Äî no network, no API key. Synthesis is **streamed per sentence**: `POST /api/tts` returns a sequence of `[4-byte LE length][WAV]` frames (one per sentence) flushed as Piper yields them; the client (`public/tts.js`) decodes each with the Web Audio API and schedules them back-to-back, so the first sentence starts playing (~0.3s) while the rest synthesize. The üîä button appears once Piper is installed (`GET /api/tts/status` gates it; Settings ‚Üí TTS installs/flips it at runtime). Auto-speak only fires after the user has interacted at least once (browsers block audio without a prior gesture); a üîä tap always works. Sub-agent text isn't auto-spoken.
- **Thinking** ‚Äî collapsible block; redacted thinking renders as a non-expandable `thinking (redacted)` line (no disclosure caret). Opus 4.7 emits only a `signature_delta`; Opus 4.8 streams empty `thinking_delta`s (`""`) ‚Äî the parser drops the empties so both collapse to the same redacted line.
- **Tool use** ‚Äî collapsed by default with smart one-line summary (`üîß Bash ¬∑ ls -la ¬∑ done`). Edit/Write/NotebookEdit render as syntax-coloured unified diffs (green/red gutters, ¬±counts, sticky file-path); Write shows a numbered preview. Other tools: raw-JSON input in its own collapsed `‚Ü™ tool_input` block.
- **Skill loading** ‚Äî a `Skill` tool call's SKILL.md content, which the CLI injects back as a plain (large) user message, renders as a dedicated collapsed `üìò Loading skill: <name>` bubble instead of a giant plain-text bubble; tap to expand the full content. Server-side correlation with the preceding `Skill` tool_use (not just the CLI's `isSynthetic` flag, which it also reuses for unrelated compaction/hook text) ‚Äî see `docs/protocol.md`.
- **Tool result** ‚Äî truncated at 4 KB with "show full". `image` content blocks (e.g. `Read` on a `.png`/`.jpg`) render as inline `<img>` thumbnails alongside the text ‚Äî base64 sources become `data:` URLs, `url` sources only pass through for `http(s)`/`file://`, `image/svg+xml` is refused (XSS). Auto-expanded when an image is present. Tap any conversation image to open it full-size in an in-page lightbox (`public/lightbox.js`) ‚Äî works for `data:`/`file://` sources that Chrome on Android refuses to open as a top-level navigation. Opens fit-to-screen; tap the image to toggle 1:1 full native resolution (backdrop scrolls to pan), tap again to fit; tap backdrop or press Esc to close.
- **Sub-agent drill-down** ‚Äî `Task` tool calls nest a mini-conversation (dashed border, `‚Ü≥ sub-agent` label) inside the parent tool block, routed by `parentToolUseId`.
- **Plan mode card** ‚Äî green-bordered card "Plan ready for approval". Body comes from `input.plan`, or ‚Äî when the model wrote the plan to `~/.claude/plans/*.md` first and called `ExitPlanMode` with empty input ‚Äî the most-recent such file. **Approve** sends `"I approve the plan. Please proceed with the implementation."` (+ feedback if provided) and flips the instance to `code` (`bypassPermissions`). **Reject** keeps plan mode active and sends `"I'd like to revise the plan. Refinement notes:\n<feedback>"`. **üìã Auto-approve plans** toggle sits inline immediately left of the mode dropdown and is only visible while the instance is in `plan` mode (one-click, mid-turn friendly); per-instance, server-side state (mirrored down through `snapshot`/`status` WS frames so every tab agrees), cleared on server restart. When on, the orchestrator fires the same setMode+approval-prompt the moment a `plan_request` lands ‚Äî works regardless of whether any tab is subscribed / focused / minimized. Green-tinted background when on, outlined when off.
- **AskUserQuestion card** ‚Äî blue card; multi-question tab strip across the top. Same input field flips role: **Other:** before any option picked (overrides), **Add a note (optional)** after a pick. Typed text persists across pick/unpick. **Send all answers** enables once every question is answered and queues if instance is busy (flushes on next `status=idle`).
- **Ask-mode card** ‚Äî purple-bordered, with tool name + summary + full diff/Write preview. **Allow** ‚Üí `permissionDecision:"allow"` resolves the held-open HTTP hook response; CLI proceeds with the original `tool_use_id` (no model regeneration). **Deny** ‚Üí inverse. `permission_resolved` flips the card to ‚úì/‚úó across tabs.
- **System notes** ‚Äî kept: `init`, `stderr`, `exit`, `permission_denied`, `compacting`, `spawn_error`, `crashed`, `history_load_error`, non-allowed `rate_limit_event`. Filtered: per-turn `status:"requesting"`, `rate_limit_event:"allowed"`, hook lifecycle pings, task progress. Non-allowed `rate_limit_event` renders as a parsed one-liner (e.g. `‚öÝ Usage 75% of 5-hour limit ¬∑ resets 6:40pm`, `‚õî 7-day limit reached ¬∑ resets Sat 5:00pm ¬∑ overage unavailable (out of credits)`), not raw JSON.
- **Turn footer** ‚Äî duration, cost, tokens.

### UI elements
- **Task panel** ‚Äî strip above composer mirroring `TaskCreate`/`TaskUpdate`. Groups into batches: while at least one task is pending/in-progress, the whole batch (including its ‚úìs) stays. Panel hides only when *all* batches are fully completed; a new task post-batch drops the historical ‚úìs and starts fresh.
- **Footer status bar** ‚Äî thin row pinned between task panel and composer. **Left**: pulsing green dot + animated `Claude is working‚Ä¶` ellipsis while the active instance is in `turn` status (respects `prefers-reduced-motion` ‚Äî stays visible but stops animating); once idle, the same slot shows a live `last response Xm ago` label (`Instance.lastResponseAt`, stamped on `turn_end` and surfaced via `summary()`; formatted with the shared `formatAgo()` from `sidebar.js`, ticking every 15s via `header.js`'s `tickIdleAgo()` ‚Äî a scoped refresh that avoids `update()`'s popover-closing side effects) ‚Äî hidden entirely for a session that hasn't completed a turn yet, rather than showing a "never" label. **Right**: the combined ctx + rate-limit chip. Collapses when no instance is selected.
- **Combined context-usage + rate-limit chip** ‚Äî `ctx N% ¬∑ 245k/1M` pinned at the footer right (the turn-indicator footer). `N%` = latest `message_start`'s `input_tokens + cache_read_input_tokens + cache_creation_input_tokens` over the model's context window. Updates **live mid-turn** (each agent-loop step fires a `message_start`). Colour-graded green/amber/red at 50%/80%. The same chip carries the **rate-limit** half ‚Äî live bucket / utilization / reset-time, with an OVERAGE badge when `isUsingOverage` ‚Äî and tapping opens the usage popover (session totals: turns, duration, cost, uncached input, output, cache reads + hit ratio, cache creation; plus account rate-limit windows). Window sizes hardcoded per family (suffix-stripped before lookup): Haiku ‚Üí 200k, Fable 5/Opus ‚Üí 1M, Sonnet 5 ‚Üí 1M, Sonnet 4.x ‚Üí the stored preference. **Known limitation:** for a bare Sonnet 4.x id the ctx % falls back to the stored preference rather than session-specific truth, so it can mis-report if the preference changed after that session was spawned; Sonnet 5 has no such ambiguity (always 1M).
- **Cost dashboard** ‚Äî a full-page `#costs` view (`public/costs.js`, built on the `hashView` scaffold) breaking down spend from the append-only per-turn cost log (`<store>/costs.jsonl`, written by `src/costTracking.js` on every `turn_end`). Shows total spend + turn count, an expandable **By project** table (per-project cost/turns/**sessions**/**cache misses**, drill-down to per-model tokens: input / output / cache-create / cache-read / turns / **sessions** / **cache misses**), a **By model** table (same columns), and a **Daily spend** bar trend. The **Cache misses** column on each table is a plain count of that model's/project's rows with the persisted `cache_miss` flag `=== true`; the **Sessions** column is the count of distinct `sessionId`s among that model's/project's rows (rows with no `sessionId` aren't counted). Backed by `GET /api/costs/summary`; shows a "no cost data recorded yet" empty state until the first turn lands.
  - **Cache-miss detection** is **cross-turn**, so it catches *partial* (minority) evictions the old stateless rule missed. Captured live in `src/instances.js`, it compares this turn's **first**-request `cache_read` (`read_N`, pre-tool-call) against the **previous** turn's **last**-request full prefix (`P = cache_read + cache_creation`, the fully-accumulated cacheable prefix ‚Äî cache_read accumulates across a turn's tool-call iterations, so the first request is used for read_N and the last for P). A miss is flagged when `read_N < P ‚àí tolerance` (`tolerance = max(1024, 1% of P)`): this turn served less of the prefix than was demonstrably cached last turn ‚áí an eviction, full or partial. A warm continuation reads exactly `P` (drop 0) and is never flagged. **Turn 1** (no prior `P`) and the turn after a **compaction / model switch / rewind** fall back to the old stateless `creation > read` rule, which still flags a genuinely cold prefix (cold start, expiry, resume, cold rewind). **Guards:** compaction (`system`/`compacting`), model switch (`_trackModel`), and rewind/respawn (`_wipeForResume`) legitimately shrink the prefix, so each sets a `_prefixBaselineInvalid` flag consumed on the next turn's first request ‚Äî the cross-turn rule is skipped there and `P` is re-established fresh, preventing false positives. **Known limitation:** a miss on a request *after* the first (e.g. a turn outliving the ~1h cache TTL) isn't caught ‚Äî detection is first-request-only. The verdict + evidence persist on each row (`cache_miss`, `first_req_cache_read`, `first_req_cache_creation`, and `first_req_evicted` = the `P ‚àí read_N` estimate on the cross-turn path), and the dashboard's **Cache misses** column counts the flag. On every detected miss an informational in-session notice appears in that session's conversation (a `SystemBlock`, same surface as the overage/rate-limit notice, no action): the cross-turn path shows `‚ôª ~<evicted> tokens evicted (<read> served of <P> cached).`, the fallback path `‚ôª <creation> tokens written to cache, <read> served.`. Rows written before this feature are backfilled once by migration 0014 using the legacy session-relative token-spike heuristic (`cache_creation_tokens >= max(50000, 4√ó session median)`), so historical rows are heuristic while new rows are decisive.

### MCP interface
Mounted at `POST /mcp` (Streamable HTTP, JSON-RPC 2.0); tools exposed as `mcp__code-conductor__*`. Auto-registered on every spawn via `--mcp-config` (opt out with `ORCH_DISABLE_MCP_AUTOREGISTER=1`). No auth ‚Äî localhost-only.

**Contract conventions (clean, no legacy aliases).** The worker handle is always `sessionId` (stable across respawn / `--resume` / restart); the per-process `instanceId` is internal and **never** appears on, nor is accepted by, the conductor-facing surface (a legacy `{id}` hard-fails; the REST/browser path keeps `id`). Every worker-addressing tool also accepts an **unambiguous prefix** of the sessionId (e.g. first 8 chars), resolved to the full id at the MCP boundary ‚Äî exact full-id matches always win, an ambiguous prefix soft-refuses `{ok:false, code:'SESSION_AMBIGUOUS', matches:[‚Ä¶]}` (see [protocol.md ‚Üí MCP tool protocol](protocol.md#mcp-tool-protocol)). Worktree is always `worktree`. Worker resolution is **strict-live + soft-erroring** ‚Äî `{ok:false, code:'SESSION_NOT_LIVE'|'SESSION_UNKNOWN', reason}`, never auto-respawns (`respawn_instance` excepted). Inputs are schema-validated (unknown properties rejected); text-payload tools (`project_read`, `project_diff` diff mode, `project_bash`, `get_recent_messages`) append raw un-escaped body blocks after the `content[0]` metadata; `ok:false`+`code` for soft refusals, prose+`statusCode` for errors. Full wire contract: [protocol.md ‚Üí MCP tool protocol](protocol.md#mcp-tool-protocol).

- **Read:** `list_projects`, `list_workspaces`, `list_instances` (each entry carries `status` ‚Äî the raw process lifecycle value ‚Äî **and** `displayStatus`/`activeAgentTasks`; a worker whose turn ended while a backgrounded `Agent`-tool subagent is still working reports `status:'idle'` but `displayStatus:'running'`/`activeAgentTasks:1` ‚Äî a conductor deciding whether a worker is truly done should check `displayStatus`, not just `status`), `list_sessions` (entries carry `conducted:bool` ‚Äî true for MCP-spawned sessions; - **Shell** (read-only inspection only): `project_bash` ‚Äî run a shell command inside a project/worktree cwd, in the exact shell environment claude's own built-in Bash tool uses (the `rg`/`find`/`grep` shims and shell functions/aliases from a captured shell snapshot, generated once per (`claude` version, shell) via a throwaway `claude -p` run, cached under `<store>/shell-env/` as `bundle-<version>-<bash|zsh>.sh`, and memoized in-memory for the life of the daemon process ‚Äî at most one `claude --version` spawn per restart; see `src/claudeShellEnv.js`). Replaces the former `grep`/`glob` tools ‚Äî use `rg`/`grep`/`find` through this tool for search.
- **Create:** `create_project` (`{name, gitInit?, conventions?:[slug,‚Ä¶]}` ‚Äî optional `conventions` appends each picked convention's CLAUDE.md fragment inline (call `list_project_conventions`; 400 on unknown slug); any picked convention with `hasScaffold:true` also contributes its one-time setup directive, composed in selection order into one framed block **returned** as the result's `scaffold` field (empty/omitted when none) ‚Äî nothing persisted, the conductor folds it into the first worker's brief).
- **Workspaces:** `create_workspace` (`{name}` ‚Äî register), `delete_workspace` (`{name}` ‚Äî removes registry entry + clears member assignments), `rename_workspace` (`{oldName, newName}` ‚Äî atomic), `set_project_workspace` (`{project, workspace}` ‚Äî assign or clear with `null`/`""`; refuses `.conduct`; auto-registers a new workspace name).

the idle-subscription graph is keyed internally by the stable instanceId (the `?caller=` sessionId is translated at the register boundary); the optional `timeoutMs` watchdog fires the subscription early with a clearly-labelled "did NOT finish" stub if the worker hasn't reached `turn_end` in time, so a hung or crashed worker is never misread as finished), `set_mode`, `interrupt_turn` (optional `force:true` ‚áí hard abort; - **Session renewal:** `renew_session` (`{summary}`) ‚Äî an agent renews its OWN session (sheds context about landed jobs ‚Äî dead weight taxing every future turn ‚Äî at a lifecycle seam, not because context is "full"). Caller identity comes from `?caller=<instanceId>` (the stable instanceId, resolved to the caller's current sessionId at the MCP boundary ‚Äî so it always targets the calling session, works only for a code-conductor-managed instance, and **stays valid across repeated renewal** even though each `/clear` rotates the sessionId); it hands off a self-authored `summary` (a three-section template: live work roster, completed work index with pointers, user context) and code-conductor drives a **server-managed `/clear`** at the caller's next `turn_end`, rotating the CLI's context in place (fresh sessionId, **same OS process/pid**, old jsonl preserved) and reseeding the cleared session with the summary plus a **server-generated mechanical state block** as its first user turn. The caller's durable sessionId-keyed markers (`temp`, `conducted`, custom title) are carried onto the rotated sessionId, and the abandoned pre-clear session is **auto-archived** (its jsonl stays resumable but moves to the archived section), so a later resume/`spawn_instance({resume})` of the renewed session keeps its temp/conducted/title state. The state block (built at reseed time, not arm time, so it reflects current ‚Äî not stale ‚Äî state) enumerates the live instances the caller spawned (`InstanceManager.liveOwnedBy`, via the same `callerInstanceId` ownership tracking the sub-agent panel uses) and the caller's own pending idle subscriptions ‚Äî a safety net so a worker a degraded summary omitted is never orphaned: the mechanical list wins for **existence**, the self-authored summary wins for **intent**. Deferred to `turn_end` (not fired inline) so the tool call's own turn finishes first; returns `{ok:true, willClearAtTurnEnd:true}`. Empty summary soft-refuses `INVALID_SUMMARY`. The clear is deferred further past an overage-queued turn or live subagents so queued/background work is never stranded by the rotation. Mechanism in `src/sessionRenew.js` (`SessionRenewController`, keyed by the stable instanceId; no side-structure migration needed since the idle-subscription graph and overage timers are instanceId-keyed too). A default-on conductor module, `context-renewal` (`conduct/modules/context-renewal.md`), teaches when to call it.
- **Auto-subscribe on turn-starting calls.** `send_prompt`, `approve_plan`, `reject_plan`, and `answer_question` each take `subscribe` (default `true`) and `subscribeTimeoutMs` ‚Äî after the prompt is sent, the handler calls the same `instances.subscribeIdle(callerId, sessionId, subscribeTimeoutMs)` used by `subscribe_to_idle`, registering the caller (from `?caller=` on the MCP URL) for a one-shot wake on the target's next `turn_end`. `send_prompt({wait:true})` never subscribes ‚Äî the turn already resolves inline, and a one-shot sub registered before a blocking wait would instead fire on the *next* turn. A subscribe failure (no caller id, self-subscribe, or the caller no longer live) never fails the send: the result carries `subscribed:false` plus `subscribeSkipped:'no-caller'|'self'|<error message>` instead of `subscribed:true, already`. This makes the two-call dispatch-and-wake idiom (`send_prompt` + `subscribe_to_idle`) a single call for the common case; `subscribe_to_idle` remains for standalone re-arming (e.g. after an auto-approved plan wake with no new prompt to send).
- **Plan handling (Conduct-mode verbs):** `approve_plan` (`{sessionId, feedback?}` ‚Äî flips to bypassPermissions + sends canonical approval prompt), `reject_plan` (`{sessionId, feedback?}` ‚Äî stays in plan mode, asks for refinement), `set_auto_approve_plan` (`{sessionId, enabled}` ‚Äî flip the per-instance auto-approve flag). All three share the same approval/rejection text as the UI's Approve & Implement / Reject buttons via `src/planApproval.js`. `approve_plan`/`reject_plan` also auto-subscribe by default (see above).
- **Question handling:** `answer_question` (`{sessionId, answers}`) ‚Äî structured answer to a worker's `AskUserQuestion`, mirroring the UI question card. `answers` aligns by index to the worker's pending questions (re-derived from the ring, same source as `get_recent_messages`); each entry is `{option}` / `{options:[‚Ä¶]}` / `{text}` / `{}` with an optional `note`. Delivers text **byte-identical** to a UI submit ‚Äî both call the DOM-free formatter in `public/userQuestionAnswers.js`. - **Worktrees:** `create_worktree`, `delete_worktree` (soft-refuses `{ok:false, reason, code:'WORKTREE_ATTACHED'|'WORKTREE_DIRTY'}` unless `force:true`), `sync_worktree`, `merge_worktree` (takes `{sessionId}` or `{project, worktree}` ‚Äî the latter works after the instance is gone; fast-forwards the worktree's own branch onto the new merge commit; refusals carry a `code`: `WORKTREE_BEHIND`/`BASE_BRANCH_MISMATCH`/`PARENT_DIRTY`/`WORKTREE_DIRTY`/`NOTHING_TO_MERGE`/`MERGE_FAILED` ‚Äî `WORKTREE_DIRTY` here is the worktree's own tree, not the parent's, and is overridable via `allowDirty:true`).

### Plugins
- **What a plugin is** ‚Äî a sibling project under `<projectsRoot>/` with a `conductor.plugin.json` at its root (schema: [protocol.md ‚Üí Plugin system](protocol.md#plugin-system)). It contributes up to four capabilities: a supervised backend process, an embedded frontend, MCP tools forwarded through the conductor's MCP server, and **project conventions** (entries joining the Conventions catalog as `<plugin-id>/<slug>`, each carrying a CLAUDE.md fragment and/or a one-time scaffold directive). The last needs **no backend** ‚Äî a contributions-only plugin (`{id,name,version,pluginApi}` + `conventions`) validates, enables, and never starts a process. Discovery scans the projects root at boot and on **‚Üª Rescan**; duplicate ids resolve first-alphabetical-project-wins (the rest show `conflict`).
- **Conventions & scaffolds** ‚Äî an enabled plugin's `conventions` join the new-project dialog + `list_project_conventions` (namespaced `<plugin-id>/<slug>`). A convention carries a CLAUDE.md **fragment** (`file`, applied inline at creation ‚Äî the copy survives disable/uninstall) and/or a **scaffold** facet (`scaffold:{text|file}`, a one-time setup directive, **plugin-only**); at least one of the two is required. A convention flagged `hasScaffold` carries its scaffold invisibly in the dialog (no tag); when picked at creation its directive is composed (in selection order across all picked scaffold-bearing conventions) into one framed block **returned** by `create_project`/`POST /api/projects` under a `scaffold` field ‚Äî nothing persisted, no spawn coupling. The conductor folds it into the first worker's brief; the UI displays it read-only in the create confirmation (v1). A **scaffold-only** convention appends nothing to CLAUDE.md but still emits its directive. Only enabled+`ok` plugins contribute; a stale/missing fragment or scaffold `file` makes the plugin `invalid` at load. Since fragment and scaffold now live on one entry, author a scaffold-bearing convention to **degrade gracefully** if the scaffold step wasn't run (e.g. "if a project-local harness wrapper exists, use it; otherwise see the shared harness to create one").
- **Convention scope** ‚Äî every plugin convention declares a required `scope`. **`project`** and **`conductor`** are accepted today; `workspace` is **planned** and rejected at manifest load with a "not yet supported" error. A **`conductor`-scope** convention joins the Conductor-conventions catalog above (Settings ‚Üí Conventions ‚Üí Conductor) as its own checkbox, namespaced `<plugin-id>/<slug>`; like a built-in/custom module it defaults to **off** and only takes effect (composed into `.conduct/CONDUCT.md`) once explicitly enabled there. Unlike a project convention it is not snapshotted, so disabling the plugin (or unchecking its convention) drops it from the next `.conduct/CONDUCT.md` regeneration. The scaffold facet has no scope of its own ‚Äî it's inherently project-scoped (it runs at project creation); a scaffold facet on a `conductor`-scope entry is simply never read (nothing outside project creation consumes it). ‚öÝÔ∏è A **`conductor`-scope** convention edits the orchestrator's **own** rules (the `.conduct/CONDUCT.md` modules that govern how it delegates/reviews/merges across all projects) ‚Äî a powerful, trust-loaded capability (consistent with "plugins are trusted code", but far beyond a per-project convention); it requires the same explicit user opt-in as any other conductor module, precisely because of that trust jump.
- **Trust stance (loud)** ‚Äî plugins are **trusted own code**. Their frontends are same-origin iframes: plugin JS can call every conductor API the UI can. Enabling a plugin is equivalent to running its code with your conductor's privileges.
- **App switcher** ‚Äî the sidebar `CodeConductor` title becomes a dropdown once an enabled plugin contributes a frontend (zero plugins ‚áí plain `<h1>`, no visual change). Selecting a plugin navigates to `#plugin/<id>/`; selecting Conductor deterministically closes the plugin view and restores the active session anchor (never `history.back()` ‚Äî a deep link has no usable back entry). Rendered by `public/appSwitcher.js` from `GET /api/plugins`.
- **Plugin view** ‚Äî `#plugin/<id>/<subpath>` shows the plugin's frontend in a full-page iframe (`public/pluginView.js`, src `/plugins/<id>/<subpath>` through the reverse proxy). Switching to an enabled-but-stopped plugin **auto-starts it behind a "Starting ‚Ä¶" overlay** (REST status‚Üístart before the iframe loads, ‚â§30 s readiness); a start failure shows the error + crash tail with a **Retry** button instead of raw proxy JSON in the frame. Entering a plugin costs exactly **one** history entry (the bridge patches `pushState`‚Üí`replaceState` inside the iframe), so hardware **Back always lands on the conductor**; Escape works too. The sidebar stays live (`/ws` events keep flowing) while a plugin is open, and **any sidebar navigation** (selecting a session, opening commits/review/settings) closes the plugin view and returns to the conductor. Leaving the hash space blanks the iframe (frees memory). Limitation: multi-page (non-SPA) plugins still pollute joint history.
- **Settings ‚Üí Plugins** ‚Äî management group (`public/pluginManager.js`): per-plugin state badge (`discovered / disabled / enabled / stopped / starting / ready / crashed / failed / invalid / conflict / incompatible`), contribution badges (`N project conventions`, `Project scaffold: N` with a per-scaffold name/description preview), immediate Enable/Disable/Start/Stop buttons, crash-output expander, ‚Üª Rescan, and an **active version** dropdown (main + the project's worktrees). A ready row shows `running <version> @ <sha7>`, ` ¬∑ update available` when stale, + port. A **backendless (contributions-only)** enabled plugin shows state `enabled` and only a Disable button (no Start/Stop/version ‚Äî it has no process).
- **Restart on stale code** ‚Äî a running plugin whose active checkout's HEAD has moved past the sha it was started at (a commit landed while it was running) is flagged `stale` and gets a **Restart** button next to Stop (stop + start in place, `POST /api/plugins/:id/restart`, 409 if not running). No button while stopped, backendless, or already at the started-at sha; a non-git checkout is never stale.
- **Plugin Library** ‚Äî below the installed-plugins list, a catalog of *installable* plugins (git repo URLs) with an **Install** button that clones the repo into the projects root (`GET`/`POST /api/plugins/library[...]`, `src/plugins/library.js`). Install only clones ‚Äî it never enables or starts anything; once cloned, the project surfaces in the list above through the normal discovery/rescan flow, same as any other plugin project. The catalog is four built-in entries ‚Äî `code-share` (`https://github.com/UnmanagedCode/code-share`), `code-playwright` (`https://github.com/UnmanagedCode/code-playwright`, Playwright + Chromium glue for visual UI debugging), `code-hub` (`https://github.com/UnmanagedCode/code-hub`, launch/monitor/stop/share other workspace webapps), and `code-karpathy-wiki` (`https://github.com/UnmanagedCode/code-karpathy-wiki`, durable-knowledge wiki conventions) ‚Äî overlaid by drop-in manifests dropped into `<orchStoreRoot()>/plugins/library/*.json` (one JSON object per file: `{id, name, description?, repo, postClone?, postPull?}`; a dropped file whose `id` matches a built-in overrides it; malformed/incomplete files are skipped with a warning, never fatal). An entry already cloned (its derived target directory exists) shows "installed as `<name>`"; an **Update** button only appears when the checkout is actually **behind its origin** ‚Äî `list()` runs a bounded (8s), best-effort `git fetch` per installed entry before comparing (reusing `getProjectUpstreamStatus`, `src/worktrees.js`), so a fetch timeout/no-remote/detached-HEAD/non-repo just hides Update rather than crashing the list; an up-to-date entry shows "installed as `<name>` ¬∑ up to date" instead. The target project name is derived from the repo URL's last path segment (`.git` suffix stripped); only `http:`/`https:`/`git:` repo URL schemes are accepted. A failed/timed-out clone shows the error + clone-output tail (reusing the crash-tail `<details>` styling) and cleans up any partial directory so the same install can be retried.
- **Plugin Library ‚Äî live install/update output** ‚Äî Install/Update stream their progress as NDJSON (`src/plugins/api.js`'s `streamLibraryAction`), so the clicked row shows its button disabled + relabeled (`Installing‚Ä¶`/`Updating‚Ä¶`) and a small live output box (`.pll-live`) filling in with clone/pull/hook stdout+stderr as it runs, instead of only a result appearing after the fact. On failure the row's live output stays visible and the button re-enables; on success the whole library list re-renders (dropping the live box, re-evaluating Update availability).
- **Plugin Library ‚Äî postClone/postPull + Update** ‚Äî an entry may declare `postClone`/`postPull` shell commands (run via `bash -lc`, cwd = the project directory, e.g. `code-playwright`'s `bash install.sh`) that install the plugin's own dependencies. `postClone` runs once, right after a successful install (clone + rescan); **Update** (when shown ‚Äî see above) runs `git pull --ff-only` then `postPull`. A failed `git clone`/`git pull` is a hard failure (error + tail shown, and a failed clone is rolled back); a failed `postClone`/`postPull`, by contrast, is a **soft warning** ‚Äî the clone/pull already succeeded and the project is already discoverable, so the entry still shows as installed, just with a warning + output tail. The documented way to retry a failed `postClone` is hitting **Update** (which reruns `postPull` ‚Äî `code-playwright` sets both to the identical command). Running a library entry's declared command is a trusted-code-execution surface (bounded timeout, output always surfaced), consistent with the plugin system's overall "plugins are trusted own code" stance.
- **Lifecycle & lazy start** ‚Äî enabling records the plugin. Any discovered plugin project (enabled or not, valid or not) with no workspace is auto-assigned to the **CC-Dev** workspace on every rescan/boot ‚Äî enable, install, and manual rescan all trigger this via the same discovery pass, and a project already assigned elsewhere is never moved. The conductor's own project gets the same CC-Dev placement via a one-time boot seed, since it isn't a plugin. Enabled-but-stopped costs zero RAM: the first proxy request or MCP tool call starts the child and waits through the ‚â§30 s readiness window. Children are detached process groups that **survive conductor restarts** (adopted on boot when still alive+answering). Health is passive (exit events, proxy errors, on-demand probes) ‚Äî on Android, Doze may kill a child at any time; the next request 503s with the crash tail, then lazy-restarts after a short backoff. **3 crashes in 60 s ‚áí `failed`** (manual Start/Enable to recover).
- **Worktree-version activation** ‚Äî the version dropdown switches which checkout the child runs from (`main` or any of the project's worktrees, resolved via the worktree store). The manifest is re-read from the active checkout on every start and must keep the same id (else 400, previous state kept). Caveat: worktrees don't share `node_modules` ‚Äî a missing-deps start failure surfaces in the crash tail; the per-project `post-worktree-create.sh` hook can `npm install` automatically.
- **Plugin MCP tools** ‚Äî manifest-declared tools appear inside the existing conductor MCP server as `<plugin-id>__<tool>` (client-visible `mcp__code-conductor__<plugin-id>__<tool>`). Visibility: **globally visible** ‚Äî every enabled plugin's tools are offered to the conductor UI and to workers in **every** project (the manifest `scope` field is accepted for compatibility but inert). A disabled plugin's tools are simply unknown. Calls lazy-start the plugin and forward over HTTP with a per-manifest timeout. Note: workers snapshot the tool list at spawn ‚Äî mid-session enable/disable reaches only new sessions.
