## Features

### Projects & workspaces
- **Project list** ŌĆö sidebar shows every dir under `<projectsRoot>/`. `+ New project` creates a dir with `CLAUDE.md` containing `@../CLAUDE.md`, optionally followed by inline snapshots of selected rule modules (see **Rule modules** below). Worktree-owned dirs are hidden from the list (filtered against a Set built from the central store).
- **Delete project** ŌĆö `├Ś` per row, typed-name confirm; cascades through instances + worktrees then `rm -rf`s the project. `~/.claude/projects/<encoded>/` jsonls are left intact (still resumable by the standalone CLI).
- **Workspaces** ŌĆö `ŌēĪ` hamburger menu hosts `+ New workspace`; nest projects under collapsible `<details>` headers, one workspace per project, empty workspaces persist. Edit (rename / re-pick members / delete) via the Ō£Ä button on each header. Workspaces default-collapsed; expand state persists in `localStorage` (`code-conductor:workspaces-expanded`). Stored server-side at `<store>/projects/<name>/project.json`; the workspace set is the union of `<store>/workspaces.json` and any referenced names. Workspace names use the same `^[a-zA-Z0-9._-]+$` regex as project names.
- **Git status pill** ŌĆö
  - **Project rows**: amber `ŌåæN ŌåōM` against the currently-checked-out branch's *configured upstream* (e.g. `origin/<branch>`), using cached refs only (no `git fetch`). No pill when in sync / detached / upstream-less.
  - **Worktree rows**: same shape but compares the worktree branch to its captured *parent base branch* (may be a local branch, not a remote). `ŌåæN` = ahead and FF-able, `ŌåōM` = behind and FF-able, `ŌåæN ŌåōM` = diverged ŌåÆ rebase needed before merge.

### Worktrees
- **Isolated environments** ŌĆö "Run in isolated git worktree" checkbox in the new-instance dialog runs `git worktree add ../<project>_worktree_<short-id> -b code-conductor/<short-id> <currentSha>` and spawns the agent with `cwd` set to the worktree. The orchestrator captures `{baseBranch, baseSha, branch}` at creation so you can spawn off any branch and land back later. Worktrees survive instance death ŌĆö re-spawn into the same worktree from the sidebar.
- **Two-step rebase-back**:
  - **Sync** ŌĆö already in sync ŌåÆ no-op; purely behind & clean ŌåÆ server-side `git merge --ff-only <baseBranch>`; diverged & clean ŌåÆ orchestrator attempts `git rebase <baseBranch>` directly (returns `rebased` on success); if rebase hits conflicts, it aborts cleanly and falls back to sending the agent a templated rebase prompt (`rebase-prompt-sent`; agent runs `git rebase` itself, asks the user via `AskUserQuestion` on non-trivial conflicts, replies `REBASE_DONE`); behind-but-dirty ŌåÆ rebase prompt sent directly (agent commits/discards first).
  - **Merge** ŌĆö `git merge --no-ff --no-edit <worktreeBranch>` on the parent. Always produces a merge commit (default msg `Merge branch 'code-conductor/<id>'`) so each contribution stays visible in `git log --graph` and is revertible as one commit. Refuses inline (not as a 500) if the worktree is behind the parent, the parent is on a different branch than `baseBranch`, or the parent tree is dirty.
- **Delete** ŌĆö refused if there's a live instance or uncommitted work, with a `force=1` override.
- **Post-worktree-create hook** ŌĆö if the project repo contains `.code-conductor/post-worktree-create.sh`, the orchestrator runs it automatically inside the new worktree after the worktree directory, branch, and metadata are written. Use case: symlink `node_modules`, copy `.env`, run `npm ci`, etc. The script receives env vars `CC_WORKTREE_PATH`, `CC_PROJECT_NAME`, `CC_BRANCH`, `CC_BASE_BRANCH`, `CC_PARENT_PATH`. Non-fatal: a failure warns but does not roll back the worktree. Output is capped to the tail ~16 KB (`truncated:true` when exceeded). Timeout: 120 s (`ORCH_POST_WORKTREE_TIMEOUT_MS`). Kill-switch: `ORCH_DISABLE_POST_WORKTREE_HOOK=1`. Result is reported in the spawn/worktree-create response as `postWorktreeCreate: { ran, exitCode, durationMs, output, truncated?, timedOut? }`. The spawn dialog re-opens briefly with a one-line summary (expandable output) when the hook ran.

### Diff / review & commit history
- **Worktree diff/review** ŌĆö `┬▒` button on every worktree row opens a full-page, mobile-friendly diff browser (`#review-view`, `public/review.js`). Fetches structured per-file diff JSON from `GET /api/projects/:name/worktrees/:wt/diff` (`<baseBranch>...HEAD`) and renders collapsible `<details>` per file (status badge, `┬▒` counts, hunks reusing `.diff-line.add/.del/.ctx`, `overflow-x:auto` for horizontal scroll). Diff capped at ~200 KB ŌåÆ `(truncated)` in the header.
- **Commit history** ŌĆö `ŌēĪ` button on every git-repo project row opens a full-page scrollable commit list (`#commits-view`, `public/commits.js`) of the project's **current branch (HEAD)**: short sha, subject, author, relative date, newest first. Backed by `GET /api/projects/:name/commits` (default 100, max 500; header shows `(showing latest N)` when truncated). Tapping a commit opens the **same diff renderer** showing just that commit's change (`GET /api/projects/:name/commits/:sha/diff`, via `git show`); the diff layers over the list and `ŌåÉ Back` returns to it. Merge commits render as "No changes" (combined diff not parsed). A **branch/merge graph rail** (git-log --graph style) is drawn to the **left** of each row: colored lanes, a dot per commit, diagonals where branches fork and merges converge. Topology is computed client-side in `computeGraph()` (`public/commits.js`) from each commit's `parents` (SHAs from `%P`); lanes use stable column indices so pass-through lines stay aligned. Each rail is a fixed-height SVG "node band" (dot + diagonals) plus full-height CSS filler lines, so it aligns with variable-height (wrapping) rows without measurement. Parents older than the cap / on unfetched branches simply trail off the bottom. Soft cap of 12 lanes (extras clamp to the last column, logged to console).

### Sessions & instances
- **Unified view** ŌĆö Sessions list per project (and per worktree) shows live + historical. Header: `Sessions (N) ┬Ę K live ┬Ę last <ago>`. Click to focus live instance, or resume stopped one (`POST /api/instances` with `--resume <sid>`). Synthetic `(new session)` row for spawned-but-no-jsonl-yet instances. Rows bucket into up to three sections: the normal list, then a dim `ŌĆö temp ŌĆö` separator, then a dim `ŌĆö conducted ŌĆö` separator (each section only renders when non-empty).
- **Conducted sessions** ŌĆö a session spawned via the MCP `spawn_instance` tool (orchestrator-driven) rather than the browser/HTTP path is marked **conducted** (a worker an orchestrator conducts; the orchestrator itself is the `.conduct` session). Orthogonal to temp: unlike temp (in-memory, wiped on exit) the conducted marker is **durable** ŌĆö persisted to `<store>/conducted-sessions.json` (a `{sessions:[sid,ŌĆ”]}` set, written on turn-end via the same path as last-prompt/permission-mode metadata) so it survives exit, restart, and `--resume` (a UI-resumed conducted session re-acquires the marker from the sidecar). Set **only** in the MCP `spawn_instance` handler ŌĆö plain UI spawns stay non-conducted. Grouping precedence: a conducted session renders under `ŌĆö conducted ŌĆö`; a conducted **and** temp session renders under `ŌĆö conducted ŌĆö` but keeps the warm temp preview colour (`.session-row.temp`); a temp-but-not-conducted session stays under `ŌĆö temp ŌĆö`. Pure conducted rows get a cool preview tint (`.session-row.conducted:not(.temp)`). It's a grouping, never a filter ŌĆö conducted sessions stay fully visible in `list_sessions`, the HTTP session APIs, and the sidebar. The marker is cleaned up alongside the session-title sidecar on session delete / temp-exit / boot sweep.
- **Unread indicator** ŌĆö small accent pill next to a row when its instance finishes a turn while you're viewing a different session (per-instance, persisted in `localStorage` key `code-conductor:unread`).
- **Archive session (the `├Ś` action)** ŌĆö the hover-revealed `├Ś` on a session row **archives** it (never deletes): single confirm, `POST ŌĆ”/sessions/:sid/archive`; on 409 the client auto-retries with `?force=1` (stops the live instance) without a second prompt. Archived sessions keep their jsonl (still resumable) + title + conducted markers and **disappear from the sidebar entirely** ŌĆö they're managed only from **ŌÜÖ Settings ŌåÆ Archived**. There is no longer an inline `ŌĆö archived ŌĆö` sidebar section, and sessions are never deleted from the sidebar.
- **Archived page (Settings ŌåÆ Archived)** ŌĆö lists every archived session grouped by owning project (`GET /api/archived`). Each project group is a `<details>` **collapsed by default**; expanding reveals its archived sessions (title/first-prompt/sid + worktree tag + last-active). Per session: **Restore** (`POST ŌĆ”/restore` ŌåÆ reappears in the normal sidebar) and **Delete** (confirm ŌåÆ `DELETE ŌĆ”/sessions/:sid`, the single permanent jsonl-removal path; cannot be undone). The `.conduct` (orchestrator) project's group is always pinned first and rendered as `­¤Ä╝ Conductor` instead of its raw folder name ŌĆö sorting/labeling happens client-side in `renderArchived()`.
- **Session summary** ŌĆö `Ōŗ«` menu's `­¤ōØ Summarize` opens a modal dialog that generates and displays an AI-written summary of the session via a one-shot `claude -p` call (Haiku). Three tiers: **Short** (~40 content words, plain prose gist), **Medium** (~150 words, scannable recap with bullets/tables), **Long** (~400 words, thorough with `##` headings and structured tables). Summaries are rendered as full markdown (headings, tables, code) via the same `renderMarkdownInto` pipeline as assistant messages. A **stale indicator** (`ŌÜĀ Session has grown since this summary`) appears when the session has received more turns since the last generation; regeneration is always user-initiated. Stored in `<store>/session-summaries.json` keyed by `sessionId` ŌĆö `{ summary, length, generatedAt, messageCount }`; auto-removed when the session jsonl is deleted. `sessionId` is stable across `--resume`, so summaries simply persist across a resume.
- **Rename session** ŌĆö `Ōŗ«` menu's `Ō£Å’ĖÅ Rename` opens a prompt to set a custom human-readable label (Ōēż100 chars, empty clears) that overrides the first-prompt preview in the sidebar and prepends an italic chip to the active header. Stored in `<store>/session-titles.json` keyed by `sessionId`; auto-cleaned when the session jsonl is deleted or a temp session exits. Survives rewind (same `sessionId`) and `--resume` (`sessionId` is stable across resume); fork starts un-titled.
- **Change model** ŌĆö `Ōŗ«` menu's `­¤¦Ā Change model` (directly below Rename) opens a family picker using the same catalog/source of truth as the spawn dialog (`Settings ŌåÆ Models`). Selecting a family switches the **live** session's model via a `control_request` (`subtype:set_model`) ŌĆö no restart, no history loss. The picker highlights the session's current family with `.qs-selected`; the button's own label is static.
- **Rewind & fork** ŌĆö every user message bubble has `ŌåČ` (rewind in place, same `sessionId`) and `Ōæé` (fork to a new `sessionId`) on hover. Both atomically rewrite the jsonl, preserve the dropped prompt in the composer, and 409 during a running turn. Bubbles anchor on the **server-stamped `userIndex`** riding on each outer `user_echo` (the absolute Nth-pure-user-prompt ordinal) ŌĆö never a client-side bubble count, which would drift once the capped ring / tail-only snapshot starts the view mid-history. An echo without a stamp (rare orphan) renders without the buttons.
  - Rewinding **to the first user message** is special-cased: prefix is empty, so the orchestrator deletes the jsonl entirely and respawns with `--session-id <sid>` (not `--resume`, which the CLI rejects on zero-line files).
  - **Temp sessions cannot be forked** ŌĆö the on-exit cleanup races the fork's source copy.
  - UI-only ŌĆö no MCP equivalent yet.
- **Crash recovery** ŌĆö subprocess death ŌåÆ `crashed`; Resume respawns with `--resume <sid>`, preserving the in-memory event ring + conversation.
- **Session anchor** ŌĆö `#session=<sid>` URL hash; on reload, auto-resumes by locating the jsonl on disk (worktree-aware) via `GET /api/sessions/:sid/locate`. `--resume` spawn costs zero API tokens until the next prompt. Stale anchors are silently cleared.

### Spawn options
- **Mode** ŌĆö `plan` (read-only, propose then `ExitPlanMode`), `ask` (every destructive tool gates through Allow/Deny card), `code` (full power, CLI's `bypassPermissions`). CLI's `default` / `acceptEdits` are not exposed: in `stream-json --print` (no SDK `canUseTool` callback) they auto-deny tool calls and the only recovery is forcing the model to regenerate the entire tool input.
- **Effort** ŌĆö `low` / `medium` / `high` (default) / `xhigh` / `max`.
- **Thinking** ŌĆö `adaptive` (default) / `enabled` / `disabled`.
- **Model** ŌĆö empty (account default) or pick a family (Haiku / Sonnet / Opus / Fable 5; Fable 5 hidden if disabled in Settings ŌåÆ Models). Context windows: **Haiku always 200k** (no 1M build), **Opus / Fable 5 always 1M** (bare CLI default), **Sonnet 4.x uses the stored preference** (configurable in **ŌÜÖ Settings ŌåÆ Models**; defaults to 1M), **Sonnet 5 always 1M** (no 200k build). The concrete version each family resolves to is also configurable in Settings ŌåÆ Models (defaults `claude-fable-5` / `claude-sonnet-4-6` / `claude-opus-4-8` / `claude-haiku-4-5`). Context window and version are both re-derived on resume (`canonicalizeModel` in `src/modelVersions.js`) ŌĆö neither is persisted in the session jsonl.
- **Temp session** ŌĆö deletes the jsonl + sibling `subagents/` dir on exit, drops itself from the sidebar's Sessions list on exit/crash (no ghost row to clean up by hand), and skips `last-prompt`/`permission-mode` metadata appends during the run; mode defaults to `code`; header shows a `TEMP` pill.
- **Debug mode** ŌĆö mirrors raw CLI traffic (`claude-stdin.jsonl`, `claude-stdout.jsonl`, `claude-stderr.log`, `meta.json`) to `<store>/projects/<p>/[worktrees/<wt>/]debug/<instance-id>/`. Append-mode, survives session end. Header shows a `DEBUG` pill.

### Live conversation
- **Tail-first load + scroll-up history** ŌĆö subscribing renders only the ring tail (Ōēż500 events, quiescent-aligned: whole blocks, resolved tools). When older history exists, a tappable `Ōŗ» earlier messages` sentinel sits above the oldest content and scrolling near the top auto-fetches the previous page (`GET /api/instances/:id/events?before=<cursor>&limit=200`, flipping the sentinel to `loading earlierŌĆ”`). Pages render through the standard block pipeline on a detached node and splice in above with the viewport preserved; a turn split across page seams **merges back into one assistant bubble** (the batch's trailing open bubble concatenates into the chunk below), and sub-agent (Task) groups always travel whole with their nested panel ŌĆö live children of a not-yet-loaded Task head are parked and adopted into the panel once its page loads. Events evicted from the in-memory ring are reconstructed from the session jsonl. Caveats: the task panel and usage chip rebuild from the tail only, and a single turn longer than the ring cap pages back with a gap (never duplicated content) ŌĆö rendered as a `ŌöĆŌöĆ Ōŗ» earlier messages unavailable ŌöĆŌöĆ` divider at the exact seam; only whole blocks are ever missing, and a genuinely interrupted tool renders `┬Ę incomplete` rather than a stuck spinner.
- **Text** ŌĆö streams as text-node deltas; on `text_end` re-renders through `public/markdown.js` (headings, lists, code, **bold**, *italic*, links, autolinked bare `http(s)://ŌĆ”` URLs, `![alt](src)` images). Link schemes restricted to `http(s)/relative/fragment/mailto`; image `src` to `http(s)/file://`/absolute path; no `innerHTML`; raw HTML in source rendered as literal text.
- **Text-to-speech (read aloud)** ŌĆö the inverse of voice dictation: finalized assistant text blocks get a trailing `­¤öŖ` button (tap to hear it), and a global **auto-speak** toggle reads each reply as it lands. Local server-side Piper ŌĆö no network, no API key. Synthesis is **streamed per sentence**: `POST /api/tts` returns a sequence of `[4-byte LE length][WAV]` frames (one per sentence) flushed as Piper yields them; the client (`public/tts.js`) decodes each with the Web Audio API and schedules them back-to-back, so the first sentence starts playing (~0.3s) while the rest synthesize. The ­¤öŖ button appears once Piper is installed (`GET /api/tts/status` gates it; Settings ŌåÆ TTS installs/flips it at runtime). Auto-speak only fires after the user has interacted at least once (browsers block audio without a prior gesture); a ­¤öŖ tap always works. Sub-agent text isn't auto-spoken.
- **Thinking** ŌĆö collapsible block; redacted thinking renders as a non-expandable `thinking (redacted)` line (no disclosure caret). Opus 4.7 emits only a `signature_delta`; Opus 4.8 streams empty `thinking_delta`s (`""`) ŌĆö the parser drops the empties so both collapse to the same redacted line.
- **Tool use** ŌĆö collapsed by default with smart one-line summary (`­¤ö¦ Bash ┬Ę ls -la ┬Ę done`). Edit/Write/NotebookEdit render as syntax-coloured unified diffs (green/red gutters, ┬▒counts, sticky file-path); Write shows a numbered preview. Other tools: raw-JSON input in its own collapsed `Ōå¬ tool_input` block.
- **Skill loading** ŌĆö a `Skill` tool call's SKILL.md content, which the CLI injects back as a plain (large) user message, renders as a dedicated collapsed `­¤ōś Loading skill: <name>` bubble instead of a giant plain-text bubble; tap to expand the full content. Server-side correlation with the preceding `Skill` tool_use (not just the CLI's `isSynthetic` flag, which it also reuses for unrelated compaction/hook text) ŌĆö see `docs/protocol.md`.
- **Tool result** ŌĆö truncated at 4 KB with "show full". `image` content blocks (e.g. `Read` on a `.png`/`.jpg`) render as inline `<img>` thumbnails alongside the text ŌĆö base64 sources become `data:` URLs, `url` sources only pass through for `http(s)`/`file://`, `image/svg+xml` is refused (XSS). Auto-expanded when an image is present. Tap any conversation image to open it full-size in an in-page lightbox (`public/lightbox.js`) ŌĆö works for `data:`/`file://` sources that Chrome on Android refuses to open as a top-level navigation. Opens fit-to-screen; tap the image to toggle 1:1 full native resolution (backdrop scrolls to pan), tap again to fit; tap backdrop or press Esc to close.
- **Sub-agent drill-down** ŌĆö `Task` tool calls nest a mini-conversation (dashed border, `Ōå│ sub-agent` label) inside the parent tool block, routed by `parentToolUseId`.
- **Plan mode card** ŌĆö green-bordered card "Plan ready for approval". Body comes from `input.plan`, or ŌĆö when the model wrote the plan to `~/.claude/plans/*.md` first and called `ExitPlanMode` with empty input ŌĆö the most-recent such file. **Approve** sends `"I approve the plan. Please proceed with the implementation."` (+ feedback if provided) and flips the instance to `code` (`bypassPermissions`). **Reject** keeps plan mode active and sends `"I'd like to revise the plan. Refinement notes:\n<feedback>"`. **­¤ōŗ Auto-approve plans** toggle sits inline immediately left of the mode dropdown and is only visible while the instance is in `plan` mode (one-click, mid-turn friendly); per-instance, server-side state (mirrored down through `snapshot`/`status` WS frames so every tab agrees), cleared on server restart. When on, the orchestrator fires the same setMode+approval-prompt the moment a `plan_request` lands ŌĆö works regardless of whether any tab is subscribed / focused / minimized. Green-tinted background when on, outlined when off.
- **AskUserQuestion card** ŌĆö blue card; multi-question tab strip across the top. - **System notes** ŌĆö kept: `init`, `stderr`, `exit`, `permission_denied`, `compacting`, `spawn_error`, `crashed`, `history_load_error`, non-allowed `rate_limit_event`. - **Turn footer** ŌĆö duration, cost, tokens.

- **Voice dictation** ŌĆö no separate mic button; dictation is folded into the **Send** button (WhatsApp-style). Two modes, both reuse the same `MediaRecorder` ŌåÆ `POST /api/transcribe` pipeline:
  - **Empty composer** (whisper installed): Send button becomes a mic (neutral styling). Tap to start recording (red pulse); tap again to stop and transcribe. Transcribed text is inserted at the caret.
  - **Non-empty composer** (whisper installed): **Hold the Send button ~500 ms** to enter recording mode (red pulse). Release to stop and transcribe; the result is **appended** to existing text with a space separator. A short tap still sends normally. The button tooltip reads "Send message (hold to dictate)" as a discoverability cue. Holding then cancelling (e.g. finger leaves mid-hold, or `pointercancel`) aborts cleanly ŌĆö nothing is appended, nothing is sent.

  While the transcription POST is in flight the button shows an amber spinning icon, then the resulting text is inserted/appended (space-joined so back-to-back words don't run together). Dictated text stays plain in the composer; when you **send** a message that included any dictation, it's prepended with a `<transcribed>` tag so the receiving instance knows it contains speech-to-text that may have transcription errors. When whisper isn't installed, an empty composer shows a disabled Send and holding the Send button is a no-op. Recording uses Pointer Events (`pointerdown`/`up`/`cancel`) with explicit pointer capture so releases register reliably on touch. Runs whisper.cpp **locally on the server** ŌĆö no network, no API key, no telemetry. The mic affordance only appears once whisper is installed (`GET /api/transcribe/status` gates it; the Settings page can flip it at runtime). Recording uses the browser's `MediaRecorder` (webm/opus); the server converts via ffmpeg ŌåÆ 16 kHz mono WAV before calling `whisper-cli`.

### MCP interface
Mounted at `POST /mcp` (Streamable HTTP, JSON-RPC 2.0); tools exposed as `mcp__code-conductor__*`. Auto-registered on every spawn via `--mcp-config` (opt out with `ORCH_DISABLE_MCP_AUTOREGISTER=1`). No auth ŌĆö localhost-only.

**Contract conventions (clean, no legacy aliases).** The worker handle is always `sessionId` (stable across respawn / `--resume` / restart); the per-process `instanceId` is internal and **never** appears on, nor is accepted by, the conductor-facing surface (a legacy `{id}` hard-fails; the REST/browser path keeps `id`). Every worker-addressing tool also accepts an **unambiguous prefix** of the sessionId (e.g. first 8 chars), resolved to the full id at the MCP boundary ŌĆö exact full-id matches always win, an ambiguous prefix soft-refuses `{ok:false, code:'SESSION_AMBIGUOUS', matches:[ŌĆ”]}` (see [protocol.md ŌåÆ MCP tool protocol](protocol.md#mcp-tool-protocol)). Worktree is always `worktree`. Worker resolution is **strict-live + soft-erroring** ŌĆö `{ok:false, code:'SESSION_NOT_LIVE'|'SESSION_UNKNOWN', reason}`, never auto-respawns (`respawn_instance` excepted). Inputs are schema-validated (unknown properties rejected); text-payload tools (`read_file`, `get_worktree_diff` diff mode, `get_recent_messages`) append raw un-escaped body blocks after the `content[0]` metadata; `ok:false`+`code` for soft refusals, prose+`statusCode` for errors. MCP-path defaults: `temp:true` but mode stays `plan` ŌĆö `InstanceManager.create()` is policy-light and never couples temp to mode, so conducted workers plan before acting; explicit `temp:false`/`mode` still win. The UI's tempŌćÆ`bypassPermissions` shortcut lives only at the `POST /api/instances` route, not in `create()`), `promote_session` (`{sessionId}` ŌĆö promote a temp session to persistent via `Instance.promoteToNormal()`; soft-refuses if not live/unknown, errors if not temp), `send_prompt` (`{sessionId, ŌĆ”}`, optional `wait:true`, 10 min cap; **auto-subscribes** to the worker's idle callback by default ŌĆö see below), `wait_for_idle`, `subscribe_to_idle` / `unsubscribe_from_idle` (`{sessionId, timeoutMs?}`; one-shot async callback ŌĆö when target hits `turn_end` the orchestrator injects a stub user prompt into the *caller* via `Instance.prompt()`, naming the worker by sessionId; **folded wake** ŌĆö on a real `turn_end` delivered to an already-idle caller the stub **folds the worker's recent output inline** (the SAME content a default `get_recent_messages({sessionId})` call returns ŌĆö bonded prose + plan/questions ŌĆö via the shared `buildRecentMessages` builder + `flattenPayload`), so the conductor needs no follow-up call; the folded stub is tagged with a `[[cc:wake-callback]]` marker (`public/wakeCallback.js`) that the conductor UI renders as a **collapsible wake-callback bubble** (worker id + "finished its turn" summary always visible, folded payload collapsed by default) with an orchestrator badge (­¤öö) marking it as system-injected; the idle-subscription graph is keyed by `sessionId`; caller identity is read from `?caller=<caller sessionId>` baked into the per-worker MCP URL at spawn time (after the sessionId is minted); **carve-outs** ŌĆö both the `timeoutMs` watchdog stub and the mid-turn-deferred stub keep the plain "call `get_recent_messages` to inspect" pointer text (no fold), but they're also tagged with the marker (via `markPlainStub`, body-less) so they render as the **same wake bubble** ŌĆö badge + summary line only, no expandable body or caret; the optional `timeoutMs` watchdog fires the subscription early with a clearly-labelled "did NOT finish" stub if the worker hasn't reached `turn_end` in time, so a hung or crashed worker is never misread as finished), `set_mode`, `interrupt_turn` (optional `force:true` ŌćÆ hard abort; default soft graceful stop), `kill_instance`, `respawn_instance`.
- **Auto-subscribe on turn-starting calls.** `send_prompt`, `approve_plan`, `reject_plan`, and `answer_question` each take `subscribe` (default `true`) and `subscribeTimeoutMs` ŌĆö after the prompt is sent, the handler calls the same `instances.subscribeIdle(callerId, sessionId, subscribeTimeoutMs)` used by `subscribe_to_idle`, registering the caller (from `?caller=` on the MCP URL) for a one-shot wake on the target's next `turn_end`. `send_prompt({wait:true})` never subscribes ŌĆö the turn already resolves inline, and a one-shot sub registered before a blocking wait would instead fire on the *next* turn. A subscribe failure (no caller id, self-subscribe, or the caller no longer live) never fails the send: the result carries `subscribed:false` plus `subscribeSkipped:'no-caller'|'self'|<error message>` instead of `subscribed:true, already`. This makes the two-call dispatch-and-wake idiom (`send_prompt` + `subscribe_to_idle`) a single call for the common case; `subscribe_to_idle` remains for standalone re-arming (e.g. after an auto-approved plan wake with no new prompt to send).
- **Plan handling (Conduct-mode verbs):** `approve_plan` (`{sessionId, feedback?}` ŌĆö flips to bypassPermissions + sends canonical approval prompt), `reject_plan` (`{sessionId, feedback?}` ŌĆö stays in plan mode, asks for refinement), `set_auto_approve_plan` (`{sessionId, enabled}` ŌĆö flip the per-instance auto-approve flag). All three share the same approval/rejection text as the UI's Approve & Implement / Reject buttons via `src/planApproval.js`. `approve_plan`/`reject_plan` also auto-subscribe by default (see above).
- **Question handling:** `answer_question` (`{sessionId, answers}`) ŌĆö structured answer to a worker's `AskUserQuestion`, mirroring the UI question card. `answers` aligns by index to the worker's pending questions (re-derived from the ring, same source as `get_recent_messages`); each entry is `{option}` / `{options:[ŌĆ”]}` / `{text}` / `{}` with an optional `note`. Delivers text **byte-identical** to a UI submit ŌĆö both call the DOM-free formatter in `public/userQuestionAnswers.js`. Soft-refuses `{ok:false, code}` with `NO_PENDING_QUESTION` / `ANSWER_COUNT_MISMATCH` / `INVALID_OPTION` / `NOT_MULTISELECT` / `EMPTY_ANSWER`. Also auto-subscribes by default (see above).
- **Worktrees:** `create_worktree`, `delete_worktree` (soft-refuses `{ok:false, reason, code:'WORKTREE_ATTACHED'|'WORKTREE_DIRTY'}` unless `force:true`), `sync_worktree`, `merge_worktree` (takes `{sessionId}` or `{project, worktree}` ŌĆö the latter works after the instance is gone; refusals carry a `code`: `WORKTREE_BEHIND`/`BASE_BRANCH_MISMATCH`/`PARENT_DIRTY`/`MERGE_FAILED`).

### Plugins
- **What a plugin is** ŌĆö a sibling project under `<projectsRoot>/` with a `conductor.plugin.json` at its root (schema: [protocol.md ŌåÆ Plugin system](protocol.md#plugin-system)). It contributes up to four capabilities: a supervised backend process, an embedded frontend, MCP tools forwarded through the conductor's MCP server, and **project conventions** (entries joining the Conventions catalog as `<plugin-id>/<slug>`, each carrying a CLAUDE.md fragment and/or a one-time scaffold directive). The last needs **no backend** ŌĆö a contributions-only plugin (`{id,name,version,pluginApi}` + `conventions`) validates, enables, and never starts a process. Discovery scans the projects root at boot and on **Ōå╗ Rescan**; duplicate ids resolve first-alphabetical-project-wins (the rest show `conflict`).
- **Conventions & scaffolds** ŌĆö an enabled plugin's `conventions` join the new-project dialog + `list_project_conventions` (namespaced `<plugin-id>/<slug>`). A convention carries a CLAUDE.md **fragment** (`file`, applied inline at creation ŌĆö the copy survives disable/uninstall) and/or a **scaffold** facet (`scaffold:{text|file}`, a one-time setup directive, **plugin-only**); at least one of the two is required. A convention flagged `hasScaffold` shows a "sets up" tag; when picked at creation its directive is composed (in selection order across all picked scaffold-bearing conventions) into one framed block **returned** by `create_project`/`POST /api/projects` under a `scaffold` field ŌĆö nothing persisted, no spawn coupling. The conductor folds it into the first worker's brief; the UI displays it read-only in the create confirmation (v1). A **scaffold-only** convention appends nothing to CLAUDE.md but still emits its directive. Only enabled+`ok` plugins contribute; a stale/missing fragment or scaffold `file` makes the plugin `invalid` at load. Since fragment and scaffold now live on one entry, author a scaffold-bearing convention to **degrade gracefully** if the scaffold step wasn't run (e.g. "if a project-local harness wrapper exists, use it; otherwise see the shared harness to create one").
- **Convention scope** ŌĆö every plugin convention declares a required `scope`. Only **`project`** is accepted today (joins the project-Conventions catalog above); `workspace` and `conductor` are **planned** and rejected at manifest load with a "not yet supported" error. The scaffold facet has no scope of its own ŌĆö it's inherently project-scoped (it runs at project creation). ŌÜĀ’ĖÅ A future **`conductor`-scope** convention would edit the orchestrator's **own** rules (the `.conduct/CONDUCT.md` modules that govern how it delegates/reviews/merges across all projects) ŌĆö a powerful, trust-loaded capability (consistent with "plugins are trusted code", but far beyond a per-project convention); deliberately not enabled yet.
- **Trust stance (loud)** ŌĆö plugins are **trusted own code**. Their frontends are same-origin iframes: plugin JS can call every conductor API the UI can. Enabling a plugin is equivalent to running its code with your conductor's privileges.
- **App switcher** ŌĆö the sidebar `CodeConductor` title becomes a dropdown once an enabled plugin contributes a frontend (zero plugins ŌćÆ plain `<h1>`, no visual change). Selecting a plugin navigates to `#plugin/<id>/`; selecting Conductor deterministically closes the plugin view and restores the active session anchor (never `history.back()` ŌĆö a deep link has no usable back entry). Rendered by `public/appSwitcher.js` from `GET /api/plugins`.
- **Plugin view** ŌĆö `#plugin/<id>/<subpath>` shows the plugin's frontend in a full-page iframe (`public/pluginView.js`, src `/plugins/<id>/<subpath>` through the reverse proxy). Switching to an enabled-but-stopped plugin **auto-starts it behind a "Starting ŌĆ”" overlay** (REST statusŌåÆstart before the iframe loads, Ōēż30 s readiness); a start failure shows the error + crash tail with a **Retry** button instead of raw proxy JSON in the frame. Entering a plugin costs exactly **one** history entry (the bridge patches `pushState`ŌåÆ`replaceState` inside the iframe), so hardware **Back always lands on the conductor**; Escape works too. The sidebar stays live (`/ws` events keep flowing) while a plugin is open, and **any sidebar navigation** (selecting a session, opening commits/review/settings) closes the plugin view and returns to the conductor. Leaving the hash space blanks the iframe (frees memory). Limitation: multi-page (non-SPA) plugins still pollute joint history.
- **Settings ŌåÆ Plugins** ŌĆö management group (`public/pluginManager.js`): per-plugin state badge (`discovered / disabled / enabled / stopped / starting / ready / crashed / failed / invalid / conflict / incompatible`), contribution badges (`N project conventions`, `Project scaffold: N` with a per-scaffold name/description preview), immediate Enable/Disable/Start/Stop buttons, crash-output expander, Ōå╗ Rescan, and an **active version** dropdown (main + the project's worktrees). A ready row shows `running <version> @ <sha7>` + port. A **backendless (contributions-only)** enabled plugin shows state `enabled` and only a Disable button (no Start/Stop/version ŌĆö it has no process).
- **Plugin Library** ŌĆö below the installed-plugins list, a catalog of *installable* plugins (git repo URLs) with an **Install** button that clones the repo into the projects root (`GET`/`POST /api/plugins/library[...]`, `src/plugins/library.js`). Install only clones ŌĆö it never enables or starts anything; once cloned, the project surfaces in the list above through the normal discovery/rescan flow, same as any other plugin project. The catalog is two built-in entries ŌĆö `code-share` (`https://github.com/UnmanagedCode/code-share`) and `code-playwright` (`https://github.com/UnmanagedCode/code-playwright`, Playwright + Chromium glue for visual UI debugging) ŌĆö overlaid by drop-in manifests dropped into `<orchStoreRoot()>/plugins/library/*.json` (one JSON object per file: `{id, name, description?, repo, postClone?, postPull?}`; a dropped file whose `id` matches a built-in overrides it; malformed/incomplete files are skipped with a warning, never fatal). An entry already cloned (its derived target directory exists) shows "installed as `<name>`" and an **Update** button in place of Install. The target project name is derived from the repo URL's last path segment (`.git` suffix stripped); only `http:`/`https:`/`git:` repo URL schemes are accepted. A failed/timed-out clone shows the error + clone-output tail (reusing the crash-tail `<details>` styling) and cleans up any partial directory so the same install can be retried.
- **Plugin Library ŌĆö postClone/postPull + Update** ŌĆö an entry may declare `postClone`/`postPull` shell commands (run via `bash -lc`, cwd = the project directory, e.g. `code-playwright`'s `bash install.sh`) that install the plugin's own dependencies. `postClone` runs once, right after a successful install (clone + rescan); **Update** (available once installed) runs `git pull --ff-only` then `postPull`. A failed `git clone`/`git pull` is a hard failure (error + tail shown, and a failed clone is rolled back); a failed `postClone`/`postPull`, by contrast, is a **soft warning** ŌĆö the clone/pull already succeeded and the project is already discoverable, so the entry still shows as installed, just with a warning + output tail. The documented way to retry a failed `postClone` is hitting **Update** (which reruns `postPull` ŌĆö `code-playwright` sets both to the identical command). Running a library entry's declared command is a trusted-code-execution surface (bounded timeout, output always surfaced), consistent with the plugin system's overall "plugins are trusted own code" stance.
- **Lifecycle & lazy start** ŌĆö enabling records the plugin. Any discovered plugin project (enabled or not, valid or not) with no workspace is auto-assigned to the **CC-Dev** workspace on every rescan/boot ŌĆö enable, install, and manual rescan all trigger this via the same discovery pass, and a project already assigned elsewhere is never moved. The conductor's own project gets the same CC-Dev placement via a one-time boot seed, since it isn't a plugin. Enabled-but-stopped costs zero RAM: the first proxy request or MCP tool call starts the child and waits through the Ōēż30 s readiness window. Children are detached process groups that **survive conductor restarts** (adopted on boot when still alive+answering). Health is passive (exit events, proxy errors, on-demand probes) ŌĆö on Android, Doze may kill a child at any time; the next request 503s with the crash tail, then lazy-restarts after a short backoff. **3 crashes in 60 s ŌćÆ `failed`** (manual Start/Enable to recover).
- **Worktree-version activation** ŌĆö the version dropdown switches which checkout the child runs from (`main` or any of the project's worktrees, resolved via the worktree store). The manifest is re-read from the active checkout on every start and must keep the same id (else 400, previous state kept). Caveat: worktrees don't share `node_modules` ŌĆö a missing-deps start failure surfaces in the crash tail; the per-project `post-worktree-create.sh` hook can `npm install` automatically.
- **Plugin MCP tools** ŌĆö manifest-declared tools appear inside the existing conductor MCP server as `<plugin-id>__<tool>` (client-visible `mcp__code-conductor__<plugin-id>__<tool>`). Visibility: **globally visible** ŌĆö every enabled plugin's tools are offered to the conductor UI and to workers in **every** project (the manifest `scope` field is accepted for compatibility but inert). A disabled plugin's tools are simply unknown. Calls lazy-start the plugin and forward over HTTP with a per-manifest timeout. Note: workers snapshot the tool list at spawn ŌĆö mid-session enable/disable reaches only new sessions.
