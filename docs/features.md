## Features

### Projects & workspaces
- **Project list** — sidebar shows every dir under `<projectsRoot>/`. `+ New project` creates a dir with `CLAUDE.md` containing `@../CLAUDE.md`, optionally followed by inline snapshots of selected rule modules (see **Rule modules** below). Worktree-owned dirs are hidden from the list (filtered against a Set built from the central store).
- **Delete project** — `×` per row, typed-name confirm; cascades through instances + worktrees then `rm -rf`s the project. `~/.claude/projects/<encoded>/` jsonls are left intact (still resumable by the standalone CLI).
- **Workspaces** — `≡` hamburger menu hosts `+ New workspace`; nest projects under collapsible `<details>` headers, one workspace per project, empty workspaces persist. Edit (rename / re-pick members / delete) via the ✎ button on each header. Workspaces default-collapsed; expand state persists in `localStorage` (`code-conductor:workspaces-expanded`). Stored server-side at `<store>/projects/<name>/project.json`; the workspace set is the union of `<store>/workspaces.json` and any referenced names. Workspace names use the same `^[a-zA-Z0-9._-]+$` regex as project names.
- **Git status pill** —
  - **Project rows**: amber `↑N ↓M` against the currently-checked-out branch's *configured upstream* (e.g. `origin/<branch>`), using cached refs only (no `git fetch`). No pill when in sync / detached / upstream-less.
  - **Worktree rows**: same shape but compares the worktree branch to its captured *parent base branch* (may be a local branch, not a remote). `↑N` = ahead and FF-able, `↓M` = behind and FF-able, `↑N ↓M` = diverged → rebase needed before merge.

### Worktrees
- **Isolated environments** — "Run in isolated git worktree" checkbox in the new-instance dialog runs `git worktree add ../<project>_worktree_<short-id> -b code-conductor/<short-id> <currentSha>` and spawns the agent with `cwd` set to the worktree. The orchestrator captures `{baseBranch, baseSha, branch}` at creation so you can spawn off any branch and land back later. Worktrees survive instance death — re-spawn into the same worktree from the sidebar.
- **Two-step rebase-back**:
  - **Sync** — already in sync → no-op; purely behind & clean → server-side `git merge --ff-only <baseBranch>`; diverged & clean → orchestrator attempts `git rebase <baseBranch>` directly (returns `rebased` on success); if rebase hits conflicts, it aborts cleanly and falls back to sending the agent a templated rebase prompt (`rebase-prompt-sent`; agent runs `git rebase` itself, asks the user via `AskUserQuestion` on non-trivial conflicts, replies `REBASE_DONE`); behind-but-dirty → rebase prompt sent directly (agent commits/discards first).
  - **Merge** — `git merge --no-ff --no-edit <worktreeBranch>` on the parent. Always produces a merge commit (default msg `Merge branch 'code-conductor/<id>'`) so each contribution stays visible in `git log --graph` and is revertible as one commit. Refuses inline (not as a 500) if the worktree is behind the parent, the parent is on a different branch than `baseBranch`, or the parent tree is dirty.
- **Delete** — refused if there's a live instance or uncommitted work, with a `force=1` override.
- **Post-worktree-create hook** — if the project repo contains `.code-conductor/post-worktree-create.sh`, the orchestrator runs it automatically inside the new worktree after the worktree directory, branch, and metadata are written. Use case: symlink `node_modules`, copy `.env`, run `npm ci`, etc. The script receives env vars `CC_WORKTREE_PATH`, `CC_PROJECT_NAME`, `CC_BRANCH`, `CC_BASE_BRANCH`, `CC_PARENT_PATH`. Non-fatal: a failure warns but does not roll back the worktree. Output is capped to the tail ~16 KB (`truncated:true` when exceeded). Timeout: 120 s (`ORCH_POST_WORKTREE_TIMEOUT_MS`). Kill-switch: `ORCH_DISABLE_POST_WORKTREE_HOOK=1`. Result is reported in the spawn/worktree-create response as `postWorktreeCreate: { ran, exitCode, durationMs, output, truncated?, timedOut? }`. The spawn dialog re-opens briefly with a one-line summary (expandable output) when the hook ran.

### Diff / review & commit history
- **Worktree diff/review** — `±` button on every worktree row opens a full-page, mobile-friendly diff browser (`#review-view`, `public/review.js`). Fetches structured per-file diff JSON from `GET /api/projects/:name/worktrees/:wt/diff` (`<baseBranch>...HEAD`) and renders collapsible `<details>` per file (status badge, `±` counts, hunks reusing `.diff-line.add/.del/.ctx`, `overflow-x:auto` for horizontal scroll). Diff capped at ~200 KB → `(truncated)` in the header.
- **Commit history** — `≡` button on every git-repo project row opens a full-page scrollable commit list (`#commits-view`, `public/commits.js`) of the project's **current branch (HEAD)**: short sha, subject, author, relative date, newest first. Backed by `GET /api/projects/:name/commits` (default 100, max 500; header shows `(showing latest N)` when truncated). Tapping a commit opens the **same diff renderer** showing just that commit's change (`GET /api/projects/:name/commits/:sha/diff`, via `git show`); the diff layers over the list and `← Back` returns to it. Merge commits render as "No changes" (combined diff not parsed). A **branch/merge graph rail** (git-log --graph style) is drawn to the **left** of each row: colored lanes, a dot per commit, diagonals where branches fork and merges converge. Topology is computed client-side in `computeGraph()` (`public/commits.js`) from each commit's `parents` (SHAs from `%P`); lanes use stable column indices so pass-through lines stay aligned. Each rail is a fixed-height SVG "node band" (dot + diagonals) plus full-height CSS filler lines, so it aligns with variable-height (wrapping) rows without measurement. Parents older than the cap / on unfetched branches simply trail off the bottom. Soft cap of 12 lanes (extras clamp to the last column, logged to console).

### Sessions & instances
- **Unified view** — Sessions list per project (and per worktree) shows live + historical. Header: `Sessions (N) · K live · last <ago>`. Click to focus live instance, or resume stopped one (`POST /api/instances` with `--resume <sid>`). Synthetic `(new session)` row for spawned-but-no-jsonl-yet instances. Rows bucket into up to three sections: the normal list, then a dim `— temp —` separator, then a dim `— conducted —` separator (each section only renders when non-empty).
- **Conducted sessions** — a session spawned via the MCP `spawn_instance` tool (orchestrator-driven) rather than the browser/HTTP path is marked **conducted** (a worker an orchestrator conducts; the orchestrator itself is the `.conduct` session). Orthogonal to temp: unlike temp (in-memory, wiped on exit) the conducted marker is **durable** — persisted to `<store>/conducted-sessions.json` (a `{sessions:[sid,…]}` set, written on turn-end via the same path as last-prompt/permission-mode metadata) so it survives exit, restart, and `--resume` (a UI-resumed conducted session re-acquires the marker from the sidecar). Set **only** in the MCP `spawn_instance` handler — plain UI spawns stay non-conducted. Grouping precedence: a conducted session renders under `— conducted —`; a conducted **and** temp session renders under `— conducted —` but keeps the warm temp preview colour (`.session-row.temp`); a temp-but-not-conducted session stays under `— temp —`. Pure conducted rows get a cool preview tint (`.session-row.conducted:not(.temp)`). It's a grouping, never a filter — conducted sessions stay fully visible in `list_sessions`, the HTTP session APIs, and the sidebar. The marker is cleaned up alongside the session-title sidecar on session delete / temp-exit / boot sweep.
- **Unread indicator** — small accent pill next to a row when its instance finishes a turn while you're viewing a different session (per-instance, persisted in `localStorage` key `code-conductor:unread`).
- **Archive session (the `×` action)** — the hover-revealed `×` on a session row **archives** it (never deletes): single confirm, `POST …/sessions/:sid/archive`; on 409 the client auto-retries with `?force=1` (stops the live instance) without a second prompt. Archived sessions keep their jsonl (still resumable) + title + conducted markers and **disappear from the sidebar entirely** — they're managed only from **⚙ Settings → Archived**. There is no longer an inline `— archived —` sidebar section, and sessions are never deleted from the sidebar.
- **Archived page (Settings → Archived)** — lists every archived session grouped by owning project (`GET /api/archived`). Each project group is a `<details>` **collapsed by default**; expanding reveals its archived sessions (title/first-prompt/sid + worktree tag + last-active). Per session: **Restore** (`POST …/restore` → reappears in the normal sidebar) and **Delete** (confirm → `DELETE …/sessions/:sid`, the single permanent jsonl-removal path; cannot be undone).
- **Session summary** — `⋮` menu's `📝 Summarize` opens a modal dialog that generates and displays an AI-written summary of the session via a one-shot `claude -p` call (Haiku). Three tiers: **Short** (~40 content words, plain prose gist), **Medium** (~150 words, scannable recap with bullets/tables), **Long** (~400 words, thorough with `##` headings and structured tables). Summaries are rendered as full markdown (headings, tables, code) via the same `renderMarkdownInto` pipeline as assistant messages. A **stale indicator** (`⚠ Session has grown since this summary`) appears when the session has received more turns since the last generation; regeneration is always user-initiated. Stored in `<store>/session-summaries.json` keyed by `sessionId` — `{ summary, length, generatedAt, messageCount }`; auto-removed when the session jsonl is deleted.
- **Rename session** — `⋮` menu's `✎ Rename` opens a prompt to set a custom human-readable label (≤100 chars, empty clears) that overrides the first-prompt preview in the sidebar and prepends an italic chip to the active header. Stored in `<store>/session-titles.json` keyed by `sessionId`; auto-cleaned when the session jsonl is deleted or a temp session exits. Survives rewind (same `sessionId`); fork starts un-titled.
- **Rewind & fork** — every user message bubble has `↶` (rewind in place, same `sessionId`) and `⑂` (fork to a new `sessionId`) on hover. Both atomically rewrite the jsonl, preserve the dropped prompt in the composer, and 409 during a running turn. Bubbles anchor on the **server-stamped `userIndex`** riding on each outer `user_echo` (the absolute Nth-pure-user-prompt ordinal) — never a client-side bubble count, which would drift once the capped ring / tail-only snapshot starts the view mid-history. An echo without a stamp (rare orphan) renders without the buttons.
  - Rewinding **to the first user message** is special-cased: prefix is empty, so the orchestrator deletes the jsonl entirely and respawns with `--session-id <sid>` (not `--resume`, which the CLI rejects on zero-line files).
  - **Temp sessions cannot be forked** — the on-exit cleanup races the fork's source copy.
  - UI-only — no MCP equivalent yet.
- **Crash recovery** — subprocess death → `crashed`; Resume respawns with `--resume <sid>`, preserving the in-memory event ring + conversation.
- **Session anchor** — `#session=<sid>` URL hash; on reload, auto-resumes by locating the jsonl on disk (worktree-aware) via `GET /api/sessions/:sid/locate`. `--resume` spawn costs zero API tokens until the next prompt. Stale anchors are silently cleared.

### Spawn options
- **Mode** — `plan` (read-only, propose then `ExitPlanMode`), `ask` (every destructive tool gates through Allow/Deny card), `code` (full power, CLI's `bypassPermissions`). CLI's `default` / `acceptEdits` are not exposed: in `stream-json --print` (no SDK `canUseTool` callback) they auto-deny tool calls and the only recovery is forcing the model to regenerate the entire tool input.
- **Effort** — `low` / `medium` / `high` (default) / `xhigh` / `max`.
- **Thinking** — `adaptive` (default) / `enabled` / `disabled`.
- **Model** — empty (account default) or pick a family (Haiku / Sonnet / Opus / Fable 5; Fable 5 hidden if disabled in Settings → Models). Context windows: **Haiku always 200k** (no 1M build), **Opus / Fable 5 always 1M** (bare CLI default), **Sonnet 4.x uses the stored preference** (configurable in **⚙ Settings → Models**; defaults to 1M), **Sonnet 5 always 1M** (no 200k build). The concrete version each family resolves to is also configurable in Settings → Models (defaults `claude-fable-5` / `claude-sonnet-4-6` / `claude-opus-4-8` / `claude-haiku-4-5`). Context window and version are both re-derived on resume (`canonicalizeModel` in `src/modelVersions.js`) — neither is persisted in the session jsonl.
- **Temp session** — deletes the jsonl + sibling `subagents/` dir on exit, drops itself from the sidebar's Sessions list on exit/crash (no ghost row to clean up by hand), and skips `last-prompt`/`permission-mode` metadata appends during the run; mode defaults to `code`; header shows a `TEMP` pill.
- **Debug mode** — mirrors raw CLI traffic (`claude-stdin.jsonl`, `claude-stdout.jsonl`, `claude-stderr.log`, `meta.json`) to `<store>/projects/<p>/[worktrees/<wt>/]debug/<instance-id>/`. Append-mode, survives session end. Header shows a `DEBUG` pill.

### Live conversation
- **Tail-first load + scroll-up history** — subscribing renders only the ring tail (≤500 events, turn-aligned). When older history exists, a tappable `⋯ earlier messages` sentinel sits above the oldest content and scrolling near the top auto-fetches the previous page (`GET /api/instances/:id/events?before=<cursor>&limit=200`, flipping the sentinel to `loading earlier…`). Pages render through the standard block pipeline on a detached node and splice in above with the viewport preserved; events evicted from the in-memory ring are reconstructed from the session jsonl. Caveats: the task panel and usage chip rebuild from the tail only, and a single turn longer than the ring cap pages back with a gap (never duplicated content).
- **Text** — streams as text-node deltas; on `text_end` re-renders through `public/markdown.js` (headings, lists, code, **bold**, *italic*, links, autolinked bare `http(s)://…` URLs, `![alt](src)` images). Link schemes restricted to `http(s)/relative/fragment/mailto`; image `src` to `http(s)/file://`/absolute path; no `innerHTML`; raw HTML in source rendered as literal text.
- **Text-to-speech (read aloud)** — the inverse of voice dictation: finalized assistant text blocks get a trailing `🔊` button (tap to hear it), and a global **auto-speak** toggle reads each reply as it lands. Local server-side Piper — no network, no API key. Synthesis is **streamed per sentence**: `POST /api/tts` returns a sequence of `[4-byte LE length][WAV]` frames (one per sentence) flushed as Piper yields them; the client (`public/tts.js`) decodes each with the Web Audio API and schedules them back-to-back, so the first sentence starts playing (~0.3s) while the rest synthesize. The 🔊 button appears once Piper is installed (`GET /api/tts/status` gates it; Settings → TTS installs/flips it at runtime). Auto-speak only fires after the user has interacted at least once (browsers block audio without a prior gesture); a 🔊 tap always works. Sub-agent text isn't auto-spoken.
- **Thinking** — collapsible block; redacted thinking renders as a non-expandable `thinking (redacted)` line (no disclosure caret). Opus 4.7 emits only a `signature_delta`; Opus 4.8 streams empty `thinking_delta`s (`""`) — the parser drops the empties so both collapse to the same redacted line.
- **Tool use** — collapsed by default with smart one-line summary (`🔧 Bash · ls -la · done`). Edit/Write/NotebookEdit render as syntax-coloured unified diffs (green/red gutters, ±counts, sticky file-path); Write shows a numbered preview. Other tools: raw-JSON input in its own collapsed `↪ tool_input` block.
- **Tool result** — truncated at 4 KB with "show full". `image` content blocks (e.g. `Read` on a `.png`/`.jpg`) render as inline `<img>` thumbnails alongside the text — base64 sources become `data:` URLs, `url` sources only pass through for `http(s)`/`file://`, `image/svg+xml` is refused (XSS). Auto-expanded when an image is present. Tap any conversation image to open it full-size in an in-page lightbox (`public/lightbox.js`) — works for `data:`/`file://` sources that Chrome on Android refuses to open as a top-level navigation. Opens fit-to-screen; tap the image to toggle 1:1 full native resolution (backdrop scrolls to pan), tap again to fit; tap backdrop or press Esc to close.
- **Sub-agent drill-down** — `Task` tool calls nest a mini-conversation (dashed border, `↳ sub-agent` label) inside the parent tool block, routed by `parentToolUseId`.
- **Plan mode card** — green-bordered card "Plan ready for approval". Body comes from `input.plan`, or — when the model wrote the plan to `~/.claude/plans/*.md` first and called `ExitPlanMode` with empty input — the most-recent such file. **Approve** sends `"I approve the plan. Please proceed with the implementation."` (+ feedback if provided) and flips the instance to `code` (`bypassPermissions`). **Reject** keeps plan mode active and sends `"I'd like to revise the plan. Refinement notes:\n<feedback>"`. **📋 Auto-approve plans** toggle sits inline immediately left of the mode dropdown and is only visible while the instance is in `plan` mode (one-click, mid-turn friendly); per-instance, server-side state (mirrored down through `snapshot`/`status` WS frames so every tab agrees), cleared on server restart. When on, the orchestrator fires the same setMode+approval-prompt the moment a `plan_request` lands — works regardless of whether any tab is subscribed / focused / minimized. Green-tinted background when on, outlined when off.
- **AskUserQuestion card** — blue card; multi-question tab strip across the top. Same input field flips role: **Other:** before any option picked (overrides), **Add a note (optional)** after a pick. Typed text persists across pick/unpick. **Send all answers** enables once every question is answered and queues if instance is busy (flushes on next `status=idle`).
- **Ask-mode card** — purple-bordered, with tool name + summary + full diff/Write preview. **Allow** → `permissionDecision:"allow"` resolves the held-open HTTP hook response; CLI proceeds with the original `tool_use_id` (no model regeneration). **Deny** → inverse. `permission_resolved` flips the card to ✓/✗ across tabs.
- **System notes** — kept: `init`, `stderr`, `exit`, `permission_denied`, `compacting`, `spawn_error`, `crashed`, `history_load_error`, non-allowed `rate_limit_event`. Filtered: per-turn `status:"requesting"`, `rate_limit_event:"allowed"`, hook lifecycle pings, task progress. Non-allowed `rate_limit_event` renders as a parsed one-liner (e.g. `⚠ Usage 75% of 5-hour limit · resets 6:40pm`, `⛔ 7-day limit reached · resets Sat 5:00pm · overage unavailable (out of credits)`), not raw JSON.
- **Turn footer** — duration, cost, tokens.

### UI elements
- **Task panel** — strip above composer mirroring `TaskCreate`/`TaskUpdate`. Groups into batches: while at least one task is pending/in-progress, the whole batch (including its ✓s) stays. Panel hides only when *all* batches are fully completed; a new task post-batch drops the historical ✓s and starts fresh.
- **Footer status bar** — thin row pinned between task panel and composer. **Left**: pulsing green dot + animated `Claude is working…` ellipsis while the active instance is in `turn` status (respects `prefers-reduced-motion` — stays visible but stops animating). **Right**: the combined ctx + rate-limit chip. Collapses when no instance is selected.
- **Combined context-usage + rate-limit chip** — `ctx N% · 245k/1M` pinned at the footer right (the turn-indicator footer). `N%` = latest `message_start`'s `input_tokens + cache_read_input_tokens + cache_creation_input_tokens` over the model's context window. Updates **live mid-turn** (each agent-loop step fires a `message_start`). Colour-graded green/amber/red at 50%/80%. The same chip carries the **rate-limit** half — live bucket / utilization / reset-time, with an OVERAGE badge when `isUsingOverage` — and tapping opens the usage popover (session totals: turns, duration, cost, uncached input, output, cache reads + hit ratio, cache creation; plus account rate-limit windows). Window sizes hardcoded per family (suffix-stripped before lookup): Haiku → 200k, Fable 5/Opus → 1M, Sonnet 5 → 1M, Sonnet 4.x → the stored preference. **Known limitation:** for a bare Sonnet 4.x id the ctx % falls back to the stored preference rather than session-specific truth, so it can mis-report if the preference changed after that session was spawned; Sonnet 5 has no such ambiguity (always 1M).
- **Cost dashboard** — a full-page `#costs` view (`public/costs.js`, built on the `hashView` scaffold) breaking down spend from the append-only per-turn cost log (`<store>/costs.jsonl`, written by `src/costTracking.js` on every `turn_end`). Shows total spend + turn count, an expandable **By project** table (per-project cost/turns, drill-down to per-model tokens: input / output / cache-create / cache-read), a **By model** table, and a **Daily spend** bar trend. Backed by `GET /api/costs/summary`; shows a "no cost data recorded yet" empty state until the first turn lands.
- **Composer** — Enter sends, Shift+Enter newlines. Placeholder reflects state ("turn running — type to steer the running turn", "click Resume", etc.). Messages typed during a running turn steer it live — not queued. Attachments via `+` button, paste, or drag-and-drop (10 MB cap each). Saved to `<store>/projects/<p>/[worktrees/<wt>/]attachments/<timestamp>-<name>` and referenced via `` Attached file: `<abs-path>` `` text blocks — no inline base64, Claude `Read`s on demand. Project/worktree dirs stay clean (no `.gitignore` needed). User-bubble thumbnails: live echoes paint from in-memory base64, replays fetch via `GET /api/instances/:id/attachments/<filename>`.
- **Voice dictation** — no separate mic button; dictation is folded into the **Send** button (WhatsApp-style). Two modes, both reuse the same `MediaRecorder` → `POST /api/transcribe` pipeline:
  - **Empty composer** (whisper installed): Send button becomes a mic (neutral styling). Tap to start recording (red pulse); tap again to stop and transcribe. Transcribed text is inserted at the caret.
  - **Non-empty composer** (whisper installed): **Hold the Send button ~500 ms** to enter recording mode (red pulse). Release to stop and transcribe; the result is **appended** to existing text with a space separator. A short tap still sends normally. The button tooltip reads "Send message (hold to dictate)" as a discoverability cue. Holding then cancelling (e.g. finger leaves mid-hold, or `pointercancel`) aborts cleanly — nothing is appended, nothing is sent.

  While the transcription POST is in flight the button shows an amber spinning icon, then the resulting text is inserted/appended (space-joined so back-to-back words don't run together). Dictated text stays plain in the composer; when you **send** a message that included any dictation, it's prepended with a `<transcribed>` tag so the receiving instance knows it contains speech-to-text that may have transcription errors. When whisper isn't installed, an empty composer shows a disabled Send and holding the Send button is a no-op. Recording uses Pointer Events (`pointerdown`/`up`/`cancel`) with explicit pointer capture so releases register reliably on touch. Runs whisper.cpp **locally on the server** — no network, no API key, no telemetry. The mic affordance only appears once whisper is installed (`GET /api/transcribe/status` gates it; the Settings page can flip it at runtime). Recording uses the browser's `MediaRecorder` (webm/opus); the server converts via ffmpeg → 16 kHz mono WAV before calling `whisper-cli`.
- **Settings page** — `⚙ Settings` in the sidebar `≡` burger menu opens a full-page view inside `#main`, hash-routed at `#settings` (survives reload; closing restores the previously-active session anchor). `public/settings.js` (+ `public/models.js` client cache/resolver). Groups:
  - **Transcribe** — whisper availability + active model, a curated model picker (`tiny.en-q5_1` → `large-v3-q5_0`, size labels + installed/active badges), and an **Install** button. Picking an installed model switches active instantly (persisted server-side, no restart); a not-installed one offers Install (runs `bin/install-whisper.sh` server-side, streams the log). Clean install sets it active and re-reveals the composer mic without a reload.
  - **TTS** — Piper availability + active voice, an auto-speak toggle, a playback-rate slider (0.5–2.0×), a curated voice picker (installed/active badges), and an **Install** button (runs `bin/install-piper.sh`, same shape as Transcribe; reveals the 🔊 buttons on clean install).
  - **Models** — one picker per family (Fable 5 / Opus / Sonnet / Haiku) selecting the concrete model *version* used by the spawn pickers; persisted in `settings.json` `models`, effective immediately (no restart). Haiku always 200k; Fable 5/Opus always 1M; **Sonnet dropdown shows 5 entries** (Sonnet 5 as a single 1M-only entry, no 200k build; Sonnet 4.6 and 4.5 each as 2 entries for their 200k/1m sub-choice) — selecting a 4.x entry persists the version (`POST /api/settings/models`) and the context-window preference (`POST /api/settings/models/prefs`); `resolveSpawnModel`/`canonicalizeModel` re-apply the window at spawn/resume. Plus per-family **enable checkbox** + **default radio** (default falls back to the first enabled family in order sonnet → haiku → opus → fable). Overage/compact prefs:
    - **Action on overage** — `Off` / `Stop` / `Stop & resume` (persisted `models.onOverage`, default `none`). `Stop`/`Stop & resume` **soft-interrupt** the turn (idle-but-alive, never killed/respawned); `Stop & resume` also arms an **in-memory** resume deadline (~5 s after the window resets, drives an "auto-resumes &lt;time&gt;" badge; survives orchestrator restart via the resume manifest) fired by a **wall-clock sweep** (not a per-session `setTimeout`) so it survives the process being suspended (Android Doze / Termux backgrounding) and fires on the first tick after wake. Trip detection + conductor-aware routing live in [architecture.md](architecture.md) → overage. The action buttons, the threshold enable checkbox, and the threshold slider (below) are staged locally and committed together — see **Apply**, below.
      - **Global stop-and-queue lockout** — under `Stop & resume`, while the window is active **every** session queues its sends, not just the one stopped mid-turn: an idle/existing chat and a **brand-new chat started during the window** both queue (each queued-only session **arms its resume deadline immediately**, since there's no mid-turn→idle transition to arm on). Typing does **not** resume a session inside the still-throttled window: each message is **queued** and delivered as **one combined prompt** (reset text + numbered list + all queued attachments) when the deadline fires. This is a **hard lockout — no override, no early-resume button** (resume happens only on the deadline; internally `fireNow()` stays as the sweep/test seam). **Safety rail:** global queueing engages only when there's a valid *future* reset time with an armed clear timer — a missing/past reset means sends flow normally (never a permanent lockout). Plain `Stop` never queues (no flush path). A **queued-only** session (idle/new, never stopped mid-work) resumes with a **softened preamble** ("Delivering the messages you queued while paused") instead of "continue where you left off". The composer stays usable but shows a **"Paused until &lt;time&gt; — messages are queued"** banner (keyed on the global `overageActive` signal, so it shows before the first message is even typed) and its send button reads **"Queue"**; queued messages render as muted **ghost bubbles** ("queued" chip + clock) that collapse (dim) into the delivered turn on fire, marked by a `Window reset — delivered N queued` system line. The auto-resume badge (header chip + sidebar) appends `· N queued`. Queued messages + the stopped-vs-queued-only flag are persisted in the resume manifest so a restart doesn't drop them.
    - **Usage threshold** — `models.overageThresholdEnabled` + `models.overageThresholdPct` (default `{false, 85}`, int percent clamped to **[10,99]**, no step snap); a checkbox + 10–99% slider that fires the same action when `utilization` crosses the percent. Watched from **two** sources: the live `rate_limit_event` stream (which Anthropic only emits near its own ~90% mark) and a periodic server-side usage poll (`src/usageOverageMonitor.js`, cadence `ORCH_USAGE_POLL_MS`, default 60 s) of the **five-hour** window — so *low* thresholds (e.g. 25%) trip even though the stream never reports that low. Both feed the same one-shot machinery (deduped, no double-trip); the poll degrades silently if its usage fetch fails. No effect when the action is `Off`.
    - **Apply** — the checkbox drag/slider used to auto-commit on release, which made a stray overshoot commit an unintended threshold. Now the action buttons, the threshold checkbox, and the slider all stage locally (an "Unsaved changes" hint appears, the **Apply** button enables) and commit together in one `POST /api/settings/models/prefs` only on Apply click; the slider's live drag only updates its `%` label. Re-opening Settings (or an unrelated save elsewhere on the page, e.g. picking a model version) never clobbers an un-applied edit — the overage group is skipped when re-syncing from the server while dirty, and reset to the server's committed values on (re)open. Apply also **re-evaluates whatever is happening right now, in both directions**: it forces one immediate server-side usage-poll tick (so a *lower* threshold that's already crossed stops a mid-turn session now, instead of waiting up to `ORCH_USAGE_POLL_MS`) and force-resolves every parked `Stop & resume` auto-resume deadline against the new threshold (so a *raised or disabled* threshold resumes an already-parked session now, instead of waiting for its deadline — which is armed at window-reset and can be hours away). Both re-checks reuse the existing trip/resume machinery unchanged (`UsageOverageMonitor._tick` / `OverageResumeController.fireNow`) — Apply only triggers them on demand.
    - **Override compact window for the Conduct session** — opt-in toggle + 20k–1000k slider (persisted as `models.conductorCompactWindowEnabled` + `models.conductorCompactWindowK`) that sets `CLAUDE_CODE_AUTO_COMPACT_WINDOW` (raw tokens = sliderK × 1000) on the `.conduct` child env — applies to the orchestrator session only, **not** its MCP workers; seeded at boot from the orchestrator's own `CLAUDE_CODE_AUTO_COMPACT_WINDOW`.
  - **Archived** — every archived session grouped by project (`GET /api/archived`), collapsed-by-default `<details>`; per session Restore / Delete (see "Archived page" above).
  - **Workspace conventions** — consolidated Settings page with two subsections:
    - **Baseline (applies to all projects)** — code-conductor owns the projects-root `CLAUDE.md` (imported by every project via `@../CLAUDE.md`). Shows the startup-reconcile status (`created` / `up-to-date` / `updated` / `kept` / `conflict`); a **conflict** renders a unified diff (your copy vs canonical) + **Keep my copy** / **Overwrite (backs up your copy)** (`<target>.bak-<YYYYMMDD-HHMMSS>`). No inline editor / merge UI.
    - **Optional guidelines (opt-in per project)** — optional CLAUDE.md sections selectable at project creation. Four **built-in seed guidelines** ship with the app (`design-guidelines`, `testing-guidelines`, `documentation-guidelines`, `migrations-over-compat`); each has a name and description shown next to a checkbox in the new-project dialog. When checked, the guideline's markdown body is appended inline (as a snapshot) to the new project's CLAUDE.md after the `@../CLAUDE.md` import — **not** as an `@import`, so each project owns its copy and later catalog edits don't retroactively affect existing projects. Workspace-level conventions (git hygiene, README maintenance, URL buttons) are intentionally **not** seeded here — they already live in the always-on baseline every project imports; duplicating them per-project would cause drift. Custom guidelines can be added, edited, and deleted here (slug / name / description / body; persisted at `<store>/optional-guidelines.json`). Seed guidelines are read-only in the UI. The catalog (seeds + custom) is also exposed via `GET /api/settings/optional-guidelines` and the MCP `list_optional_guidelines` tool for scripted discovery.
- **Controls** — 📋 auto-approve-plans toggle (plan mode only), mode dropdown (live `control_request set_permission_mode`), **Resume** when exited/crashed. Context-aware **⏸ Interrupt** (turn running) / **🛑 Terminate** (otherwise, with confirm) lives in the ⋮ menu alongside 🐛 Debug.
- **Two-tier interrupt** — ⏸ Interrupt is a **soft** stop by default: it injects a steering message (tagged `[[cc:soft-interrupt]]`) asking the model to stop work and end its turn silently, preserving partial work. Live, the message is shown in the conversation as a `soft_interrupted` system annotation carrying the steer text ("⏸ Turn interrupted: Stop now...."); on JSONL replay/resume the same bare annotation ("⏸ Turn interrupted") appears without the text. It is never rendered as a user bubble and never counted as a real user prompt (so rewind/fork indices stay correct). While a soft stop is winding down, the header status chip reads **stopping…** and the bottom turn-indicator label switches to **Stopping…** with an inline **⏹ Interrupt now** button that escalates to a **forced** hard abort (`control_request` interrupt — severs the turn, discards in-progress work). The split is driven by a transient `interrupting` flag on the `status`/`snapshot` WS frames; it auto-clears when the turn ends. Backward compatible: omitting `force` (WS `interrupt`, MCP `interrupt_turn`) is always soft.
  - **Post-hard-abort drain window** — after a forced hard abort the CLI's internal stdin queue is not cleared: any messages written mid-turn (soft-interrupt steer, prompts sent while busy) remain queued, and the CLI will dequeue them and start spurious new turns. The orchestrator opens a **drain window** (`POST_ABORT_DRAIN_WINDOW_MS`, default 3 s) after every hard abort. The moment a spurious turn announces itself (via `system/init` on the event channel, ~39 ms after the abort and ~3.7 s before the first API round-trip), the orchestrator fires an immediate `control_request interrupt` to sever it — each spurious drain resets the window deadline (sliding window) so a queue of N messages is fully drained. Each drained turn shows a `drain_abort` system annotation ("⏹ Drained queued turn after interrupt") in the conversation. The window closes automatically when it elapses with no new turn-start. Calling `prompt()` (any explicit new-turn path) closes the window immediately so intentional follow-up prompts are never intercepted. Constant `POST_ABORT_DRAIN_MAX` (default 20) caps the number of drains per window as a safety guard.
- **Notifications** — 🔔/🔕 toggle in sidebar header (global, not per-instance). Pings OS when any instance finishes while tab is hidden (errors notify even when visible). On page reload the bell auto-enables if Notification permission was previously granted. Dispatched via Service Worker (`public/sw.js`) because mobile Chrome refuses page-level `new Notification(...)`. `notificationclick` focuses the existing tab. When ≥ 2 per-instance pings (tag `instance:<id>`) are live in the tray, an extra summary notification (tag `cc-summary`) is added: title `"N turns complete"`, body `"projA, projB, projC[ …+M more]"` (≤ 3 project names). Summary auto-clears when the tab regains focus, along with every per-instance ping. SW-only path — desktop page-level fallback skips summarization (the host OS already groups differently).
- **Resume after restart** — when the `⟲` button is pressed **while live instances exist**, a dialog offers **Resume after restart** / **Restart only** / Cancel (no live instances ⇒ immediate hard restart, as before). Choosing Resume POSTs `/api/admin/restart {resume:true}` and runs a drain → restart → resurrect cycle: every mid-turn instance is wound down to idle (networking stays up through the drain so in-flight hook round-trips finish), sessions (incl. temps) are carried over via `<store>/pending-resume.json`, and on boot each is re-spawned (`--resume`) and notified — conductors additionally re-spawn their workers from an injected project+sessionId+worktree list, while conducted workers are left for their conductor to revive. A session that was idle with nothing pending is resurrected silently. The frontend widens its poll budget (~120 s) on the resume branch. Internals (drain grace, stagger, three-group split, temp-preserve guards): see [architecture.md](architecture.md) → `resumeRestart.js`.
- **Restart server button** — `⟲` at sidebar bottom. POSTs `/api/admin/restart` (202 immediate, spawns a detached child, exits). Frontend waits 800 ms (let the old socket release), polls `GET /api/projects` with `cache:'no-store'` until 200, then `location.reload()` so HTML/CSS/JS get re-fetched. Anchor-auto-resume kicks in post-reload. Temp sessions are SIGKILLed and their jsonl + `subagents/` removed before exit, with a `<store>/pending-temp-cleanup.json` manifest swept on next boot to GC anything orphaned subagents recreated. Cleanup internals/rationale (SIGKILL-not-SIGTERM, two-layer sweep): see [architecture.md](architecture.md) → `restart.js` / `tempCleanup.js`.

The Conduct dialog is click-to-spawn (each model button spawns immediately; there is no pre-selected default).
- **`.conduct` project** lazy-created on first dialog open (`POST /api/projects/.conduct/ensure`). Lives at `<projectsRoot>/.conduct/`. `CLAUDE.md` contains a single in-project `@CONDUCT.md` import; `CONDUCT.md` itself is a symlink in `.conduct/` pointing at the repo's `CONDUCT.md` (the conductor role prompt-engineering, committed to source control). Claude Code only resolves `@-import` paths inside the project root — external paths need a UI approval that never fires in `-p` / headless mode (how every conductor session is spawned) and silently no-op, so the symlink keeps the import in-project while still tracking the committed file. The workspace-wide `@../CLAUDE.md` is omitted on purpose: Claude Code's ancestor walk-up already pulls in the projects-root `CLAUDE.md`. The dir is filtered out of `listProjects()` by the existing dot-prefix rule, so it never appears in the sidebar; the sidebar synthesises a `🎼 Conduct` row only while a live conductor instance exists. Reserved name: `POST /api/projects {name:".conduct"}`, `DELETE /api/projects/.conduct`, and `PUT /api/projects/.conduct/workspace` all return 400.
- **Workflow** (full details in `CONDUCT.md`): conductor uses MCP to `spawn_instance({mode:'plan', worktree:true})`, reviews via `get_recent_messages` / `get_worktree_diff`, lands via `approve_plan` → `sync_worktree` → `merge_worktree` → `kill_instance` + `delete_worktree`.
- **Sub-agent panel** — while viewing a conductor session, a **Sub-agents** strip appears above the task panel listing every worker spawned by that conductor via `spawn_instance`. Each card shows a label (custom title → first-prompt preview → `project · id8`) plus a live status marker: `○` spawning, `●` idle, `▶` running, `✓` exited, `✗` crashed. Status updates arrive via the existing `instances` hint → `refreshInstances()` path (the same mechanism that drives sidebar live dots). Tap any card to navigate directly to that worker's conversation view. Panel is scoped to the active conductor session and hidden for normal (non-conductor) sessions. Implemented in `public/subagents.js` (`SubagentPanel`); backed by the `callerInstanceId` field on `Instance` (set when `spawn_instance` is called with `?caller=<conductor sessionId>` on the MCP URL — the sessionId is resolved back to the conductor's live `instanceId` so `callerInstanceId` stays an instanceId for the panel's `i.callerInstanceId === activeId` filter).

### MCP interface
Mounted at `POST /mcp` (Streamable HTTP, JSON-RPC 2.0); tools exposed as `mcp__code-conductor__*`. Auto-registered on every spawn via `--mcp-config` (opt out with `ORCH_DISABLE_MCP_AUTOREGISTER=1`). No auth — localhost-only.

**Contract conventions (clean, no legacy aliases).** The worker handle is always `sessionId` (stable across respawn / `--resume` / restart); the per-process `instanceId` is internal and **never** appears on, nor is accepted by, the conductor-facing surface (a legacy `{id}` hard-fails; the REST/browser path keeps `id`). Every worker-addressing tool also accepts an **unambiguous prefix** of the sessionId (e.g. first 8 chars), resolved to the full id at the MCP boundary — exact full-id matches always win, an ambiguous prefix soft-refuses `{ok:false, code:'SESSION_AMBIGUOUS', matches:[…]}` (see [protocol.md → MCP tool protocol](protocol.md#mcp-tool-protocol)). Worktree is always `worktree`. Worker resolution is **strict-live + soft-erroring** — `{ok:false, code:'SESSION_NOT_LIVE'|'SESSION_UNKNOWN', reason}`, never auto-respawns (`respawn_instance` excepted). Inputs are schema-validated (unknown properties rejected); text-payload tools (`read_file`, `get_worktree_diff` diff mode, `get_recent_messages`) append raw un-escaped body blocks after the `content[0]` metadata; `ok:false`+`code` for soft refusals, prose+`statusCode` for errors. Full wire contract: [protocol.md → MCP tool protocol](protocol.md#mcp-tool-protocol).

- **Read:** `list_projects`, `list_workspaces`, `list_instances`, `list_sessions` (entries carry `conducted:bool` — true for MCP-spawned sessions; all sessions returned), `list_worktrees`, `locate_session`, `get_transcript` (**disk-backed, ring-first** event stream via `pageInstanceEvents`; `sinceSeq` is forward paging — a `sinceSeq` below `trimmedBefore` is served from the on-disk transcript reconciled with the ring by `_seq`, so eviction is invisible; response adds `hasMore`/`nextAfter` — poll with `sinceSeq = nextAfter`; `limit` clamped `[1,500]`), `get_recent_messages` (optional `count` → last N messages, default 1, max 50; `includeToolCalls` → include tool-call-only messages; `includeThinking` → include thinking blocks in `blocks[]`; **multi-block** — metadata `messages[]` carries `{index,msgId,hasToolUse,textChars,textTruncated,plan?,questions?,blocks?}` and one raw text body block follows per message, block k+1 ↔ messages[k]; text capped at 32 KB with `textTruncated`, `blocks[].input` capped inline with `inputTruncated`; **default-call bonding** — when `count` is omitted (not merely `1`) and the last message is plain prose whose immediate predecessor carries `plan`/`questions`, that predecessor is bonded in too (2 messages returned), so a turn that splits its plan/question and trailing prose across two assistant messages is still surfaced together by a plain default call; bonds at most one message back, never applies with an explicit `count`, and a message that already carries its own `plan`/`questions` is always returned alone; **disk-backed, ring-first** — if the ring's retained tail can't satisfy the recent text messages (tool-event eviction) it reads back into the on-disk transcript and merges by `msgId`, so a busy worker never returns a false-empty; metadata adds `source:"ring"|"disk"`, `omittedToolOnly`, `retained`, and a `hint` so empty-but-active is distinguishable from idle), `project_status` (`dirty` capped at 500 lines with `dirtyTruncated`/`dirtyTotal`), `read_file` (path-traversal guarded; `lineCountExact:false` when the byte-capped fast path may hold a partial final line; line params ignored for binary), `grep` (content search by regex across the project tree; path-traversal guarded; excludes `.git/`/`node_modules/`, no symlink follow; prefers ripgrep when on PATH else pure JS; three `outputMode` values — `"files_with_matches"` (default, single JSON `{files, fileCount, truncated}`), `"count"` (single JSON `{files:[{path,count}], totalMatches, truncated}`), `"content"` (**multi-block**: metadata `{matchCount, fileCount, truncated}` + raw grep-style body, capped ~200 KB); `glob` / `type` filter file set; `before`/`after`/`context` for context lines; `headLimit` caps result count; `truncated:true` means results clipped), `glob` (file discovery by minimatch-style pattern e.g. `"**/*.test.mjs"`; same traversal guards; returns single JSON `{files:[relPath,...], total, truncated}` sorted newest-first by mtime; `headLimit` default 1000), `get_worktree_diff` (unified diff of `<base>...HEAD`, always including the working tree's uncommitted state; tiered drill-down for large diffs — `summary:true` → single-JSON per-file stat `{totals, files:[{path,status,oldPath?,additions,deletions,binary}]}` plus an always-present `uncommitted:{totals, files, untracked}` section (never truncated); `paths:[…]` scopes to chosen files; otherwise **multi-block** — metadata `{project,worktree,baseRef,head:<sha>,contextLines,offset,truncated,nextOffset,totalLines,totalBytes,hasUncommittedChanges:bool,untracked:[paths],includedFiles?,omittedFiles?}` + a raw diff body, paginated by **line index** (≤~200 KB of whole lines from `offset`, default 0), mid-file pages re-emit file/hunk headers and a truncated page lists `includedFiles`/`omittedFiles`; never a silent cut; `head` is the resolved HEAD sha; staged+unstaged changes vs HEAD are appended after a `@@@ uncommitted … @@@` separator whenever any exist (absent on a clean tree), plus `untracked` lists new files not yet git-added), `list_optional_guidelines` (read-only; returns `[{slug,name,description,builtin}]` — body omitted; reflects seeds + any custom guidelines).

- **Workspaces:** `create_workspace` (`{name}` — register), `delete_workspace` (`{name}` — removes registry entry + clears member assignments), `rename_workspace` (`{oldName, newName}` — atomic), `set_project_workspace` (`{project, workspace}` — assign or clear with `null`/`""`; refuses `.conduct`; auto-registers a new workspace name).
- **Spawn/drive:** `spawn_instance` (`createWorktree:true` makes a fresh worktree off HEAD; `worktree:"<name>"` attaches to an existing one — the old `worktree: boolean|string` overload is gone. soft-refuses if not live/unknown, errors if not temp), `send_prompt` (`{sessionId, …}`, optional `wait:true`, 10 min cap; **auto-subscribes** to the worker's idle callback by default — see below), `wait_for_idle`, `subscribe_to_idle` / `unsubscribe_from_idle` (`{sessionId, timeoutMs?}`; one-shot async callback — when target hits `turn_end` the orchestrator injects a stub user prompt into the *caller* via `Instance.prompt()`, naming the worker by sessionId; **folded wake** — on a real `turn_end` delivered to an already-idle caller the stub **folds the worker's recent output inline** (the SAME content a default `get_recent_messages({sessionId})` call returns — bonded prose + plan/questions — via the shared `buildRecentMessages` builder + `flattenPayload`), so the conductor needs no follow-up call; the folded stub is tagged with a `[[cc:wake-callback]]` marker (`public/wakeCallback.js`) that the conductor UI renders as a **collapsible wake-callback bubble** (worker id + "finished its turn" summary always visible, folded payload collapsed by default) with an orchestrator badge (🔔) marking it as system-injected; the idle-subscription graph is keyed by `sessionId`; caller identity is read from `?caller=<caller sessionId>` baked into the per-worker MCP URL at spawn time (after the sessionId is minted); **carve-outs** — both the `timeoutMs` watchdog stub and the mid-turn-deferred stub keep the plain "call `get_recent_messages` to inspect" pointer text (no fold), but they're also tagged with the marker (via `markPlainStub`, body-less) so they render as the **same wake bubble** — badge + summary line only, no expandable body or caret; - **Auto-subscribe on turn-starting calls.** `send_prompt`, `approve_plan`, `reject_plan`, and `answer_question` each take `subscribe` (default `true`) and `subscribeTimeoutMs` — after the prompt is sent, the handler calls the same `instances.subscribeIdle(callerId, sessionId, subscribeTimeoutMs)` used by `subscribe_to_idle`, registering the caller (from `?caller=` on the MCP URL) for a one-shot wake on the target's next `turn_end`. `send_prompt({wait:true})` never subscribes — the turn already resolves inline, and a one-shot sub registered before a blocking wait would instead fire on the *next* turn. A subscribe failure (no caller id, self-subscribe, or the caller no longer live) never fails the send: the result carries `subscribed:false` plus `subscribeSkipped:'no-caller'|'self'|<error message>` instead of `subscribed:true, already`. This makes the two-call dispatch-and-wake idiom (`send_prompt` + `subscribe_to_idle`) a single call for the common case; `subscribe_to_idle` remains for standalone re-arming (e.g. after an auto-approved plan wake with no new prompt to send).
- **Plan handling (Conduct-mode verbs):** `approve_plan` (`{sessionId, feedback?}` — flips to bypassPermissions + sends canonical approval prompt), `reject_plan` (`{sessionId, feedback?}` — stays in plan mode, asks for refinement), `set_auto_approve_plan` (`{sessionId, enabled}` — flip the per-instance auto-approve flag). All three share the same approval/rejection text as the UI's Approve & Implement / Reject buttons via `src/planApproval.js`. `approve_plan`/`reject_plan` also auto-subscribe by default (see above).
- **Question handling:** `answer_question` (`{sessionId, answers}`) — structured answer to a worker's `AskUserQuestion`, mirroring the UI question card. `answers` aligns by index to the worker's pending questions (re-derived from the ring, same source as `get_recent_messages`); each entry is `{option}` / `{options:[…]}` / `{text}` / `{}` with an optional `note`. Delivers text **byte-identical** to a UI submit — both call the DOM-free formatter in `public/userQuestionAnswers.js`. Soft-refuses `{ok:false, code}` with `NO_PENDING_QUESTION` / `ANSWER_COUNT_MISMATCH` / `INVALID_OPTION` / `NOT_MULTISELECT` / `EMPTY_ANSWER`. Also auto-subscribes by default (see above).
- **Worktrees:** `create_worktree`, `delete_worktree` (soft-refuses `{ok:false, reason, code:'WORKTREE_ATTACHED'|'WORKTREE_DIRTY'}` unless `force:true`), `sync_worktree`, `merge_worktree` (takes `{sessionId}` or `{project, worktree}` — the latter works after the instance is gone; refusals carry a `code`: `WORKTREE_BEHIND`/`BASE_BRANCH_MISMATCH`/`PARENT_DIRTY`/`MERGE_FAILED`).
