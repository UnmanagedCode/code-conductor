## Worker lifecycle: reuse across merges, retire when the thread ends

This governs **worktree-backed** workers. (Read-only/operational workers have nothing to merge or pollute — keep those warm per "Operational tasks".)

**A merge is a checkpoint, not an endpoint** — `merge_worktree` fast-forwards the worktree onto the new base as its last step, so the same worker + worktree can pick up a strongly-related follow-up right away and merge again later; no post-merge sync needed. Gate reuse on relatedness, not merge state:

- **Strongly-related follow-up (same files / same feature) → keep the worker**, merged or not. Don't spawn — the worker's loaded context skips re-exploration. The worker is in `bypassPermissions` after the earlier `approve_plan`: for a substantial follow-up you want to review, `set_mode({sessionId, mode:'plan'})` first; for a small one, let it code. (If a feature legitimately spans many turns, `promote_session({sessionId})` keeps the worker as a named session.)
- **Genuinely unrelated work → spawn a fresh worker** in its own worktree off the updated base. One clean branch is one merge unit.

**Retire a worker — `kill_instance({sessionId})` then `delete_worktree({project, worktree})` (no `force`)** — do it when the thread of work ends, the worker is wedged or crashed, its context is polluted, or you're holding more live workers than you can track. A `WORKTREE_DIRTY` refusal guards uncommitted work only — pass `force:true` to deliberately discard it. It does *not* guard committed-but-unmerged commits (branch deletion there is silent best-effort): confirm `mergeStatus.ahead:0` in `project_status`, or that those commits are meant to be discarded, before retiring. A *refused or conflicted* merge is not done — `sync_worktree` and retry, keeping the worker for conflict resolution. When you keep a worker up across merges for more follow-ups, tell the user ("leaving `<sessionId>` up for more of feature X") so the live instance in the sidebar isn't a surprise.
