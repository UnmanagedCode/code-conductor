## Worker lifecycle: reuse across merges, retire when the thread ends

This governs **worktree-backed** workers. (Read-only/operational workers have nothing to merge or pollute — keep those warm per "Operational tasks".)

**A merge is a checkpoint, not an endpoint** — `merge_worktree` fast-forwards the worktree onto the new base as its last step, so the same worker + worktree can pick up a strongly-related follow-up right away and merge again later; no post-merge sync needed. Gate reuse on relatedness, not merge state:

- **Strongly-related follow-up (same files / same feature) → keep the worker**, merged or not. Don't spawn — the worker's loaded context skips re-exploration. Merge when the user needs the change on base or the feature is complete — not as a default turn-closing move. The worker is in `bypassPermissions` after the earlier `approve_plan`: for a substantial follow-up you want to review, `set_mode({sessionId, mode:'plan'})` first; for a small one, let it code. (If a feature legitimately spans many turns, `promote_session({sessionId})` keeps the worker as a named session.)
- **Genuinely unrelated work → spawn a fresh worker** in its own worktree off the updated base. One clean branch is one merge unit.

**Retire a worker — `delete_worktree({force:true})`, which kills the attached instance too — only after first making sure its work is landed or intentionally discarded** (force bypasses the dirty-tree refusal, so this is the only remaining guard against deleting unlanded changes); do it when the thread of work ends, the worker is wedged or crashed, its context is polluted, or you're holding more live workers than you can track. A *refused or conflicted* merge is not done — `sync_worktree` and retry, keeping the worker for conflict resolution; never delete a worktree with changes you still want. When you keep a worker up across merges for more follow-ups, tell the user ("leaving `<sessionId>` up for more of feature X") so the live instance in the sidebar isn't a surprise.
